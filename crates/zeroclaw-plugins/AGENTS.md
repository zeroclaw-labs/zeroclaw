# zeroclaw-plugins

## What this crate is

WASM plugin host for ZeroClaw. Handles plugin discovery, manifest parsing,
Ed25519 signature verification, and the current Extism-based WASM execution
bridge while the WIT / direct `wasmtime` host lands. Bridges plugin-exported
functions into ZeroClaw's `Tool` and `Channel` traits so plugins appear as
native capabilities to the agent runtime.

## What this crate is allowed to depend on

- `zeroclaw-api` (traits only — `Tool`, `Channel`, `ToolResult`)
- `extism` (legacy WASM runtime — being superseded by the `plugins-wasmtime`
  Component Model path under `component/v0/`)
- `wasmtime` / `wasmtime-wasi` / `wasmtime-wasi-http` (Component Model /
  WASIP2 runtime, `plugins-wasmtime` feature)
- `reqwest` (blocking, for legacy host function HTTP support; also the
  proxy-aware client used by the component-model `send_request` hook and
  `http-helpers`)
- `ring` (Ed25519 signatures)
- `serde`, `serde_json`, `toml` (serialization)
- `tokio` (async bridging via `spawn_blocking`)
- `tracing` (logging)
- `anyhow`, `thiserror` (error handling)
- **Narrow, justified exception:** `zeroclaw-config` (for
  `schema::build_channel_proxy_client`/`ws_connect_with_proxy` only) and
  `zeroclaw-infra` (for `stall_watchdog` only), both gated behind
  `plugins-wasmtime`. This crate still must not depend on channel/provider
  *business logic* or config *schema types* — these two narrow imports exist
  solely so the component-model plugin host can reuse the exact
  proxy-resolution and stall-detection code native channels already use,
  rather than reimplementing (and risking divergence from) it. Do not widen
  this to a general `zeroclaw-config`/`zeroclaw-channels` dependency.

Do not add dependencies on specific tools, channels, providers, or config
schemas beyond the exception above. This crate knows how to run plugins, not
what they do.

## Component Model (WASIP2) plugin host — `component/v0/`

Three plugin worlds, generated by `wasmtime::component::bindgen!` in
`component/v0/bindings.rs`: `tool-plugin`, `memory-plugin`, `channel-plugin`
(WIT sources in `wit/v0/`). Each has a thin adapter
(`ComponentTool`/`ComponentMemory`/`ComponentChannel`) implementing the
matching `zeroclaw-api` trait.

**Security invariant — do not weaken without updating this section:** a
plugin's outbound networking must always go through code that honours this
instance's `PluginNetworkConfig` (per-plugin `proxy_url` override + the
global `ProxyConfig`) and its `fine_grained_permissions` host allow-list.
Concretely:

- `wasi:http` is wired into every world, but `PluginHttpHooks::send_request`
  (`plugin_store.rs`) does **not** call
  `wasmtime_wasi_http::p2::default_send_request` (which opens a raw,
  unproxied connection) — it dispatches through a `reqwest::Client` built via
  `zeroclaw_config::schema::build_channel_proxy_client`, after the same
  allow-list check. This is what makes `wasi:http` safe to expose to every
  plugin world.
- `websocket.wit` (channel world only — WASI has no WebSocket primitive) is
  the *only* other networking surface a plugin gets, and `connect`
  (`websocket_host.rs`) wraps `zeroclaw_config::schema::ws_connect_with_proxy`
  behind the identical allow-list check (`PluginStore::is_url_host_allowed`).
- `gateway.wit` (`gateway_host.rs`) is a heartbeat/stall-detection driver
  layered on the same `websocket` connect path — it adds no new networking
  capability.
- `http-helpers.wit` (`http_helpers_host.rs`, multipart upload + bounded
  attachment download) is convenience sugar over the same proxy-aware client
  and allow-list — not a parallel security boundary.
- `plugin-config.wit` (`plugin_config.rs`) exposes `get-proxy-url`, backed
  by `PluginStore::network_config` (`host.rs::network_config_for`). Plugin
  access to secrets/config is not yet implemented — that's a follow-up.

If you add a new networking-adjacent WIT interface to the `channel-plugin`
world, it must route through `PluginStore::is_url_host_allowed` /
`PluginStore::proxy_client()` (or `websocket`/`gateway`'s equivalent) before
touching the network — never give a plugin a raw socket or a second,
unproxied HTTP client.

## Extension points

- **New host functions:** Add to `runtime.rs` alongside `zc_http_request`.
  Register in `create_plugin()`. Gate on a `PluginPermission` variant (add to
  `lib.rs` if needed). Plugins receive their own resolved config section in the
  `execute` input under `__config`; there is no host call for reading raw
  process environment.
- **New capability bridges:** Add alongside `wasm_tool.rs` and
  `wasm_channel.rs` (e.g., `wasm_memory.rs` for memory backend plugins).
- **New permissions:** Add variants to `PluginPermission` in `lib.rs`.

## What does NOT belong here

- Concrete tool or channel implementations (those go in `zeroclaw-tools` or
  `zeroclaw-channels`, or in a WASM plugin)
- Plugin business logic (that belongs in the plugin's own crate)
- Config schema definitions (those go in `zeroclaw-config`)
- Plugin registry client or distribution (future separate crate)
