# zeroclaw-plugin-sdk

## What this crate is

Guest-side SDK for authoring ZeroClaw WASI Preview 2 / Component Model
plugins. This is the counterpart to `zeroclaw-plugins` (the host): where
that crate hosts and dispatches to plugins, this crate is what a plugin
author depends on to *write* one. Enable exactly one of the `tool`,
`memory`, or `channel` features for the world your plugin implements,
implement the matching trait (`tool::ToolPlugin`, `memory::MemoryPlugin`,
`channel::ChannelPlugin`), and call the matching `export_*!` macro once.

Bindings are generated by `wit_bindgen::generate!` directly from
`wit/v0/*.wit` — the same files `zeroclaw-plugins`' host-side
`wasmtime::component::bindgen!` reads. Never hand-copy or fork WIT content
into this crate; both the `tool`/`memory`/`channel` world feature and the
`plugins-wit-v0` version feature mirror the host crate's own feature
naming so the two can't silently drift onto different WIT versions.

## What this crate is allowed to depend on

- `wit-bindgen` (guest Component Model codegen) — the only required
  dependency.
- Nothing else in non-dev paths. This crate compiles to `wasm32-wasip2`
  and must have zero host-runtime dependencies — no `tokio`, `wasmtime`,
  `reqwest`, or anything that assumes a native host environment. Guests
  never get their own async runtime or thread; the host drives everything
  through a poll-based model (`poll-message`, etc.).
- Dev-dependencies (`zeroclaw-plugins`, `zeroclaw-api`, `zeroclaw-spawn`,
  `tokio`, `tempfile`, `serde_json`) exist solely for the `tests/spike_*`
  integration tests, which build the `examples/*` plugins for
  `wasm32-wasip2` and load them through the real host to prove guest and
  host bindings agree on the wire format. These never appear in the
  published/non-dev dependency graph.

## A single build of this crate cannot enable more than one world feature

`wit_bindgen::generate!` calls for worlds that share an identical exported
interface (every world here exports `plugin-info`) produce a
`#[macro_export]`-style cabi shim named after that interface, which
collides at the crate root regardless of `mod` nesting once two such
worlds are generated into the same crate — confirmed empirically, not
just a theoretical concern. This is fine in practice: a single plugin
component implements exactly one world anyway. The crate's own test
targets enforce this via `required-features` in `Cargo.toml` (one
`[[test]]` per world); a real plugin author's `Cargo.toml` should likewise
enable exactly one of `tool`/`memory`/`channel`.

## What does NOT belong here

- Business logic for any specific plugin — that's the plugin author's own
  crate (see `examples/*` for the minimal shape one should take).
- The host-side adapters, `PluginHost`, manifest parsing, or signature
  verification — that's `zeroclaw-plugins`.
- Speculative new WIT interfaces (timers, event emission, streaming
  bodies, etc.) — those go through the `wit/VERSIONING.md` process in the
  main repo's `wit/` directory, not invented here.
- A typed JSON-Schema builder DSL for `tool::ToolMetadata` — deliberately
  out of scope; `parameters_schema` takes a raw JSON string. Don't add one
  without a concrete request driving it.

## Extension points

- **New world**: add a `wit_bindgen::generate!` call in `bindings.rs`
  gated by a new feature + `plugins-wit-v0`, a matching `<world>.rs`
  ergonomics module (trait with capability-gated defaults matching the WIT
  doc comments verbatim, logging/config helpers, re-exported generated
  types), and an `export_<world>!` macro in `macros.rs`. Add an example
  under `examples/` and a `[[test]]` entry with `required-features`.
- **New WIT major version** (`wit/v1/` ships): add a parallel
  `plugins-wit-v1` feature and sibling `bindings.rs` module/`<world>_v1.rs`
  ergonomics, without touching the `v0` path — matches the host's
  documented N/N-1 compatibility window (`wit/VERSIONING.md`).
