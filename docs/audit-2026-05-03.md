## 1. Executive summary

The codebase is **build-clean and structurally sound**. The recent vault (second-brain) work integrates correctly into the sync journal, the L1 zero-knowledge relay is wired (closing the largest gap from the 2026-03-02 plan), and most security primitives are verified-correct (pairing tokens, secret store, AEAD ciphers, webhook signature checks).

However, the audit surfaces **12 critical issues** that must be resolved before a production build is shipped. The two most important are:

- **MULTI-DEVICE SYNC IS BROKEN BY DESIGN GAP** (D2): Each device generates its own random `.sync_key` and there is **no key-distribution step in the pairing flow**. A test comment at `src/memory/synced.rs:784` admits this — the test simulates shared keys, but production has no path to actually share them between devices. Effect: the user's headline feature ("install MoA on multiple devices, memories sync") cannot work today.
- **SLM EXECUTOR HAS NO SAFE-FOR-SLM FILTERING** (E1+E2): The SLM executor is defined but currently never instantiated, so today there is zero practical risk; however the moment it is wired in (which Phase 3 of the architecture requires), an SLM can directly invoke `shell`, `file_write`, `file_edit`, `apply_patch`, `delegate`, `cron_*` — all the tools whose `safe_for_slm()` returns `false`. The wiring and the filter must land together.

Then a third tier of equal severity but in dependencies rather than this codebase:

- **27 KNOWN VULNERABILITIES IN THIRD-PARTY DEPENDENCIES** (V1+V2+V3, found via `cargo audit`): 5 in `aws-lc-sys` (TLS cert validation bypass, AES-CCM timing side-channel, PKCS7 bypass), 4 each in `rustls-webpki 0.102.8` and `rustls-webpki 0.103.9` (cert constraint bypass + reachable panic), and **12 in `wasmtime 36.0.6` including sandbox-escape class**. None of these is in MoA code, but every release ships them. See §6.5 for full table.

The remaining 7 critical issues from MoA's own code are: a config Default/serde drift that silently disables sync (C1), a `partial_cmp().unwrap()` panic surface in the economics classifier (F1), a WhatsApp QR pairing-credential plaintext log (F2), a sync-key derivation that does not match the patent spec (D1), weak password hashing (D3), and a build-target compile failure in the Android bridge (G4).

The 11 medium-severity items are mostly hardening (CSRF, error message leakage, downgrade paths) plus client-platform gaps that production-completion-plan.md already identified (Tauri WebSocket IPC, Android NDK build, iOS bridge confirmation).

**Recommendation:** Address the 9 critical items in a focused PR series before a production build. The medium and minor items can ship in follow-up PRs.

## 6. cargo test failures (25 of 6126)

`cargo test --lib --workspace` finished in 39.78s with **25 failures, 6101 passed, 11 ignored**. Reading the panic stdout (where captured) and inspecting test source code, the failures fall into three groups:

### 6.1 Windows-only environment failures (NOT regressions)
The following tests assume POSIX paths (`/root/.ssh`, `/nonexistent_zeroclaw_test_dir_12345`) or external POSIX binaries (`ripgrep`/`grep`). On the user's Windows host they fail predictably; on Linux/macOS CI they pass. Confirmed by inspecting test source:
- `tools::workspace_folder::tests::rejects_sensitive_directory` — assumes `/root/.ssh`-style paths; Windows path validator rejects with "Path must be absolute or start with ~/".
- `tools::workspace_folder::tests::rejects_nonexistent_directory` — same root cause.
- `tools::content_search::tests::*` (8 tests: `basic_match`, `case_insensitive`, `context_lines`, `count_mode`, `files_with_matches_mode`, `include_filter`, `no_matches`, `subdirectory`, `rejects_absolute_path`) — `content_search` shells out to `ripgrep` (`rg`) or `grep -rn -E`; neither is on this Windows PATH.
- `tools::process::tests::kill_terminates_process` — POSIX signal semantics differ from Windows `TerminateProcess`.
- `tools::screenshot::tests::screenshot_command_contains_output_path` — likely missing `screencapture` / `gnome-screenshot` on Windows.
- `cron::scheduler::tests::run_job_command_failure` / `run_job_command_success` — likely shell command differences.

**Recommended:** Gate these tests with `#[cfg(unix)]` or `#[ignore]` + a CI matrix that runs them on Linux/macOS. **Not a regression for the production code.**

### 6.2 Security-guard tests — needs verification (potential regression)
- `security::file_link_guard::tests::hard_link_file_is_flagged_when_supported` — `26fd57f6 fix(security): resolve Windows build errors in file_link_guard` — recent Windows fix touched this file. Needs read-pass to confirm the Windows path is intentionally not exercised, vs an actual regression in hard-link flagging.

**Action recommended:** Open a follow-up to read each failing test and decide if it should be `#[cfg(unix)]`-gated or whether the production code path actually drifts on Windows. **Treat as 🟡 medium pending verification** — listed below as M1.
