# ZeroClaw PR 워크플로우 (대량 협업)

이 문서는 ZeroClaw가 대량의 PR을 처리하면서 다음을 유지하는 방법을 정의합니다:

- 높은 성능
- 높은 효율성
- 높은 안정성
- 높은 확장성
- 높은 지속가능성
- 높은 보안성

관련 참고자료:

- [`docs/README.md`](../README.md) 문서 분류 체계 및 탐색 안내.
- [`ci-map.md`](./ci-map.md) 워크플로우별 소유권, 트리거 및 분류 흐름.
- [`reviewer-playbook.md`](./reviewer-playbook.md) 일상적인 리뷰어 실행 모델.

## 0. 요약

- **목적:** 대량 처리 협업을 위한 결정적(deterministic), 리스크 기반 PR 운영 모델을 제공합니다.
- **대상:** 기여자, 메인테이너, 에이전트 보조 리뷰어.
- **범위:** 저장소 설정, PR 생명주기, 준비 상태 계약, 리스크 라우팅, 큐 규율, 복구 프로토콜.
- **비목표:** 브랜치 보호 설정이나 CI 워크플로우 소스 파일을 구현 권한으로 대체하는 것.

## 2. 거버넌스 목표 및 제어 루프

### 2.1 거버넌스 목표

1. 대량 PR 부하에서 merge 처리량을 예측 가능하게 유지합니다.
2. CI 신호 품질을 높게 유지합니다 (빠른 피드백, 낮은 오탐률).
3. 위험한 영역에 대해 보안 리뷰를 명시적으로 유지합니다.
4. 변경사항을 이해하기 쉽고 되돌리기 쉽게 유지합니다.
5. 저장소 아티팩트에 개인/민감 데이터 유출이 없도록 유지합니다.

### 2.2 거버넌스 설계 논리 (제어 루프)

이 워크플로우는 리뷰어 부담을 줄이면서 책임을 명확하게 유지하기 위해 의도적으로 계층화되어 있습니다:

1. **접수 분류:** 경로/크기/리스크/모듈 라벨이 PR을 적절한 리뷰 깊이로 라우팅합니다.
2. **결정적 검증:** merge 게이트는 주관적인 코멘트가 아닌 재현 가능한 검사에 의존합니다.
3. **리스크 기반 리뷰 깊이:** 고위험 경로는 심층 리뷰를 트리거하고, 저위험 경로는 빠르게 처리됩니다.
4. **롤백 우선 merge 계약:** 모든 merge 경로에 구체적인 복구 단계가 포함됩니다.

자동화는 분류와 가드레일을 지원하지만, 최종 merge 책임은 사람 메인테이너와 PR 작성자에게 있습니다.

## 3. 필수 저장소 설정

`master`에서 다음 브랜치 보호 규칙을 유지합니다:

- merge 전 상태 검사를 요구합니다.
- `CI Required Gate` 검사를 요구합니다.
- merge 전 pull request 리뷰를 요구합니다.
- 보호된 경로에 대해 CODEOWNERS 리뷰를 요구합니다.
- `.github/workflows/**`의 경우, `CI Required Gate` (`WORKFLOW_OWNER_LOGINS`)를 통한 소유자 승인을 요구하고 브랜치/규칙셋 우회를 조직 소유자로 제한합니다.
- 기본 워크플로우 소유자 허용 목록은 `WORKFLOW_OWNER_LOGINS` 저장소 변수를 통해 구성됩니다 (현재 메인테이너는 CODEOWNERS 참조).
- 새 커밋이 push되면 오래된 승인을 해제합니다.
- 보호된 브랜치에서 force-push를 제한합니다.
- 모든 기여자 PR은 `master`를 직접 대상으로 합니다.

## 4. PR 생명주기 실행서

### 4.1 단계 A: 접수

- 기여자가 완전한 `.github/pull_request_template.md`로 PR을 엽니다.
- `PR Labeler`가 범위/경로 라벨 + 크기 라벨 + 리스크 라벨 + 모듈 라벨(예: `channel:telegram`, `provider:kimi`, `tool:shell`) 및 merge된 PR 수에 따른 기여자 등급(`trusted` >=5, `experienced` >=10, `principal` >=20, `distinguished` >=50)을 적용하며, 더 구체적인 모듈 라벨이 있을 때 덜 구체적인 범위 라벨을 중복 제거합니다.
- 모든 모듈 접두사에 대해, 노이즈를 줄이기 위해 모듈 라벨이 압축됩니다: 하나의 구체적 모듈은 `prefix:component`를 유지하지만, 여러 구체적 모듈은 기본 범위 라벨 `prefix`로 축소됩니다.
- 라벨 순서는 우선순위 기반입니다: `risk:*` -> `size:*` -> 기여자 등급 -> 모듈/경로 라벨.
- 메인테이너는 `PR Labeler`를 수동으로(`workflow_dispatch`) `audit` 모드에서 드리프트 가시성을 위해, 또는 `repair` 모드에서 저장소 전체의 관리 라벨 메타데이터를 정규화하기 위해 실행할 수 있습니다.
- GitHub에서 라벨 위에 마우스를 올리면 자동 관리되는 설명(규칙/임계값 요약)이 표시됩니다.
- 관리되는 라벨 색상은 긴 라벨 행에서 부드러운 그라데이션을 만들기 위해 표시 순서에 따라 배열됩니다.
- `PR Auto Responder`가 첫 기여자 안내를 게시하고, 라벨 기반 라우팅을 처리하며, `PR Labeler`와 동일한 임계값(`trusted` >=5, `experienced` >=10, `principal` >=20, `distinguished` >=50)을 사용하여 이슈 기여자 등급을 자동 적용합니다.

### 4.2 단계 B: 검증

- `CI Required Gate`가 merge 게이트입니다.
- 문서 전용 PR은 빠른 경로를 사용하여 무거운 Rust 작업을 건너뜁니다.
- 비문서 PR은 lint, 테스트 및 릴리스 빌드 스모크 검사를 통과해야 합니다.
- Rust에 영향을 미치는 PR은 `master` push와 동일한 필수 게이트 세트를 사용합니다 (PR 전용 빌드 단축 없음).

### 4.3 단계 C: 리뷰

- 리뷰어는 리스크 및 크기 라벨에 따라 우선순위를 정합니다.
- 보안에 민감한 경로(`src/security`, `src/runtime`, `src/gateway` 및 CI 워크플로우)는 메인테이너의 주의가 필요합니다.
- 대형 PR(`size: L`/`size: XL`)은 강력한 정당성이 없는 한 분할해야 합니다.

### 4.4 단계 D: Merge

- 히스토리를 간결하게 유지하기 위해 **squash merge**를 선호합니다.
- PR 제목은 Conventional Commit 스타일을 따라야 합니다.
- 롤백 경로가 문서화된 경우에만 merge합니다.

## 5. PR 준비 상태 계약 (DoR / DoD)

### 5.1 리뷰 요청 전 준비 완료 정의(DoR)

- PR 템플릿이 완전히 작성되었습니다.
- 범위 경계가 명시적입니다 (변경된 것 / 변경되지 않은 것).
- 검증 증거가 첨부되었습니다 ("CI가 확인할 것"이 아닌).
- 위험한 경로에 대해 보안 및 롤백 필드가 작성되었습니다.
- 개인정보/데이터 위생 검사가 완료되었고 테스트 언어가 중립적/프로젝트 범위입니다.
- 테스트/예제에 정체성과 관련된 문구가 있는 경우, ZeroClaw/프로젝트 고유 라벨로 정규화되었습니다.

### 5.2 merge 준비 완료 정의(DoD)

- `CI Required Gate`가 녹색입니다.
- 필수 리뷰어가 승인했습니다 (CODEOWNERS 경로 포함).
- 리스크 클래스 라벨이 변경된 경로와 일치합니다.
- 마이그레이션/호환성 영향이 문서화되었습니다.
- 롤백 경로가 구체적이고 빠릅니다.

## 6. PR 크기 및 배치 정책

### 6.1 크기 등급

- `size: XS` <= 80 변경 라인
- `size: S` <= 250 변경 라인
- `size: M` <= 500 변경 라인
- `size: L` <= 1000 변경 라인
- `size: XL` > 1000 변경 라인

### 6.2 정책

- 기본적으로 `XS/S/M`을 목표로 합니다.
- `L/XL` PR은 명시적인 정당성과 더 엄격한 테스트 증거가 필요합니다.
- 큰 기능이 불가피한 경우, 스택형 PR로 분할합니다.

### 6.3 자동화 동작

- `PR Labeler`가 유효 변경 라인에서 `size:*` 라벨을 적용합니다.
- 문서 전용/lockfile 중심 PR은 크기 인플레이션을 방지하기 위해 정규화됩니다.

## 7. AI/에이전트 기여 정책

AI 보조 PR은 환영하며, 리뷰도 에이전트 보조가 가능합니다.

### 7.1 필수

1. 범위 경계가 명확한 PR 요약.
2. 명시적인 테스트/검증 증거.
3. 위험한 변경에 대한 보안 영향 및 롤백 메모.

### 7.2 권장

1. 자동화가 변경에 실질적으로 영향을 미친 경우 간략한 도구/워크플로우 메모.
2. 재현성을 위한 선택적 프롬프트/계획 스니펫.

기여자에게 AI 대 인간 라인 소유권을 정량화하도록 **요구하지 않습니다**.

### 7.3 AI 중심 PR에 대한 리뷰 강조 사항

- 계약 호환성.
- 보안 경계.
- 오류 처리 및 폴백 동작.
- 성능 및 메모리 회귀.

## 8. 리뷰 SLA 및 큐 규율

- 첫 메인테이너 분류 목표: 48시간 이내.
- PR이 차단된 경우, 메인테이너가 하나의 실행 가능한 체크리스트를 남깁니다.
- `stale` 자동화는 큐를 건강하게 유지하는 데 사용됩니다; 필요 시 메인테이너가 `no-stale`을 적용할 수 있습니다.
- `pr-hygiene` 자동화는 12시간마다 열린 PR을 확인하고, 48시간 이상 새 커밋이 없으며 `master`보다 뒤처져 있거나 헤드 커밋에서 `CI Required Gate`가 누락/실패한 경우 알림을 게시합니다.

### 8.1 큐 예산 제어

- 리뷰 큐 예산을 사용합니다: 메인테이너당 동시 심층 리뷰 PR 수를 제한하고 나머지는 분류 상태로 유지합니다.
- 스택형 작업의 경우, 리뷰 순서가 결정적이 되도록 명시적인 `Depends on #...`을 요구합니다.

### 8.2 백로그 압력 제어

- 새 PR이 기존 열린 PR을 대체하는 경우, `Supersedes #...`를 요구하고 메인테이너 확인 후 기존 PR을 닫습니다.
- 휴면/중복 PR에 `stale-candidate` 또는 `superseded`를 표시하여 중복 리뷰 노력을 줄입니다.

### 8.3 이슈 분류 규율

- `r:needs-repro`는 불완전한 버그 리포트용입니다 (심층 분류 전 결정적 재현을 요청합니다).
- `r:support`는 버그 백로그 외부에서 처리하는 것이 나은 사용법/도움 항목용입니다.
- `invalid` / `duplicate` 라벨은 안내와 함께 **이슈 전용** 자동 닫기를 트리거합니다.

### 8.4 자동화 부작용 방지

- `PR Auto Responder`가 라벨 기반 코멘트를 중복 제거하여 스팸을 방지합니다.
- 자동 닫기 경로는 이슈에만 제한되며, PR에는 적용되지 않습니다.
- 컨텍스트에 따라 수동 오버라이드가 필요한 경우 메인테이너가 `risk: manual`로 자동 리스크 재계산을 동결할 수 있습니다.

## 9. 보안 및 안정성 규칙

이러한 영역의 변경은 더 엄격한 리뷰와 더 강력한 테스트 증거가 필요합니다:

- `src/security/**`
- 런타임 프로세스 관리.
- 게이트웨이 인그레스/인증 동작 (`src/gateway/**`).
- 파일시스템 접근 경계.
- 네트워크/인증 동작.
- GitHub 워크플로우 및 릴리스 파이프라인.
- 실행 기능이 있는 도구 (`src/tools/**`).

### 9.1 위험한 PR의 최소 요건

- 위협/리스크 설명.
- 완화 메모.
- 롤백 단계.

### 9.2 고위험 PR 권장 사항

- 경계 동작을 증명하는 집중 테스트를 포함합니다.
- 하나의 명시적인 장애 모드 시나리오와 예상되는 성능 저하를 포함합니다.

에이전트 보조 기여의 경우, 리뷰어는 작성자가 런타임 동작과 영향 범위에 대한 이해를 보여주는지도 확인해야 합니다.

## 10. 장애 복구 프로토콜

merge된 PR이 회귀를 일으킨 경우:

1. `master`에서 즉시 revert PR을 생성합니다.
2. 근본 원인 분석이 포함된 후속 이슈를 엽니다.
3. 회귀 테스트와 함께만 수정을 재도입합니다.

지연된 완벽한 수정보다 빠른 서비스 품질 복구를 선호합니다.

---

## 11. 메인테이너 Merge 체크리스트

- 범위가 집중적이고 이해하기 쉽습니다.
- CI 게이트가 녹색입니다.
- 문서가 변경된 경우 문서 품질 검사가 녹색입니다.
- 보안 영향 필드가 완성되었습니다.
- 개인정보/데이터 위생 필드가 완성되었고 증거가 수정/익명화되었습니다.
- 자동화가 사용된 경우 에이전트 워크플로우 메모가 재현성에 충분합니다.
- 롤백 계획이 명시적입니다.
- 커밋 제목이 Conventional Commits를 따릅니다.

## 12. 에이전트 리뷰 운영 모델

대량 PR에서 리뷰 품질을 안정적으로 유지하기 위해 2레인 리뷰 모델을 사용합니다.

### 12.1 레인 A: 빠른 분류 (에이전트 친화적)

- PR 템플릿 완성도를 확인합니다.
- CI 게이트 신호를 확인합니다 (`CI Required Gate`).
- 라벨과 변경된 경로를 통해 리스크 클래스를 확인합니다.
- 롤백 설명이 존재하는지 확인합니다.
- 개인정보/데이터 위생 섹션과 중립적 문구 요건이 충족되는지 확인합니다.
- 필요한 정체성 관련 문구가 ZeroClaw/프로젝트 고유 용어를 사용하는지 확인합니다.

### 12.2 레인 B: 심층 리뷰 (리스크 기반)

고위험 변경(보안/런타임/게이트웨이/CI)에 필요합니다:

- 위협 모델 가정을 검증합니다.
- 장애 모드 및 성능 저하 동작을 검증합니다.
- 하위 호환성 및 마이그레이션 영향을 검증합니다.
- 관측성/로깅 영향을 검증합니다.

## 13. 큐 우선순위 및 라벨 규율

### 13.1 분류 순서 권장

1. `size: XS`/`size: S` + 버그/보안 수정.
2. `size: M` 집중 변경.
3. `size: L`/`size: XL` 분할 요청 또는 단계적 리뷰.

### 13.2 라벨 규율

- 경로 라벨은 서브시스템 소유권을 빠르게 식별합니다.
- 크기 라벨은 배치 전략을 결정합니다.
- 리스크 라벨은 리뷰 깊이를 결정합니다 (`risk: low/medium/high`).
- 모듈 라벨(`<module>: <component>`)은 통합별 변경 및 향후 새로 추가되는 모듈에 대한 리뷰어 라우팅을 개선합니다.
- `risk: manual`은 자동화에 컨텍스트가 부족한 경우 메인테이너가 수동 리스크 판단을 보존할 수 있게 합니다.
- `no-stale`은 수락되었지만 차단된 작업에 예약됩니다.

## 14. 에이전트 핸드오프 계약

한 에이전트가 다른 에이전트(또는 메인테이너)에게 핸드오프할 때 다음을 포함합니다:

1. 범위 경계 (변경된 것 / 변경되지 않은 것).
2. 검증 증거.
3. 열린 리스크와 미지의 사항.
4. 제안된 다음 조치.

이를 통해 컨텍스트 손실을 줄이고 반복적인 심층 분석을 방지합니다.
