# 변경 플레이북

ZeroClaw의 일반적인 확장 및 수정 패턴에 대한 단계별 가이드입니다.

각 확장 trait의 완전한 코드 예제는 [extension-examples.md](./extension-examples.md)를 참조합니다.

## 프로바이더 추가

- `src/providers/`에서 `Provider`를 구현합니다.
- `src/providers/mod.rs` 팩토리에 등록합니다.
- 팩토리 연결 및 오류 경로에 대한 집중 테스트를 추가합니다.
- 프로바이더 고유 동작이 공유 오케스트레이션 코드로 유출되지 않도록 합니다.

## 채널 추가

- `src/channels/`에서 `Channel`을 구현합니다.
- `send`, `listen`, `health_check`, 타이핑 시맨틱스를 일관성 있게 유지합니다.
- 인증/허용 목록/헬스 동작을 테스트로 커버합니다.

## 도구 추가

- `src/tools/`에서 엄격한 파라미터 스키마와 함께 `Tool`을 구현합니다.
- 모든 입력을 검증하고 살균합니다.
- 구조화된 `ToolResult`를 반환합니다; 런타임 경로에서 panic을 피합니다.

## 주변기기 추가

- `src/peripherals/`에서 `Peripheral`을 구현합니다.
- 주변기기는 `tools()`를 노출합니다 — 각 도구는 하드웨어(GPIO, 센서 등)에 위임합니다.
- 필요 시 설정 스키마에 보드 유형을 등록합니다.
- 프로토콜 및 펌웨어 참고사항은 `docs/hardware/hardware-peripherals-design.md`를 참조합니다.

## 보안 / 런타임 / 게이트웨이 변경

- 위협/리스크 메모와 롤백 전략을 포함합니다.
- 장애 모드 및 경계에 대한 테스트 또는 검증 증거를 추가/업데이트합니다.
- 관측성을 유용하지만 민감하지 않게 유지합니다.
- `.github/workflows/**` 변경의 경우, PR 메모에 Actions 허용 목록 영향을 포함하고 소스가 변경될 때 `docs/contributing/actions-source-policy.md`를 업데이트합니다.

## 문서 시스템 / README / IA 변경

- 문서 탐색을 제품 UX로 취급합니다: README -> docs hub -> SUMMARY -> 카테고리 인덱스로의 명확한 경로를 보존합니다.
- 최상위 탐색을 간결하게 유지합니다; 인접 탐색 블록 간 중복 링크를 피합니다.
- 런타임 영역이 변경되면 `docs/reference/`의 관련 참조를 업데이트합니다.
- 탐색 또는 주요 문구가 변경될 때 지원되는 모든 로케일(`en`, `zh-CN`, `ja`, `ru`, `fr`, `vi`)에 대해 다국어 진입점 동등성을 유지합니다.
- 공유 문서 문구가 변경되면 동일 PR에서 해당 현지화 문서를 동기화합니다 (또는 명시적으로 연기를 문서화하고 후속 PR을 작성합니다).

## 도구 공유 상태

- 장기 공유 상태를 소유하는 도구에 대해 `Arc<RwLock<T>>` 핸들 패턴을 따릅니다.
- 생성 시 핸들을 받습니다; 전역/정적 가변 상태를 생성하지 않습니다.
- 클라이언트별 상태에 네임스페이스를 지정하기 위해 `ClientId` (데몬이 제공)를 사용합니다 — 도구 내부에서 식별 키를 생성하지 않습니다.
- 보안에 민감한 상태(자격 증명, 할당량)를 클라이언트별로 격리합니다; 브로드캐스트/표시 상태는 선택적 네임스페이스 접두사로 공유할 수 있습니다.
- 캐시된 검증은 설정 변경 시 무효화됩니다 — 도구는 신호를 받으면 다음 실행 전에 재검증해야 합니다.
- 전체 계약은 [ADR-004: Tool Shared State Ownership](../../../architecture/adr-004-tool-shared-state-ownership.md)을 참조합니다.

## 아키텍처 경계 규칙

- trait 구현 + 팩토리 연결 추가로 기능을 확장합니다; 격리된 기능에 대한 크로스 모듈 재작성을 피합니다.
- 의존성 방향을 계약 방향으로 유지합니다: 구체적 통합이 trait/config/util 계층에 의존하며, 다른 구체적 통합에 의존하지 않습니다.
- 크로스 서브시스템 결합을 피합니다 (예: 프로바이더 코드가 채널 내부를 임포트하거나, 도구 코드가 게이트웨이 정책을 직접 변경).
- 모듈 책임을 단일 목적으로 유지합니다: `agent/`에서 오케스트레이션, `channels/`에서 전송, `providers/`에서 모델 I/O, `security/`에서 정책, `tools/`에서 실행.
- 반복적 사용(3회 규칙) 후에만 새로운 공유 추상화를 도입하며, 최소 하나의 실제 호출자가 있어야 합니다.
- config/스키마 변경의 경우, 키를 공개 계약으로 취급합니다: 기본값, 호환성 영향, 마이그레이션/롤백 경로를 문서화합니다.
