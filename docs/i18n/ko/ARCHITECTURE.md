# MoA — 아키텍처 및 제품 비전

> **날짜**: 2026-03-15
> **상태**: 살아있는 문서 — 주요 기능 마일스톤마다 업데이트
> **대상**: AI 리뷰어 (Gemini, Claude), 인간 기여자, 향후 유지보수 담당자

---

## 1. 제품 비전

### MoA란?

**MoA (Mixture of Agents)** 는 각 사용자의 기기에서 **독립적으로 실행되는**
크로스 플랫폼 AI 개인 비서 앱입니다 — 데스크톱 (Windows, macOS, Linux / Tauri)
및 모바일 (iOS, Android). 각 MoA 앱 인스턴스에는 자체 로컬 SQLite
데이터베이스를 갖춘 완전한 **ZeroClaw 자율 에이전트 런타임**이 포함되어
있습니다. 동일 사용자의 여러 기기는 경량 릴레이 서버를 통해 **실시간으로 장기
기억을 동기화**하며, 서버에 기억을 영구 저장하지 않습니다 (특허: 서버
비저장 E2E 암호화 메모리 동기화).

MoA는 여러 AI 모델을 협업적으로 결합하여 7가지 작업 카테고리에서 결과를
제공합니다 — 특히 **실시간 동시 통역**과 **AI 협업 코딩**에 중점을 둡니다.

### 핵심 논리

> 단일 모델 AI는 한계가 있습니다. 최상의 결과는 여러 전문 AI 모델이
> **협업하고, 리뷰하고, 서로의 작업을 개선**할 때 나옵니다 —
> 마치 인간 전문가 팀처럼.

이 "에이전트 혼합" 철학은 모든 곳에 적용됩니다:
- **코딩**: Claude Opus 4.6이 코드 작성 → Gemini 3.1 Pro가 아키텍처 리뷰
  → Claude가 Gemini 피드백 검증 → 합의 기반 품질
- **통역**: Gemini Live가 실시간 오디오 처리 → 세그멘테이션 엔진이 구문 단위
  청크 커밋 → 번역이 연속적으로 스트리밍
- **일반 작업**: 로컬 SLM (게이트키퍼)이 간단한 질의 처리 → 클라우드 LLM이
  복잡한 것 처리 → 라우팅이 비용/지연시간 최적화
- **기억**: 각 기기가 독립적으로 작동하지만 모든 기억이 델타 기반 E2E 암호화
  동기화를 통해 수렴

## 2. 배포 아키텍처

### 사용자별, 기기별, 독립 앱

```
┌─────────────────────────────────────────────────────────────────┐
│                        사용자 "Alice"                           │
│                                                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│  │  데스크톱 앱  │  │  모바일 앱   │  │  모바일 앱           │  │
│  │  (Tauri/Win)  │  │  (Android)   │  │  (iOS)               │  │
│  │              │  │              │  │                      │  │
│  │  ZeroClaw    │  │  ZeroClaw    │  │  ZeroClaw            │  │
│  │  + SQLite    │  │  + SQLite    │  │  + SQLite            │  │
│  │  + sqlite-vec│  │  + sqlite-vec│  │  + sqlite-vec        │  │
│  │  + FTS5      │  │  + FTS5      │  │  + FTS5              │  │
│  └──────┬───────┘  └──────┬───────┘  └──────────┬───────────┘  │
│         │                 │                      │              │
│         └────────┬────────┴──────────────────────┘              │
│                  │ E2E 암호화 델타 동기화                       │
│                  ▼                                              │
│         ┌────────────────┐                                     │
│         │ Railway 릴레이  │  ← 5분 TTL 버퍼만                  │
│         │ 서버            │  ← 영구 기억 저장 없음              │
│         └────────────────┘                                     │
└─────────────────────────────────────────────────────────────────┘
```

**핵심 원칙:**
1. 각 MoA 앱 인스턴스는 **독립적으로 작동** — 일반 AI 작업에 서버 불필요
2. 각 기기에 **자체 SQLite 장기 기억** (sqlite-vec 임베딩, FTS5 전문 검색)
3. 기억 동기화는 **릴레이를 통한 P2P** — 릴레이 서버는 최대 **5분**만 보관 후 삭제
4. 사용자는 **여러 기기에 MoA 설치** 가능 — 모두 실시간 동기화로 같은 기억 공유
5. **일반 AI 작업은 릴레이 서버를 거치지 않음** — 앱이 기기에서 직접 LLM API 호출 (사용자 키 사용 시)
6. **MoA = 하나의 GUI 앱** — ZeroClaw 런타임은 모든 MoA 설치 파일에 사이드카
   바이너리로 번들됨

### LLM API 키 모델 — 3단계 프로바이더 접근 모드

MoA는 **3단계 프로바이더 접근 모델**을 사용하여 LLM 호출의 라우팅, 과금,
사용 모델을 결정합니다.

#### 단계 개요

```
┌─────────────────────────────────────────────────────────────────────┐
│  3단계 프로바이더 접근 모델                                         │
│                                                                     │
│  ① UserKey 모드 (유저 자체 키 모드)                                 │
│     조건: 사용자가 자체 API 키를 제공한 경우                         │
│     → 앱이 기기에서 LLM 프로바이더에 직접 호출                       │
│     → 사용자가 사용할 모델 선택 (최신 최고급 모델 사용 가능)          │
│     → 크레딧 차감 없음 (사용자가 프로바이더에 직접 결제)              │
│     → LLM 호출에 Railway 릴레이 개입 없음                            │
│                                                                     │
│  ② Platform Selected 모드 (플랫폼 모델 선택 모드)                   │
│     조건: API 키 없음 + 사용자가 수동으로 모델 선택                   │
│     → LLM 호출이 Railway 릴레이를 통해 라우팅 (운영자 API 키)        │
│     → 사용자가 선택한 모델 사용                                      │
│     → 실제 API 비용의 2.2배로 크레딧 차감 (2배 마진 + 부가세)        │
│                                                                     │
│  ③ Platform Default 모드 (플랫폼 기본 모드)                         │
│     조건: API 키 없음 + 모델 미선택 (신규 사용자)                    │
│     → LLM 호출이 Railway 릴레이를 통해 라우팅 (운영자 API 키)        │
│     → 작업 유형별 자동 모델 라우팅 (아래 표 참조)                     │
│     → 실제 API 비용의 2.2배로 크레딧 차감 (2배 마진 + 부가세)        │
│     → 신규 사용자는 가입 시 보너스 크레딧 지급                        │
└─────────────────────────────────────────────────────────────────────┘
```

#### 접근 모드 결정 표

| 모드 | 조건 | LLM 호출 경로 | 모델 선택 | 과금 |
|------|------|--------------|----------|------|
| **UserKey** | 사용자가 API 키 제공 | 기기 → 프로바이더 직접 | 사용자 선택 (최고급 모델) | 무료 (사용자가 프로바이더에 직접 결제) |
| **Platform (선택)** | API 키 없음 + 모델 선택 | Railway 릴레이 (운영자 키) | 사용자 선택 모델 | 실제 API 비용의 2.2배 크레딧 |
| **Platform (기본)** | API 키 없음 + 미선택 | Railway 릴레이 (운영자 키) | 작업별 자동 라우팅 | 실제 API 비용의 2.2배 크레딧 |

#### 작업별 기본 모델 라우팅 (Platform Default 모드)

사용자에게 API 키가 없고 모델을 선택하지 않은 경우, 시스템이 작업 유형에 따라
가장 적합한 모델로 자동 라우팅합니다:

| 작업 카테고리 | 프로바이더 | 기본 모델 | 이유 |
|--------------|-----------|----------|------|
| **일반 채팅** | Gemini | `gemini-3.1-flash-lite-preview` | 일상 대화에 가장 비용 효율적 |
| **추론/문서** | Gemini | `gemini-3.1-pro-preview` | 고품질 추론 및 문서 분석 |
| **코딩** | Anthropic | `claude-opus-4-20250514` | 최고 수준 코드 생성 |
| **코드 리뷰** | Gemini | `gemini-3.1-pro-preview` | 아키텍처 인식 리뷰 |
| **이미지** | Gemini | `gemini-3.1-flash-lite-preview` | 비용 효율적 비전 작업 |
| **음악** | Gemini | `gemini-3.1-flash-lite-preview` | 경량 오케스트레이션 |
| **비디오** | Gemini | `gemini-3.1-flash-lite-preview` | 경량 오케스트레이션 |
| **통역** | Gemini | Gemini 2.5 Flash Live API | 실시간 음성 스트리밍 |

#### 크레딧 시스템 및 과금 로직

```
┌─────────────────────────────────────────────────────────────────────┐
│  크레딧 과금 흐름 (Platform 모드만 해당)                            │
│                                                                     │
│  1. 신규 사용자 등록 → 가입 보너스 크레딧 지급                      │
│     (약 $3~5 상당의 사용량)                                         │
│                                                                     │
│  2. 각 LLM API 호출:                                               │
│     실제_API_비용_USD = (입력_토큰 × 입력_가격/1M)                  │
│                       + (출력_토큰 × 출력_가격/1M)                  │
│     차감_크레딧 = 실제_API_비용_USD × 2.2                           │
│     (2.0배 운영자 마진 + 10% 부가세 = 2.2배)                       │
│                                                                     │
│  3. 매 차감 전 잔액 확인:                                           │
│     ├─ 잔액 > 경고_임계값  → 조용히 진행                           │
│     ├─ 잔액 ≤ 경고_임계값  → 경고 알림 표시:                       │
│     │   "크레딧이 부족합니다. 충전하시거나 직접 API 키를 입력하세요" │
│     │   → 옵션 A: 크레딧 추가 구매 (결제)                          │
│     │   → 옵션 B: 직접 API 키 입력 (설정 → API 키)                 │
│     │     지원 프로바이더: Claude, OpenAI, Gemini (3종)             │
│     └─ 잔액 = 0  → 요청 차단, 충전 또는 API 키 입력 필요           │
│                                                                     │
│  4. 사용자는 언제든지 자체 API 키 입력 가능:                        │
│     → Claude (Anthropic) API 키                                     │
│     → OpenAI API 키                                                 │
│     → Gemini (Google) API 키                                        │
│     키를 입력하면 해당 프로바이더 호출이 UserKey 모드로 전환          │
│     (크레딧 차감 없음, 기기에서 프로바이더로 직접 호출)              │
└─────────────────────────────────────────────────────────────────────┘
```

#### Railway 릴레이 vs 직접 API 호출

```
┌─────────────────────────────────────────────────────────────────────┐
│  Railway 릴레이를 사용하는 경우 (LLM 호출)                          │
│                                                                     │
│  Railway 릴레이 (운영자 API 키):                                    │
│  ├─ 사용자에게 해당 프로바이더의 API 키가 없는 경우                  │
│  ├─ LLM 요청이 Railway 서버를 통해 프록시됨                         │
│  ├─ 운영자의 API 키 (ADMIN_*_API_KEY 환경변수) 사용                 │
│  ├─ 사용자 잔액에서 2.2배로 크레딧 차감                             │
│  └─ 운영자의 키는 절대 서버 밖으로 유출되지 않음                     │
│                                                                     │
│  직접 기기→프로바이더 호출 (사용자 자체 키):                         │
│  ├─ 사용자가 해당 프로바이더의 자체 API 키를 입력한 경우             │
│  ├─ 앱이 사용자 기기에서 LLM API를 직접 호출                        │
│  ├─ Railway 릴레이 개입 없음                                        │
│  ├─ 크레딧 차감 없음                                                │
│  └─ 사용자가 표준 API 요금으로 프로바이더에 직접 결제                │
│                                                                     │
│  중요: Railway 릴레이는 항상 다음에 사용됨:                         │
│  ├─ 메모리 동기화 (E2E 암호화 델타 교환) — 키 유무와 무관           │
│  └─ 원격 채널 라우팅 (카카오톡, 텔레그램 등)                        │
│  이들은 LLM 호출이 아니며 크레딧을 소비하지 않음.                    │
└─────────────────────────────────────────────────────────────────────┘
```

| 시나리오 | API 키 출처 | 경로 | 사용 모델 | 과금 |
|---------|------------|------|----------|------|
| 프로바이더 키 보유 | 사용자 자체 | 기기 → 프로바이더 직접 | 사용자 선택 (최고급) | 무료 (사용자가 프로바이더에 직접 결제) |
| 키 없음 (기본) | 운영자 (Railway 환경변수) | 기기 → Railway 릴레이 → 프로바이더 | 작업별 자동 라우팅 | 실제 API 비용의 2.2배 크레딧 |
| 키 없음 (모델 선택) | 운영자 (Railway 환경변수) | 기기 → Railway 릴레이 → 프로바이더 | 사용자 선택 모델 | 실제 API 비용의 2.2배 크레딧 |
| 음성 통역 | 사용자 또는 운영자 | 위와 동일한 규칙 | Gemini 2.5 Flash Live API | 위와 동일한 규칙 |

### 채널을 통한 원격 접속

사용자는 메시징 채널을 통해 **어느 기기에서든** (MoA가 설치되지 않은 기기
포함) MoA 앱과 상호작용할 수 있습니다:

**지원 채널:**
- **카카오톡** (MoA 추가 — 업스트림 ZeroClaw에 없음)
- 텔레그램
- 디스코드
- 슬랙
- LINE
- 웹 채팅 (홈페이지)

### 통합 앱 경험 (MoA + ZeroClaw = 하나의 앱)

> **필수 요구사항**: MoA와 ZeroClaw는 최종 사용자에게 **분리 불가능한 단일
> 애플리케이션**으로 보여야 합니다. 사이드카 아키텍처는 사용자 경험에 절대
> 노출되지 않는 내부 구현 세부사항입니다.

#### 원칙

1. **하나의 다운로드, 하나의 설치, 하나의 앱**
2. **제3자가 분리 불가** — 사이드카 바이너리가 앱 패키지 내부에 번들됨
3. **자동 수명 주기 관리** — 앱 실행 시 ZeroClaw 자동 시작, 종료 시 자동 종료
4. **통합 업데이트** — 프론트엔드와 ZeroClaw 바이너리가 함께 업데이트
5. **단일 설정 흐름** — 모든 ZeroClaw 설정이 MoA GUI를 통해 구성

## 3. 특허: 서버 비저장 E2E 암호화 메모리 동기화

### 제목 (발명의 명칭)

**서버 비저장 방식의 다중 기기 간 종단간 암호화 메모리 동기화 시스템 및 방법**

### 문제 정의

기존 클라우드 동기화 방식은 중앙 서버에 사용자 데이터를 영구 저장하여:
- 프라이버시 리스크 (서버 침해 시 전체 사용자 데이터 노출)
- 단일 장애점
- 규정 준수 부담 (GDPR, 데이터 거주지)
- 사용자 수에 비례하는 서버 저장소 비용

### 발명 요약

각 사용자 기기가 로컬 SQLite 데이터베이스에 **자체 권한 있는 복사본**을
유지하고, **데이터를 영구 저장하지 않는 릴레이 서버**를 통해 다른 기기와
**변경사항(델타)을 동기화**하는 시스템.

### 핵심 메커니즘

1. **델타 기반 동기화** — 전체 기억 저장소가 아닌 변경사항(델타)만 전송
2. **종단간 암호화** — 릴레이 서버가 콘텐츠를 해독할 수 없음
3. **서버 TTL 버퍼 (5분)** — 암호화된 델타를 최대 5분 보관 후 영구 삭제
4. **오프라인 조정** — 5분 이상 오프라인 기기는 P2P 전체 조정 수행
5. **충돌 해결** — 최후 기록 우선 (타임스탬프 기준)

### 보안 속성

1. **제로 지식 릴레이**: 서버가 어떤 데이터도 해독 불가
2. **순방향 비밀성**: 동기화 세션별 키 로테이션
3. **기기 침해 격리**: 하나의 기기 침해가 다른 기기의 키에 영향 없음
4. **삭제 보장**: 서버 데이터는 일시적 (5분 TTL)
5. **서버 측 백업 없음**: 사용자 데이터의 "클라우드 복사본" 없음

## 8. 코딩 / 멀티모델 리뷰 파이프라인

### 목표

**Claude Opus 4.6이 코드를 작성**하고 **Gemini 3.1 Pro가 아키텍처 정합성을
리뷰**한 후, Claude가 Gemini의 발견 사항을 검증하는 자율 코딩 어시스턴트 —
AI 대 AI 협업을 통해 셀프 체크된 고품질 코드 생산.

### 파이프라인

```
코드 diff ──┬─▸ GeminiReviewer ─▸ ReviewReport ─┐
            │   (아키텍처 게이트키퍼)             │
            │   Gemini 3.1 Pro                   ▼
            └─▸ ClaudeReviewer ──────────────────┼─▸ ConsensusReport
                (Gemini의 발견사항 확인,         │
                 검증 또는 반박)                   │
                Claude Opus 4.6                  ▼
                               발견사항 병합 + 합의 판정
```

### 코딩 장기 기억 (MoA 차별점)

**핵심 차별화**: Claude Code, Cursor 등 다른 AI 코딩 도구가 컨텍스트 윈도우
한계로 **세션 간 모든 것을 잊는** 반면, MoA는 **모든 코딩 활동을 장기 기억에
유지**하고 사용자의 모든 기기에 **실시간 동기화**합니다.

## 12. 설계 원칙

| 원칙 | 규칙 |
|------|------|
| **KISS** | 영리한 메타 프로그래밍보다 직관적인 제어 흐름 |
| **YAGNI** | 추측적 기능 금지 — 구체적인 승인된 사용 사례 필요 |
| **DRY + 3의 법칙** | 3회 이상 반복 후에만 공유 로직 추출 |
| **SRP + ISP** | 모듈당 하나의 관심사, 좁은 트레이트 인터페이스 |
| **빠른 실패** | 지원되지 않는 상태에 대한 명시적 에러 |
| **기본 보안** | 기본 거부, 시크릿 로깅 금지, 최소 노출 |
| **결정론** | 재현 가능한 동작, 불안정 테스트 없음 |
| **가역성** | 작은 커밋, 명확한 롤백 경로 |
