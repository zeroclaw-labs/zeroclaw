# ROS2 整合指引

本文件記錄 ROS2/ROS1 環境的建議整合方式。
文件以架構面為主，同時確保 ZeroClaw 核心邊界的穩定性。

## 建議做法

優先採用外掛/轉接器路線。

- 將機器人通訊放在整合用 crate 或模組中，負責橋接 ROS topic/service/action 與 ZeroClaw 的 tool/channel/runtime adapter。
- 將高頻控制迴路保留在 ROS 原生執行環境中。
- 使用 ZeroClaw 處理規劃、編排、策略和受防護的動作派發。

深度核心耦合應作為最後手段，且僅在經過量測後確認延遲限制無法透過橋接方式達成時才考慮。

## 為何採用此預設做法

- 升級安全性：基於 trait 的轉接器比核心補丁更能承受上游變更。
- 影響範圍控制：傳輸細節保持在安全/執行階段核心模組之外。
- 可重現性：隔離後的整合行為更容易測試和回退。
- 安全態勢：審批、策略和閘門控制仍集中在現有的 ZeroClaw 路徑中。

## 即時性邊界規則

不要將硬即時的馬達/安全迴路經由 LLM 回合延遲來路由。

- ROS 節點圖負責緊密迴路控制和看門狗。
- ZeroClaw 發送意圖層級的指令，並接收摘要狀態。
- 安全關鍵的緊急停止路徑無論 agent 健康狀態如何，都保持在機器人執行階段本地。

## 建議的基準架構

1. ROS2 橋接節點訂閱高頻率感測器 topic。
2. 橋接節點執行本地歸約/窗口化，並將精簡摘要轉送至 ZeroClaw。
3. ZeroClaw 在現有策略和審批限制下決定意圖/工具呼叫。
4. 橋接節點將核准的意圖轉換為 ROS 指令，並加上指令頻率限制。
5. 遙測和故障狀態回流至 ZeroClaw，用於推理和可稽核性。

## 核心整合的升級條件

僅在以下條件全部滿足時，才考慮更深度的 ZeroClaw 執行階段整合：

- 在類生產環境負載下，已量測並驗證橋接開銷為瓶頸。
- 所需的延遲/抖動預算已書面記錄且可重現。
- 提議的核心變更有明確的回退方案和子系統負責人。
- 安全性和策略保證維持等同或更強的水準。

若上述條件未滿足，請維持使用轉接器/外掛整合方式。
