# MoA 특허명세서 — 제1호

## AI 에이전트의 비례적 메모 치환 기반 대화 컨텍스트 최적화 방법 및 시스템

### 기술 분야

본 발명은 인공지능 대화 시스템에서 다중 세션 간 대화 연속성을 유지하면서도 LLM(대규모 언어 모델)에 전달되는 컨텍스트 토큰을 효율적으로 관리하는 방법 및 시스템에 관한 것이다.

### 배경 기술

기존의 AI 대화 시스템은 대화 기록을 LLM에 전달할 때 두 가지 극단적인 방식 중 하나를 사용한다:

1. **전문 전달 방식**: 이전 대화의 모든 텍스트를 그대로 LLM에 전달. 토큰 비용이 기하급수적으로 증가하며, 10만자 분량의 문서가 포함된 대화를 이어갈 경우 대부분의 토큰이 현재 질문과 무관한 이전 참고자료에 소비됨.
2. **요약 방식**: 이전 대화를 AI가 요약하여 전달. 대화의 뉘앙스, 표현 방식, 맥락 전환 지점이 소실되어 자연스러운 대화 연속이 불가능.

두 방식 모두 인간의 실제 대화 방식과 괴리가 있다. 인간은 대화의 흐름(인사, 질문, 감정, 맥락 전환)은 상세히 기억하되, 참고 문서나 전문 자료의 내용은 메모나 문서 형태로 별도 보관하고 필요 시 참조한다.

### 해결하고자 하는 과제

본 발명은 인간의 대화 기억 방식을 모방하여, 대화의 자연스러운 흐름은 원문 그대로 보존하면서도, 전문적/기술적 장문 콘텐츠는 비례적 메모로 치환함으로써 LLM 컨텍스트 토큰을 70-88% 절감하는 방법을 제공하는 것을 과제로 한다.

### 과제 해결 수단

#### 1. 대화 턴 분류 기준

대화의 각 턴(사용자 발화 또는 AI 응답)을 글자 수 기준으로 분류한다:

- **일상 대화 턴** (2,000자 미만): 원문 그대로 LLM 컨텍스트에 포함
- **장문 턴** (2,000자 이상): 메모 치환 대상

2,000자 기준은 인간의 일상적 대화 턴이 거의 이 범위를 초과하지 않으며, 초과 시 99% 이상이 첨부 문서, 검색 결과, 코드 블록, RAG 출력물 등 참고 자료라는 실증적 관찰에 기반한다.

#### 2. 비례적 메모 생성 (10% 규칙)

장문 턴에 대한 메모의 크기는 원문의 **10%**로 설정한다 (최소 100자, 최대 10,000자). 고정 크기 요약이 아닌 비례적 크기를 사용하는 이유:

| 원문 크기 | 메모 크기 | 용도 |
|-----------|-----------|------|
| 2,000자 | ~200자 | 간단한 전문 내용 |
| 10,000자 | ~1,000자 | 다중 항목 분석 |
| 50,000자 | ~5,000자 | 대규모 문서 |
| 100,000자 | ~10,000자 | 보고서 전문 (섹션별 요약) |

#### 3. 메모 치환 구조

장문 턴은 다음 3개 요소로 치환된다:

```
[도입부 원문 300자] — 대화의 시작, 주제 설정, 맥락
[비례적 메모]       — 섹션별 6하원칙 요약 + 콘텐츠 유형 + 말미 질문
[말미 원문 300자]   — 추가 질문, 제안, 대화 흐름 전환점
```

도입부와 말미를 원문으로 보존하는 이유: 이 부분이 대화의 흐름과 맥락 전환을 담당하며, 자연스러운 대화 연속에 필수적이기 때문이다.

#### 4. 섹션 분할 및 6W 요약

장문 콘텐츠가 다수의 주제를 포함하는 경우, 빈 줄, 마크다운 헤더(##, ###), 번호 목록(1., 2.) 등을 기준으로 논리적 섹션으로 분할하고, 각 섹션을 6하원칙(누가, 무엇을, 언제, 어디서, 왜, 어떻게)에 따라 독립적으로 요약한다.

#### 5. 온디맨드 심층 검색

메모만으로 불충분한 경우, LLM이 메모에 포함된 키워드를 사용하여 벡터 검색 또는 키워드 검색으로 원문을 즉시 검색할 수 있다. 원문은 항상 전체가 저장되어 있으므로 정보 손실이 없다.

### 발명의 효과

1. **토큰 절감**: 일반적인 대화에서 70-88%의 컨텍스트 토큰 절감
2. **대화 연속성**: 도입부/말미 원문 보존으로 자연스러운 대화 흐름 유지
3. **정보 무손실**: 원문 전체 저장 + 온디맨드 검색으로 필요 시 전문 접근 가능
4. **비례적 확장**: 콘텐츠 크기에 비례하는 메모 크기로 정보 밀도 일정하게 유지
5. **비용 절감**: LLM API 호출 비용의 대폭 절감

### 청구항

1. AI 대화 시스템에서 다중 세션 간 대화 컨텍스트를 최적화하는 방법에 있어서,
   (a) 대화의 각 턴을 미리 정해진 글자 수 기준(문턱값)에 따라 일상 대화 턴과 장문 턴으로 분류하는 단계;
   (b) 일상 대화 턴은 원문 그대로 LLM 컨텍스트에 포함하는 단계;
   (c) 장문 턴은, 원문의 도입부(제1 소정 글자 수)와 말미(제2 소정 글자 수)를 원문으로 보존하고, 중간 부분을 원문 글자 수의 소정 비율(예: 10%)에 해당하는 크기의 메모로 치환하는 단계;
   (d) 상기 메모를 생성할 때, 장문 콘텐츠를 논리적 섹션으로 분할하고, 각 섹션을 독립적으로 요약하는 단계;
   (e) 상기 장문 턴의 원문 전체를 별도의 검색 가능한 저장소에 보존하는 단계;
   를 포함하는 것을 특징으로 하는 대화 컨텍스트 최적화 방법.

2. 제1항에 있어서, 상기 메모의 크기가 원문 글자 수의 10%이며, 최소 100자 최대 10,000자의 범위 내에서 결정되는 것을 특징으로 하는 방법.

3. 제1항에 있어서, 상기 문턱값이 2,000자이며, 이는 인간의 일상적 대화 턴이 해당 문턱값을 초과하지 않는다는 경험적 관찰에 기반한 것을 특징으로 하는 방법.

4. 제1항에 있어서, 상기 메모에 콘텐츠 유형 식별자(코드, URL, 검색결과 등)와 대화 말미의 질문 또는 행동 요청을 포함하는 것을 특징으로 하는 방법.

5. 제1항에 있어서, 상기 메모만으로 사용자의 질문에 대한 응답이 불충분한 경우, LLM이 메모에 포함된 키워드를 이용하여 벡터 검색 또는 키워드 검색으로 원문을 온디맨드(on-demand)로 검색하여 LLM 컨텍스트에 추가하는 단계를 더 포함하는 것을 특징으로 하는 방법.

6. AI 대화 시스템에서 대화의 각 턴을 분류하고, 장문 턴에 대해 비례적 메모 치환을 수행하며, 원문을 검색 가능한 저장소에 보존하는 프로세서와 메모리를 포함하는 대화 컨텍스트 최적화 시스템.

## 도면의 간단한 설명

### [도 1] 메모 치환 프로세스 흐름도

```
대화 턴 입력
    │
    ▼
글자 수 < 2,000? ──Yes──→ 원문 그대로 컨텍스트에 포함
    │
   No
    ▼
원문 전체를 검색 가능 저장소에 보존
    │
    ▼
도입부 300자 추출 (원문)
    │
    ▼
콘텐츠를 논리적 섹션으로 분할
    │
    ▼
각 섹션을 6W 기준으로 요약
(전체 메모 크기 = 원문의 10%)
    │
    ▼
말미 300자 추출 (원문)
    │
    ▼
[도입부] + [메모] + [말미] → 컨텍스트에 포함
```

### [도 2] 온디맨드 심층 검색 흐름도

```
LLM이 메모 컨텍스트로 응답 생성 시도
    │
    ▼
메모만으로 충분? ──Yes──→ 응답 생성 완료
    │
   No
    ▼
메모의 키워드로 벡터/키워드 검색
    │
    ▼
원문에서 관련 부분 검색
    │
    ▼
검색된 원문을 추가 컨텍스트로 LLM에 전달
    │
    ▼
응답 생성 완료
```
