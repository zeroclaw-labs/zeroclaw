# MoA 특허명세서 — 제1호 (개정판)

## 분산 디바이스 장기기억 기반 적응형 대화 컨텍스트 구성 방법 및 시스템
## (Adaptive Context Engine: ACE)

> **기존 명칭**: AI 에이전트의 비례적 메모 치환 기반 대화 컨텍스트 최적화 방법 및 시스템
> **개정 사유**: 메모 치환(Layer 1)을 포함한 4-Layer 통합 아키텍처로 확장

### 기술 분야

본 발명은 인공지능 대화 시스템에서 (1) 다중 세션 간 대화 연속성을 유지하면서도 LLM에 전달되는 컨텍스트 토큰을 효율적으로 관리하고, (2) 로컬 디바이스의 장기기억을 실시간 벡터 검색 및 온톨로지 교차검색을 통해 관련 과거 대화만 선별적으로 컨텍스트에 포함하며, (3) 이용자의 개인 기억을 살아있는 형태로 관리(변경 감지, 망각 요청, 빈도 기반 우선순위, 핫 캐시)하는 방법 및 시스템에 관한 것이다.

### 배경 기술

기존의 AI 대화 시스템은 대화 기록을 LLM에 전달할 때 세 가지 방식 중 하나를 사용한다:

1. **전문 전달 방식** (Claude Code 등): 이전 대화의 모든 텍스트를 그대로 LLM에 전달. 토큰 비용이 기하급수적으로 증가하며, 관련 없는 과거 대화도 매 턴마다 반복 전송됨. 세션이 끝나면 대화 기록이 소멸.
2. **요약 방식** (ChatGPT 등): 이전 대화를 AI가 요약하여 전달. 대화의 뉘앙스, 표현 방식, 맥락 전환 지점이 소실되어 자연스러운 대화 연속이 불가능.
3. **고정 윈도우 방식** (대부분의 챗봇): 최근 N개 메시지만 보존하고 나머지 삭제. N개를 넘어서면 맥락이 완전히 소실됨.

세 방식 모두 다음과 같은 한계를 갖는다:
- 첨부문서, 검색결과, 도구 응답 등 대용량 참고자료가 매 턴마다 원문 반복 전송됨
- 과거의 관련된 대화는 검색할 수 없고, 세션 종료 시 소실됨
- 이용자의 개인정보 변경(이사, 이직 등)을 감지하지 못하고 구정보를 계속 참조함
- 멀티디바이스 환경에서 대화 기억이 동기화되지 않음

### 해결하고자 하는 과제

본 발명은 인간의 대화 기억 방식을 모방한 4-Layer 적응형 컨텍스트 엔진(ACE)을 제공하여:
1. 직전 대화의 뉘앙스를 100% 보존하면서도 (Layer 0)
2. 첨부문서/참고자료는 구조화된 메모로 대체하여 토큰을 절감하고 (Layer 1)
3. 로컬 장기기억의 실시간 RAG 검색으로 과거의 관련 대화를 시간 무제한으로 활용하며 (Layer 2)
4. 이용자의 기억을 살아있는 형태로 관리 (변경 감지, 망각 요청, 빈도 우선순위, 핫 캐시)

하는 것을 과제로 한다.

### 과제 해결 수단

#### Layer 0: 즉시 맥락 계층 (Immediate Context)

직전 N턴(기본 10턴)의 대화를 원문 그대로 보존한다. 이 범위는 절대 압축하거나 제거하지 않는다. "방금 말한 거", "아까 그거" 등의 즉시 참조를 보장하며, 대화의 뉘앙스와 감정 맥락을 완벽히 유지한다.

#### Layer 1: 첨부문서 메모 치환 계층 (Attachment Memo Substitution)

대화의 각 턴에서 다음 콘텐츠 패턴을 감지한다 (글자 수 기준이 아닌 **콘텐츠 유형 기반 판단**):
- 코드 블록 (````...````)
- 인용문 (`> ...`)
- 검색 결과 (URL 3개 이상 연속)
- 도구 응답 (JSON, 테이블)
- 명시적 첨부 (`[Document:]`, `[PDF:]` 등)

감지된 첨부(500자 이상)를 구조화된 YAML 형태의 메모로 대체한다:

```yaml
📋 첨부 메모:
  제목: {감지된 제목}
  유형: {문서/기사/검색결과/코드}
  키워드: {추출된 핵심 키워드}
  요약: {원문의 10% 비례 요약}
  원문접근: memory_recall("키워드")로 검색 가능
```

일반 대화 텍스트는 길이에 관계없이 **절대 건드리지 않는다**. 이 처리는 순수 규칙 기반 문자열 처리로 LLM 호출 없이 비용 제로로 수행된다.

#### Layer 2: 장기기억 RAG 보강 계층 (Long-Term Memory RAG Enrichment)

로컬 디바이스에 E2E 암호화된 장기기억(SQLite + sqlite-vec 벡터 인덱스 + FTS5 전문 검색)을 구비하고, 매 대화 턴마다 다음 3단계 검색을 수행한다:

1. **벡터+키워드 복합검색**: 현재 대화 주제와 의미적으로 관련된 과거 대화를 검색
2. **온톨로지 그래프 검색**: 인물/사건/장소 관계 정보를 구조화된 그래프에서 검색
3. **상호 교차검색**: 기억 검색 결과의 키워드로 온톨로지를 재검색하고, 온톨로지 결과의 키워드로 기억을 재검색하여 관련성을 심층 파악

이 검색은 로컬 SQLite에서 ms 단위로 수행되므로 비용이 제로이며, 3일 전, 1주 전, 1달 전, 1년 전의 관련 대화도 포함할 수 있다.

#### Layer 3: 예산 가드 계층 (Budget Guard)

Layer 0~2의 결과를 합산한 총 컨텍스트가 모델별 토큰 예산(기본: 컨텍스트 윈도우 100%)을 초과할 경우:
1. 교차검색 결과 → RAG 메모리 결과 → 온톨로지 결과 순으로 가장 낮은 우선순위부터 제거
2. Layer 0 (직전 10턴) + 이용자 프로필 + 이용자 지시사항은 절대 제거하지 않음
3. **제거된 기억의 일시와 주제를 이용자에게 안내**: "이런 과거 기억이 저장되어 있는데 추가로 검색해드릴까요?"

#### 기억 위생 관리 시스템 (Memory Hygiene)

**정보 변경 감지**: 이용자의 주소, 직장, 연락처, 관계 상태 등 시간에 따라 변경되는 정보가 새로 언급되면, 기존 기억과 비교하여 이용자에게 업데이트 확인을 요청한다. 확인 시 구정보를 삭제하고 신정보로 대체한다.

**선택적 망각**: 이용자가 특정 주제/인물에 대한 기억 삭제를 요청하면, 관련 기억을 일괄 검색하여 목록을 보여주고, 명시적 확인 후 완전 삭제한다.

**빈도 기반 우선순위**: 각 기억 항목의 검색 횟수(`recall_count`)를 추적하고, 자주 검색되는 기억을 RAG 검색 결과에서 우선 노출한다.

**핫 메모리 캐시**: 이용자 프로필(7개 키), 이용자 지시사항(5개 접두어: `user_instruction_*`, `user_standing_order_*`, `user_cron_*`, `user_reminder_*`, `user_schedule_*`), 빈도 상위 50개 기억을 인메모리에 캐시하여 SQLite 검색(~5ms) 대비 500배 빠른 접근(~0.01ms)을 제공한다.

### 발명의 효과

1. **토큰 절감**: 첨부문서 메모 치환(Layer 1)으로 70-88% 절감, RAG 선별 검색(Layer 2)으로 추가 절감
2. **시간 무제한 기억**: 1년 전 대화도 관련 있으면 컨텍스트에 포함 (기존: 세션 내에서만)
3. **맥락 완벽 보존**: 직전 10턴 원문 보존 + 온톨로지 관계 정보로 뉘앙스 유지
4. **정보 무손실**: 원문 전체 로컬 저장 + 온디맨드 검색으로 필요 시 전문 접근
5. **살아있는 기억**: 변경 감지/망각 요청/빈도 우선순위/핫 캐시로 기억의 정확성 유지
6. **멀티디바이스**: E2E 암호화 델타 동기화로 모든 디바이스에서 동일 기억 접근
7. **비용 효율**: Layer 1~2는 비용 제로(규칙 기반 + 로컬 검색), Layer 3는 선택적 제거

### 청구항

#### 독립항

1. AI 대화 시스템에서 LLM에 전송하는 컨텍스트를 구성하는 방법에 있어서,
   (a) 직전 N턴의 대화 원문을 무조건 보존하는 즉시 맥락 계층(Layer 0);
   (b) 각 대화 턴에서 첨부문서, 코드 블록, 검색 결과, 도구 응답을 콘텐츠 패턴 기반으로 감지하여 구조화된 메모(제목, 유형, 키워드, 비례 요약을 포함)로 대체하되, 일상 대화 텍스트는 길이에 관계없이 원문을 유지하는 첨부 메모 치환 계층(Layer 1);
   (c) 로컬 디바이스의 벡터+키워드 복합 검색 엔진 및 온톨로지 그래프를 사용하여 현재 대화 주제와 관련된 과거 대화를 시간순으로 검색하고, 기억과 온톨로지 간 상호 교차검색을 수행하여 관계 정보를 추가하는 장기기억 보강 계층(Layer 2);
   (d) 상기 (a)(b)(c) 결과의 총량이 모델별 토큰 예산을 초과할 때, 이용자 프로필 및 지시사항을 제외한 가장 낮은 우선순위의 검색 결과부터 제거하되, 제거된 기억의 일시와 주제를 이용자에게 안내하는 예산 가드 계층(Layer 3);
   을 포함하는 것을 특징으로 하는 컨텍스트 구성 방법.

2. AI 대화 시스템에서 이용자의 장기기억을 관리하는 방법에 있어서,
   (a) 이용자가 언급한 새 정보가 기존 저장 기억과 모순되는 경우 이를 감지하여 이용자에게 업데이트 확인을 요청하는 정보 변경 감지 단계;
   (b) 이용자가 특정 주제 또는 인물에 대한 기억 삭제를 요청한 경우 관련 기억을 일괄 검색하여 이용자의 명시적 확인 후 완전 삭제하는 선택적 망각 단계;
   (c) 각 기억 항목의 검색 빈도를 추적하여 자주 참조되는 기억을 우선 노출하는 빈도 기반 우선순위 단계;
   (d) 이용자 프로필 및 상시 지시사항과 빈도 상위 항목을 인메모리에 캐시하는 핫 캐시 단계;
   를 포함하는 것을 특징으로 하는 기억 관리 방법.

#### 종속항

3. 제1항에 있어서, 상기 Layer 2의 장기기억은 E2E 암호화된 델타 기반 동기화를 통해 복수의 디바이스에서 동일한 검색 결과를 제공하는 것을 특징으로 하는 방법.

4. 제1항에 있어서, 상기 Layer 1의 메모는 콘텐츠 유형(코드/인용/검색/도구/문서), 제목, 키워드, 일시, 장소, 당사자, 비례 요약(원문의 10%)을 포함하는 것을 특징으로 하는 방법.

5. 제1항에 있어서, 상기 Layer 0의 보존 범위는 설정 가능한 턴 수(기본 10턴) 및 시간 기반(기본 72시간)이며, 시간 범위 내의 대화 중 첨부문서 부분만 Layer 1에 의해 메모로 대체되는 것을 특징으로 하는 방법.

6. 제1항에 있어서, 상기 Layer 3의 예산은 사용 중인 LLM 모델의 컨텍스트 윈도우 크기에 따라 동적으로 조절되며, 3-tier 모델 라우팅(경제/표준/프리미엄)과 연동하여 작업 유형별 최적의 예산을 할당하는 것을 특징으로 하는 방법.

7. 제2항에 있어서, 상기 핫 캐시에 이용자 프로필(identity, family, work, lifestyle, communication, routine, preferences)과 상시 지시사항(instruction, standing_order, cron, reminder, schedule 접두어)이 항상 포함되며, 기억 변경 시 즉시 캐시가 무효화되는 것을 특징으로 하는 방법.

8. 제1항 및 제2항의 방법을 수행하는 프로세서, 메모리, 로컬 벡터 검색 엔진, 온톨로지 그래프, 및 E2E 암호화 동기화 모듈을 포함하는 적응형 대화 컨텍스트 구성 시스템.

## 도면의 간단한 설명

### [도 1] ACE 4-Layer 컨텍스트 구성 흐름도

```
이용자 메시지 입력
    │
    ▼
┌───────────────────────────────────────────────────────┐
│ Layer 0: 직전 10턴 원문 보존 (history[] 유지)          │
│ → 절대 압축/제거 안 함                                 │
└───────────────┬───────────────────────────────────────┘
                ▼
┌───────────────────────────────────────────────────────┐
│ Layer 1: 첨부문서 감지 (콘텐츠 패턴 기반)              │
│ → 코드/인용/검색/도구/문서 패턴? ─Yes─→ YAML 메모 생성 │
│ → 일반 대화? ─────────────────────No──→ 원문 유지      │
│ → 비용: 제로 (규칙 기반)                                │
└───────────────┬───────────────────────────────────────┘
                ▼
┌───────────────────────────────────────────────────────┐
│ Layer 2: RAG 장기기억 검색 (로컬, ms 단위)             │
│ ① 벡터+키워드 복합검색 → 관련 과거 대화                │
│ ② 온톨로지 그래프 검색 → 인물/사건/장소 관계           │
│ ③ 상호 교차검색 → 기억↔온톨로지 관련성 심화            │
│ → 비용: 제로 (로컬 SQLite)                             │
└───────────────┬───────────────────────────────────────┘
                ▼
┌───────────────────────────────────────────────────────┐
│ Layer 3: 예산 가드 (총량 > 모델 윈도우 시)             │
│ → 교차검색 → RAG → 온톨로지 순으로 제거               │
│ → 프로필 + 지시사항 + 직전 10턴은 절대 보존            │
│ → ★ 제거된 기억 목록을 이용자에게 안내                  │
│   "이런 기억이 있는데 검색해드릴까요?"                  │
└───────────────┬───────────────────────────────────────┘
                ▼
      최종 컨텍스트 + 현재 메시지 → LLM 전달
```

### [도 2] 기억 위생 관리 흐름도

```
대화 중 새 정보 감지
    │
    ▼
기존 기억과 비교 (detect_conflict)
    │
    ├─ 충돌 있음 → "기존: OO / 새: XX. 업데이트할까요?"
    │              → 확인 시: 구정보 삭제 + 신정보 저장 + 캐시 갱신
    │
    └─ 충돌 없음 → 새 기억으로 저장

망각 요청 감지 ("지워줘", "잊어줘")
    │
    ▼
관련 기억 검색 (forget_matching)
    │
    ▼
"N건의 관련 기억이 있습니다. 삭제할까요?"
    │
    ├─ 확인 → 일괄 삭제 + 캐시 무효화
    └─ 취소 → 유지
```
