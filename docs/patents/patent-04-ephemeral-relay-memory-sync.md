**서버 비저장 방식의 다중 기기 간 종단간 암호화 메모리 동기화 시스템 및 방법 — 시간-위치 기반 크로스-디바이스 이벤트 정렬을 포함**

System and Method for End-to-End Encrypted Memory Synchronization Between Multiple Devices Without Persistent Server Storage — Including Time-Location Based Cross-Device Event Ordering

## [기술분야]

본 발명은 복수의 사용자 기기 간에 장기기억 데이터를 동기화하는 기술에 관한 것으로, 보다 구체적으로는 서버에 사용자 데이터를 영구적으로 저장하지 않으면서도 기기 간 실시간 동기화, 오프라인 기기에 대한 사후 동기화, 및 장기 오프라인 기기에 대한 수동 전체 동기화를 종단간 암호화 환경에서 안정적으로 수행하는 시스템 및 방법에 관한 것이다.

## [배경기술]

### 종래 기술의 문제점

현재 복수 기기 간 데이터 동기화 기술은 크게 두 가지 방식으로 분류된다.

**제1 방식: 클라우드 중앙 저장 방식**

Apple iCloud, Google Drive 등으로 대표되는 클라우드 동기화 방식은 사용자 데이터를 중앙 서버에 영구적으로 저장하고, 각 기기가 서버로부터 데이터를 다운로드하는 구조이다. 이 방식은 동기화의 안정성이 높으나, 다음과 같은 근본적 문제를 가진다.

첫째, 서버 운영자가 사용자 데이터에 접근할 수 있어 프라이버시 침해 위험이 존재한다. 종단간 암호화를 적용하더라도 서버에 암호화된 데이터가 영구적으로 보관되므로, 암호화 키의 유출 또는 미래의 암호 해독 기술 발전에 의한 데이터 노출 위험이 상존한다.

둘째, 서버 저장 비용이 사용자 수 및 데이터 양에 비례하여 지속적으로 증가한다.

셋째, 서버 장애 시 모든 기기의 동기화가 중단되는 단일 장애점(Single Point of Failure) 문제가 있다.

**제2 방식: P2P 직접 동기화 방식**

Syncthing 등으로 대표되는 P2P 동기화 방식은 기기 간 직접 데이터를 전송하여 서버 의존도를 줄인다. 그러나 이 방식은 다음과 같은 한계를 가진다.

첫째, 두 기기가 동시에 온라인 상태여야 동기화가 가능하다. 사용자가 기기 A에서 작업한 후 기기 A를 종료하고, 나중에 기기 B를 켜면 동기화가 불가능하다.

둘째, NAT 통과(NAT traversal) 문제로 인해 네트워크 환경에 따라 기기 간 직접 연결이 불가능한 경우가 빈번하다.

셋째, 대화형 AI의 장기기억(long-term memory)과 같이 지속적으로 축적되는 데이터의 경우, 동기화 누락이 AI 서비스의 품질 저하로 직결되어 높은 신뢰성이 요구된다.

한편, 대화형 AI 에이전트의 장기기억은 단순한 에피소딕 기억(episodic memory)뿐 아니라, 사용자의 인물, 업무, 문서, 선호 등 실세계 개체와 그 관계를 구조적으로 표현하는 관계형 기억(relational memory)으로 확장되고 있다. 이러한 구조화된 관계형 기억 데이터는 종래의 단순 키-값 동기화 대상에 비하여 개체 간 참조 무결성 유지, 관계의 중복 생성 방지, 행동 이력의 순서 보장 등 추가적인 동기화 요건을 가지며, 이에 특화된 동기화 방법이 요구된다.

**제3 문제: 다중 기기 간 시간 및 위치 기반 이벤트 정렬의 부재**

복수의 기기가 서로 다른 시간대(timezone)에 위치하는 경우, 각 기기에서 기록한 이벤트의 시간 순서를 일관되게 유지하는 것이 중대한 기술적 과제가 된다. 예를 들어, 사용자가 한국의 PC(Asia/Seoul, UTC+9)에서 오후 2시에 작업하고, 출장 중 미국 뉴욕의 휴대폰(America/New_York, UTC-4)에서 밤 10시에 작업한 경우, 두 이벤트의 시간적 선후 관계를 정확히 판정하려면 각 기기의 로컬 시각이 아닌 절대적인 UTC 시각을 기준으로 정렬하여야 한다.

종래의 Palantir Foundry, Notion, Obsidian 등의 시스템은 중앙 서버가 이벤트 시각을 서버 수신 시각으로 기록하거나, 단순히 각 기기의 로컬 시각을 그대로 저장하여, 다중 시간대 환경에서의 정확한 시간 순서 재구성이 불가능하거나 중앙 서버에 의존적인 한계를 가진다.

또한, 기기의 시스템 시계가 부정확한 경우(시계 드리프트), 이벤트의 절대 시각이 왜곡되어 크로스-디바이스 타임라인의 신뢰성이 저하된다. NTP(Network Time Protocol)를 통한 시계 보정은 UDP 포트 123을 사용하여 방화벽이나 NAT 환경에서 차단되는 경우가 빈번하다.

한편, 이벤트의 발생 장소(위치) 정보는 시간 정보와 결합하여 사용자의 활동 패턴을 추론하는 데 핵심적인 역할을 함에도, 종래 기술에서는 위치를 보조적 메타데이터로만 취급하여 시간-위치 복합 정렬 및 공간-시간 패턴 탐지가 지원되지 않는다.

**종래 기술의 미해결 과제**

따라서 (1) 서버에 사용자 데이터를 영구 저장하지 않아 프라이버시를 보장하면서도, (2) 기기가 동시에 온라인이 아닌 상황에서도 안정적인 동기화를 수행하고, (3) 데이터의 전송 순서 보장 및 누락 방지를 달성하며, (4) 장기간 오프라인이었던 기기에 대해서도 완전한 데이터 일관성을 복원할 수 있고, (5) 서로 다른 시간대에 위치한 복수의 기기에서 발생한 이벤트를 절대적 시간 순서에 따라 일관되게 정렬하고 사용자의 홈 시간대로 통합 표시할 수 있으며, (6) 이벤트의 발생 시각과 장소를 1급 정렬 키로 활용하여 시간-위치 복합 타임라인을 구성할 수 있는 기술이 요구되고 있다.

## [발명의 내용]

### [해결하고자 하는 과제]

본 발명은 상기한 종래 기술의 문제점을 해결하기 위한 것으로, 다음의 과제를 해결하는 것을 목적으로 한다.

제1 과제는, 서버에 사용자 데이터를 영구적으로 저장하지 않으면서도 비동시 온라인 상태의 복수 기기 간 데이터 동기화를 가능하게 하는 것이다.

제2 과제는, 네트워크 지연 및 전송 순서 역전 상황에서도 데이터의 올바른 순서를 보장하는 것이다.

제3 과제는, 수신 기기의 일시적 오프라인으로 인한 데이터 누락을 방지하되, 서버 저장 없이 이를 달성하는 것이다.

제4 과제는, 장기간(소정의 보관 기간을 초과하여) 오프라인이었던 기기에 대해서도 수동 개입을 통해 완전한 데이터 일관성을 복원할 수 있게 하는 것이다.

제5 과제는, 상기 모든 과정에서 종단간 암호화를 유지하여 서버 및 네트워크 중간자가 평문 데이터에 접근할 수 없도록 하는 것이다.

제6 과제는, 서로 다른 시간대에 위치한 복수의 기기에서 발생한 이벤트를 절대적 UTC 시각 기준으로 정확히 정렬하고, 동시에 사용자의 홈 시간대로 변환하여 일관된 타임라인을 표시할 수 있게 하는 것이다.

제7 과제는, 기기의 시스템 시계 부정확(드리프트)을 검출하여, 크로스-디바이스 타임라인의 시간적 신뢰성을 보장하는 것이다.

제8 과제는, 이벤트의 발생 시각과 장소를 1급(first-class) 정렬 키로 활용하여, 시간 순서 정렬뿐 아니라 시간-위치 복합 질의(예: "지난주 서울에서 한 일", "골프장에서의 미팅 이력")를 가능하게 하는 것이다.

### [과제의 해결 수단]

상기 과제를 해결하기 위하여 본 발명은 다음의 구성을 포함하는 **3단계 계층적 동기화 시스템**을 제공한다.

#### 제1 구성: 임시 릴레이 기반 실시간 동기화 (제1 계층)

본 발명의 제1 계층은, 서버를 데이터의 일시적인 중계점(릴레이)으로만 활용하는 임시 릴레이 동기화 방식을 포함한다.

송신 기기는 동기화 대상 데이터를 종단간 암호화한 후, 소정의 유효기간(TTL; Time To Live)을 부여하여 서버에 업로드한다. 서버는 상기 유효기간이 경과한 데이터를 자동으로 삭제한다. 서버는 실시간 통신 채널(WebSocket 기반 실시간 브로드캐스트)을 통하여 수신 기기에 데이터 도착을 즉시 통지하고, 수신 기기는 유효기간 내에 데이터를 다운로드하여 복호화한다.

상기 구성에 의하여, 수신 기기가 온라인 상태인 경우 실시간에 가까운 동기화가 달성되며, 유효기간 경과 후 서버에는 사용자 데이터가 잔존하지 않는다.

#### 제2 구성: 로컬 델타 저널 기반 사후 재동기화 (제2 계층)

상기 제1 계층은 수신 기기가 유효기간 내에 온라인 상태가 아닌 경우 데이터 누락이 발생하는 문제를 가진다. 본 발명의 제2 계층은 이 문제를 해결하기 위한 로컬 델타 저널(Local Delta Journal) 기반 사후 재동기화 방식을 포함한다.

##### 제2-1 구성: 시퀀스 넘버 및 버전 벡터

각 기기는 자신이 생성하는 각 데이터 변경 단위(이하 "델타"라 함)에 대하여 단조 증가하는 시퀀스 넘버(sequence number)를 부여한다. 상기 시퀀스 넘버는 기기 식별자와 결합되어 (기기 식별자, 시퀀스 넘버) 쌍으로 각 델타를 전역적으로 고유하게 식별한다.

각 기기는 버전 벡터(Version Vector)를 유지한다. 상기 버전 벡터는 {기기 식별자 → 해당 기기로부터 마지막으로 수신한 시퀀스 넘버}의 매핑 구조를 가지며, 각 기기가 다른 모든 기기의 데이터를 어디까지 수신하였는지를 추적한다.

##### 제2-2 구성: 로컬 델타 저널

각 기기는 자신이 생성한 델타를 소정의 보관 기간(예: 30일) 동안 로컬 저장소에 보관하는 델타 저널을 유지한다. 상기 델타 저널은 서버가 아닌 기기의 로컬 저장소에만 보관되며, 암호화된 상태의 데이터를 포함한다.

상기 보관 기간이 경과한 델타는 저널에서 자동으로 삭제(pruning)되어 로컬 저장 공간을 절약한다.

##### 제2-3 구성: 순서 보장 메커니즘

수신 기기는 수신한 델타의 시퀀스 넘버를 검사하여, 기대 시퀀스(버전 벡터의 마지막 수신 시퀀스 + 1)와 일치하는 경우에만 즉시 적용한다.

기대 시퀀스보다 큰 시퀀스의 델타가 수신된 경우(즉, 갭이 발생한 경우), 해당 델타를 순서 대기 버퍼(Order Buffer)에 임시 보관하고, 누락된 시퀀스의 델타를 재전송 요청한다.

기대 시퀀스 이하의 시퀀스를 가진 델타가 수신된 경우, 이미 수신 완료된 중복 데이터로 판단하여 무시한다(멱등성 보장).

순서 대기 버퍼에 연속된 시퀀스의 델타가 축적되면, 일괄적으로 시퀀스 순서대로 적용한다(버퍼 플러시).

##### 제2-4 구성: 재연결 시 자동 재동기화 프로토콜

오프라인이었던 기기가 온라인으로 복귀하면, 데이터베이스 저장을 수반하지 않는 실시간 브로드캐스트 채널(WebSocket 브로드캐스트)을 통하여 자신의 버전 벡터를 포함한 동기화 요청(sync_request) 메시지를 브로드캐스트한다.

상기 동기화 요청을 수신한 온라인 기기는, 요청 기기의 버전 벡터와 자신의 로컬 델타 저널을 비교하여, 요청 기기가 수신하지 못한 델타를 식별한다. 식별된 누락 델타를 상기 브로드캐스트 채널을 통하여 재전송한다.

상기 브로드캐스트 채널은 순수 WebSocket 메시지 전달만을 수행하며, 메시지 내용이 서버의 데이터베이스에 저장되지 않는다. 이에 의하여 재동기화 과정에서도 서버에 사용자 데이터가 보관되지 않는다.

요청 기기는 재전송 수신 완료 후 수신 확인(ACK) 메시지를 발신하여, 양측의 버전 벡터를 일치시킨다.

##### 제2-5 구성: 상태 영속화 및 복원

각 기기는 애플리케이션 종료 시 현재의 시퀀스 넘버, 버전 벡터, 및 델타 저널을 로컬 영속 저장소(예: AsyncStorage, localStorage, SQLite)에 저장한다.

애플리케이션 재시작 시 상기 저장된 상태를 복원하여, 애플리케이션 재시작 전후의 동기화 연속성을 보장한다.

#### 제3 구성: 매니페스트 비교 기반 수동 전체 동기화 (제3 계층)

상기 제2 계층의 로컬 델타 저널은 소정의 보관 기간(예: 30일)이 경과하면 자동 삭제되므로, 보관 기간을 초과하여 오프라인이었던 기기에 대해서는 제2 계층의 사후 재동기화가 불가능하다. 본 발명의 제3 계층은 이러한 경우를 위한 수동 전체 동기화 방식을 포함한다.

##### 제3-1 구성: 매니페스트 생성 및 교환

사용자가 수동 동기화를 요청하면, 요청 기기는 자신이 보유한 모든 엔티티(장기기억 청크, 대화 세션, 설정 등)의 식별자 목록인 매니페스트(Manifest)를 생성한다.

상기 매니페스트를 브로드캐스트 채널을 통하여 전체 동기화 요청(full_sync_request) 메시지로 전송한다.

상기 요청을 수신한 온라인 기기는 자신의 매니페스트를 생성하여 응답한다.

##### 제3-2 구성: 양방향 차분 비교 및 선택적 전송

양측 기기는 수신한 상대 매니페스트와 자신의 매니페스트를 비교하여, 각각이 보유하고 상대방에게 누락된 엔티티를 식별한다.

상기 비교는 엔티티 식별자의 집합 차(set difference) 연산으로 수행되며, 기존 데이터의 재전송 없이 누락된 엔티티만을 선택적으로 전송한다.

전송은 엔티티 단위로 종단간 암호화된 상태에서 브로드캐스트 채널을 통하여 수행된다.

##### 제3-3 구성: 진행 상황 추적 및 완료 처리

수신 기기는 수신한 엔티티 수를 추적하며, 사용자에게 진행 상황(비교 중, 수신 중, 전송 중, 완료)을 표시한다.

전송 측 기기는 모든 누락 엔티티의 전송을 완료한 후 완료 알림(full_sync_complete) 메시지를 발신한다.

소정의 시간(예: 60초) 내에 응답이 없는 경우, 상대 기기가 오프라인 상태인 것으로 판단하여 타임아웃 처리한다.

#### 제4 구성: 종단간 암호화

상기 제1 내지 제3 구성의 모든 데이터 전송에 있어서, 데이터는 송신 기기에서 AES-256-GCM 알고리즘으로 암호화되고, 수신 기기에서만 복호화된다.

암호화 키는 사용자가 설정한 패스프레이즈(passphrase)로부터 PBKDF2 키 유도 함수를 통하여 파생되며, 서버에는 전송되지 않는다.

각 암호화 작업에는 고유한 초기화 벡터(IV)와 인증 태그(Auth Tag)가 사용되어, 동일한 평문에 대해서도 서로 다른 암호문이 생성된다.

서버는 모든 과정에서 암호화된 데이터만을 중계하며, 평문 데이터에 접근할 수 없다.

#### 제5 구성: HTTP 기반 시계 교정 및 타임스탬프 삼중 표현 (시간 동기화 계층)

본 발명의 제5 구성은, 서로 다른 시간대에 위치한 복수의 기기에서 발생한 이벤트를 절대적 시간 순서로 정렬하고, 사용자의 홈 시간대로 통합 표시하기 위한 시간 동기화 방식을 포함한다.

##### 제5-1 구성: HTTP Date 헤더 기반 시계 드리프트 검출

각 기기는 시작 시 및 주기적으로(소정의 간격, 예: 30분) 잘 알려진 HTTPS 엔드포인트(예: google.com, cloudflare.com, apple.com)에 대하여 HTTP HEAD 요청을 발신하고, 응답의 `Date` 헤더에 포함된 서버 시각과 기기의 로컬 시각을 비교하여 시계 드리프트를 측정한다.

상기 HTTP Date 헤더 방식은 종래의 NTP(Network Time Protocol)와 비교하여 다음의 이점을 가진다: (i) HTTPS(TCP 포트 443)를 사용하므로 방화벽, NAT, 기업 프록시 환경에서 차단되지 않으며, (ii) 별도의 NTP 클라이언트 라이브러리 없이 기존 HTTP 클라이언트로 구현 가능하고, (iii) 1초 이내의 정밀도가 사용자 행동 이벤트의 시간 순서 판정에 충분하다.

드리프트 측정은 다음 절차로 수행된다: (i) HEAD 요청 발신 직전의 로컬 시각(T1)과 응답 수신 직후의 로컬 시각(T2)의 중간값((T1+T2)/2)을 로컬 기준 시각으로 산출하고, (ii) 응답 Date 헤더의 시각을 원격 기준 시각으로 파싱하며, (iii) 양 시각의 차이의 절대값을 드리프트로 산출한다.

드리프트가 소정의 임계값(예: 5초)을 초과하는 경우, 사용자에게 시계 보정을 권고하는 경고를 발생시킨다.

##### 제5-2 구성: 타임스탬프 삼중 표현 (Timestamp Triple)

각 기기에서 발생한 이벤트에 대하여, 다음의 3가지 시각 표현을 동시에 생성하고 저장한다:

(i) **UTC 절대 시각** (`occurred_at_utc`): ISO-8601 UTC 형식(예: "2026-03-18T05:30:00.000Z"). 이것이 크로스-디바이스 타임라인의 **1차 정렬 키(primary sort key)**로서, 어떤 기기에서 기록되었든 동일한 절대 시각 기준으로 이벤트가 정렬된다.

(ii) **기기 로컬 시각** (`occurred_at_local`): 이벤트 기록 시점의 기기 시간대 오프셋을 포함한 ISO-8601 형식(예: "2026-03-18T01:30:00.000-04:00"). 이벤트가 기록된 기기의 시간적 맥락을 보존한다.

(iii) **홈 시간대 시각** (`occurred_at_home`): 사용자가 설정한 홈 시간대(예: "Asia/Seoul")로 변환된 ISO-8601 형식(예: "2026-03-18T14:30:00.000+09:00"). 이것이 사용자에게 표시되는 **디스플레이 시각**으로서, 어떤 기기에서 기록된 이벤트든 사용자의 생활 기준 시간대로 통합 표시된다.

상기 3가지 시각 표현에 추가하여, 기기의 IANA 시간대 이름(예: "America/New_York")과 사용자의 홈 시간대 이름(예: "Asia/Seoul")을 함께 저장하여, 시간대 변환의 추적성을 보장한다.

##### 제5-3 구성: 기기 시간대 자동 검출

각 기기는 시작 시 자신의 현재 IANA 시간대를 다음의 우선순위로 자동 검출한다: (i) `TZ` 환경 변수에 설정된 시간대 이름, (ii) 운영체제의 시간대 설정 파일(예: Unix/macOS의 `/etc/localtime` 심볼릭 링크 대상에서 추출), (iii) 상기 방법으로 검출이 불가능한 경우 UTC를 폴백으로 사용.

##### 제5-4 구성: 다중 시간대 타임라인 통합 시나리오

사용자가 한국의 PC(Asia/Seoul)에서 2026년 3월 18일 오후 2시 30분에 행동을 기록하고, 미국 뉴욕의 휴대폰(America/New_York)에서 같은 날 밤 10시에 행동을 기록한 경우:

```text
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐
│ 한국 PC           │   │ 사무실 PC         │   │ 미국 휴대폰       │
│ Asia/Seoul        │   │ Asia/Seoul        │   │ America/New_York  │
│ 2026-03-18        │   │ 2026-03-18        │   │ 2026-03-17        │
│ 14:30 KST         │   │ 16:00 KST         │   │ 22:00 EST         │
└────────┬─────────┘   └────────┬─────────┘   └────────┬─────────┘
         │                      │                      │
         ▼                      ▼                      ▼
    occurred_at_utc        occurred_at_utc        occurred_at_utc
    05:30:00Z              07:00:00Z              03:00:00Z  ← 정렬 키
         │                      │                      │
         ▼                      ▼                      ▼
    occurred_at_home       occurred_at_home       occurred_at_home
    14:30 KST              16:00 KST              12:00 KST  ← 표시
```

위 예시에서, UTC 기준으로 미국 휴대폰의 이벤트(03:00Z)가 가장 먼저 발생하였으므로 타임라인에서 가장 먼저 표시되며, 홈 시간대(KST)로 변환하면 12:00으로서 사용자에게 직관적으로 시간 순서가 전달된다.

#### 제6 구성: 시간-위치 복합 이벤트 정렬 (게이트웨이-온톨로지 연결)

본 발명의 제6 구성은, 상기 제5 구성의 시간 동기화 계층과 구조화된 관계형 기억(온톨로지)을 직접 연결하여, 모든 행동 이력에 대하여 **언제(When)**와 **어디서(Where)**를 1급 정렬 키로 기록하고 질의하는 방식을 포함한다.

##### 제6-1 구성: 행동 이력의 5W1H 구조

온톨로지의 행동 이력(action log)에 기록되는 각 이벤트는 다음의 5W1H 구조를 가진다:

- **누가(Who)**: 행위자 식별자 및 행위자 유형(사용자/에이전트/시스템)
- **무엇을(What)**: 행동 유형(예: SendMessage, CreateTask, FetchResource)
- **누구에게(Whom)**: 대상 개체 식별자
- **언제(When)**: 타임스탬프 삼중 표현(UTC 절대 시각 + 기기 로컬 시각 + 홈 시간대 시각)
- **어디서(Where)**: 자유 형식 위치 문자열(예: "서울 서초구 법원로 서울중앙지방법원")
- **어떻게(How)**: 실행 매개변수 및 결과

##### 제6-2 구성: 복합 인덱싱 및 질의

상기 시간(occurred_at_utc)과 위치(location) 필드에 대하여 개별 인덱스 및 복합 인덱스를 생성하여, 다음과 같은 시간-위치 복합 질의를 효율적으로 수행한다:

- 시간 범위 질의: "2026년 3월 첫째 주에 발생한 모든 이벤트"
- 위치 질의: "서울중앙지방법원에서 발생한 모든 이벤트"
- 시간-위치 복합 질의: "지난주 골프장에서의 미팅 이력"
- 크로스-디바이스 통합 질의: "모든 기기에서 발생한 이벤트를 시간 순서대로 통합 표시"

##### 제6-3 구성: 게이트웨이-온톨로지 직접 연결

기기의 게이트웨이 모듈(네트워크 통신 담당)이 시작 시 상기 제5-1 구성의 시계 드리프트 검출을 수행하고, 검출된 기기 시간대 정보 및 사용자 홈 시간대 정보를 전역 상태로 보관한다. 온톨로지 모듈의 행동 기록 함수는 상기 전역 상태로부터 시간대 정보를 취득하여, 매 행동 기록 시 타임스탬프 삼중 표현을 자동으로 생성한다.

이에 의하여, 사용자 또는 에이전트가 시간이나 시간대를 의식적으로 입력하지 않아도, 모든 행동 이력에 대하여 정확한 UTC 절대 시각, 기기 로컬 시각, 및 홈 시간대 시각이 자동으로 기록된다.

##### 제6-4 구성: 동기화 델타에의 시간 정보 포함

상기 제2 구성(로컬 델타 저널 기반 동기화)의 행동 로그 델타에 타임스탬프 삼중 표현의 전체(UTC, 기기 로컬, 기기 시간대, 홈 시간대 시각, 홈 시간대 이름)를 포함하여, 원격 기기가 동기화 수신 시에도 완전한 시간 맥락을 복원할 수 있다.

### [발명의 효과]

본 발명에 의하면 다음의 효과가 달성된다.

**제1 효과 (프라이버시 보장):** 서버에 사용자 데이터가 영구적으로 저장되지 않는다. 임시 릴레이 데이터는 소정의 유효기간 후 자동 삭제되고, 재동기화 및 수동 동기화 과정은 데이터베이스 저장을 수반하지 않는 브로드캐스트 채널만을 사용한다. 종단간 암호화에 의하여 서버 운영자도 데이터 내용을 열람할 수 없다.

**제2 효과 (비동시 온라인 동기화):** 종래의 P2P 방식과 달리, 두 기기가 동시에 온라인 상태가 아니더라도 동기화가 가능하다. 로컬 델타 저널에 의하여 소정의 보관 기간 내에 상대 기기가 온라인으로 복귀하면 누락된 데이터가 자동으로 재전송된다.

**제3 효과 (순서 보장):** 시퀀스 넘버, 순서 대기 버퍼, 및 갭 탐지 메커니즘에 의하여, 네트워크 지연으로 인한 전송 순서 역전이 발생하더라도 데이터가 올바른 순서로 적용된다.

**제4 효과 (멱등성 및 중복 방지):** 기기 식별자와 시퀀스 넘버의 결합에 의한 전역적 고유 식별과, 버전 벡터에 의한 수신 이력 관리에 의하여, 동일한 델타가 중복 적용되는 것이 방지된다.

**제5 효과 (장기 오프라인 복구):** 소정의 보관 기간을 초과하여 오프라인이었던 기기에 대해서도, 매니페스트 비교 기반 수동 전체 동기화에 의하여 완전한 데이터 일관성의 복원이 가능하다.

**제6 효과 (서버 비용 절감):** 서버가 데이터를 영구 저장하지 않으므로, 사용자 수 증가에 따른 저장 비용이 발생하지 않는다. 서버의 역할은 일시적 중계와 WebSocket 시그널링으로 한정된다.

**제7 효과 (크로스-디바이스 시간 정렬):** 타임스탬프 삼중 표현에 의하여, 서로 다른 시간대에 위치한 복수의 기기에서 발생한 이벤트가 UTC 절대 시각을 기준으로 정확히 정렬되고, 동시에 사용자의 홈 시간대로 통합 표시된다. 종래 시스템과 달리 중앙 서버에 의존하지 않고도 분산 환경에서 시간적 일관성이 보장된다.

**제8 효과 (시계 드리프트 검출):** HTTP Date 헤더 기반 시계 교정에 의하여, 기기의 시스템 시계 부정확이 자동으로 검출되어 크로스-디바이스 타임라인의 시간적 신뢰성이 확보된다. UDP 기반 NTP의 방화벽 차단 문제 없이 HTTPS만으로 시계 교정이 가능하다.

**제9 효과 (시간-위치 복합 질의):** 이벤트의 발생 시각과 장소가 1급 정렬 키로 기록됨에 따라, "지난주 서울에서 한 일", "특정 장소에서의 미팅 이력" 등 시간-위치 복합 질의가 가능하며, 이는 사용자의 활동 패턴 추론 및 AI 에이전트의 맥락 인식에 직접 활용된다.

**제10 효과 (자동 시간대 변환):** 게이트웨이-온톨로지 직접 연결에 의하여, 사용자가 시간대를 의식적으로 입력하지 않아도 모든 행동 이력에 대하여 정확한 시간대 변환이 자동 수행된다. 이는 종래의 중앙 서버 기반 시스템이 서버 수신 시각에 의존하는 것과 차별화된다.

### 실시예 2: 제1 계층 — 임시 릴레이 기반 실시간 동기화

도 2는 수신 기기가 온라인 상태인 경우의 실시간 동기화 흐름을 나타내는 시퀀스 다이어그램이다.

```
┌──────────────────────────────────────────────────────────────┐
│           제1 계층: 실시간 동기화 시퀀스 다이어그램               │
│                                                              │
│  기기 A                  서버                    기기 B        │
│    │                      │                       │          │
│    │  ① 데이터 변경 발생    │                       │          │
│    │  ② AES-256-GCM 암호화  │                       │          │
│    │  ③ 시퀀스 넘버 부여     │                       │          │
│    │  ④ 델타 저널에 보관     │                       │          │
│    │                      │                       │          │
│    ├──⑤ 업로드 ──────────▶│                       │          │
│    │  (암호문 + IV +       │                       │          │
│    │   AuthTag +           │                       │          │
│    │   TTL=5분)            │                       │          │
│    │                      │                       │          │
│    │                      ├──⑥ Realtime 통지 ──▶│          │
│    │                      │  (INSERT 이벤트)      │          │
│    │                      │                       │          │
│    │                      │◀──⑦ 다운로드 요청 ──┤          │
│    │                      │                       │          │
│    │                      ├──⑧ 암호문 전달 ────▶│          │
│    │                      │                       │          │
│    │                      │                  ⑨ 복호화       │
│    │                      │                  ⑩ 적용         │
│    │                      │                  ⑪ 버전벡터 갱신  │

│    │              ⑫ TTL 경과 후                     │          │
│    │              서버 데이터 삭제                    │          │
│    │                      │                       │          │
└──────────────────────────────────────────────────────────────┘
```

**단계 S210:** 기기 A에서 데이터 변경(예: 장기기억 추가)이 발생한다.

**단계 S220:** 기기 A의 암호화 엔진(110)이 변경 데이터를 AES-256-GCM 알고리즘으로 암호화한다. 암호화 키는 사용자의 패스프레이즈로부터 PBKDF2를 통해 파생된 키를 사용하며, 각 암호화 시 고유한 IV(초기화 벡터)를 생성한다.

**단계 S230:** 기기 A의 재조정기(150)가 해당 변경에 대하여 단조 증가하는 시퀀스 넘버를 부여한다. 시퀀스 넘버는 기기 A 내에서 고유하며, (기기 A 식별자, 시퀀스 넘버) 쌍으로 전역적 고유성이 보장된다.

**단계 S240:** 상기 암호화된 델타를 기기 A의 로컬 델타 저널(120)에 보관한다.

**단계 S250:** 암호화된 데이터, IV, 인증 태그, 기기 식별자, 시퀀스 넘버, 및 유효기간(현재 시각 + 5분)을 릴레이 서버(200)의 임시 데이터 저장소(210)에 업로드한다.

**단계 S260:** 릴레이 서버(200)가 WebSocket 실시간 채널을 통하여 기기 B에 데이터 도착을 통지한다.

**단계 S270-S280:** 기기 B가 릴레이 서버(200)로부터 암호화된 데이터를 다운로드한다.

**단계 S290:** 기기 B의 암호화 엔진(110)이 데이터를 복호화하고, 순서 검증 후 적용한다. 기기 B의 버전 벡터(130)에서 기기 A에 대한 마지막 수신 시퀀스를 갱신한다.

**단계 S2100:** 유효기간(5분) 경과 후, 릴레이 서버(200)의 임시 데이터 저장소(210)에서 해당 데이터가 자동 삭제된다.

### 실시예 3: 제2 계층 — 순서 보장 메커니즘

도 3은 네트워크 지연으로 인한 전송 순서 역전 시의 순서 보장 처리를 나타내는 도면이다.

```
┌──────────────────────────────────────────────────────────────────┐
│                제2 계층: 순서 보장 메커니즘                          │
│                                                                  │
│  기기 A 전송 순서:  seq 1 → seq 2 → seq 3 → seq 4 → seq 5        │
│                                                                  │
│  기기 B 수신 순서:  seq 1, seq 3, seq 2, seq 5, seq 4             │
│                    (네트워크 지연으로 순서 역전)                     │
│                                                                  │
│  기기 B 처리 흐름:                                                 │
│  ┌──────────────────────────────────────────────────────────────┐ │
│  │ 수신      │ 기대 시퀀스 │ 동작          │ 버전벡터 │ 순서버퍼  │ │
│  ├───────────┼───────────┼──────────────┼────────┼─────────── │ │
│  │ seq 1     │     1     │ ✅ 즉시 적용  │ A→1    │ (비어있음) │ │
│  │ seq 3     │     2     │ ⏳ 버퍼 저장  │ A→1    │ [3]       │ │
│  │ seq 2     │     2     │ ✅ 즉시 적용  │ A→2    │ [3]       │ │
│  │           │           │ ✅ 버퍼 플러시 │ A→3    │ (비어있음) │ │
│  │ seq 5     │     4     │ ⏳ 버퍼 저장  │ A→3    │ [5]       │ │
│  │ seq 4     │     4     │ ✅ 즉시 적용  │ A→4    │ [5]       │ │
│  │           │           │ ✅ 버퍼 플러시 │ A→5    │ (비어있음) │ │
│  └──────────────────────────────────────────────────────────────┘ │
│                                                                  │
│  최종 적용 순서: seq 1 → seq 2 → seq 3 → seq 4 → seq 5 ✅        │
└──────────────────────────────────────────────────────────────────┘
```

**단계 S310:** 수신 기기의 재조정기(150)는 수신한 델타의 시퀀스 넘버(Sn)와 버전 벡터(130)에 기록된 해당 송신 기기의 마지막 수신 시퀀스(Lv)를 비교한다.

**단계 S320:** Sn ≤ Lv인 경우 (이미 수신된 시퀀스), 해당 델타를 무시한다. 이에 의하여 네트워크 재전송 등에 의한 중복 적용이 방지된다.

**단계 S330:** Sn = Lv + 1인 경우 (기대 시퀀스), 해당 델타를 즉시 적용하고 버전 벡터를 Lv = Sn으로 갱신한다. 이후 순서 대기 버퍼(140)에 연속된 다음 시퀀스(Sn + 1, Sn + 2, ...)가 존재하는지 검사하고, 존재하는 경우 연속적으로 적용한다(버퍼 플러시).

**단계 S340:** Sn > Lv + 1인 경우 (갭 발생), 해당 델타를 순서 대기 버퍼(140)에 시퀀스 순으로 삽입한다. 동시에 누락된 시퀀스(Lv + 1 ~ Sn - 1)에 대한 재전송 요청을 브로드캐스트 채널을 통하여 발신한다.

### 실시예 4: 제2 계층 — 오프라인 후 재연결 시 자동 재동기화

도 4는 수신 기기가 오프라인이었다가 복귀한 경우의 자동 재동기화 흐름을 나타내는 시퀀스 다이어그램이다.

```
┌──────────────────────────────────────────────────────────────────┐
│        제2 계층: 오프라인 후 재연결 자동 재동기화                     │
│                                                                  │
│  기기 A                    서버                     기기 B         │
│    │                        │                        │           │
│    │  ① seq 4, 5, 6 생성    │                        │ (오프라인) │
│    │  ② 서버에 업로드        │                        │           │
│    ├──────────────────────▶│                        │           │
│    │  ③ 델타 저널에 보관     │                        │           │
│    │                        │  ④ TTL 5분 경과         │           │
│    │                        │     서버 데이터 삭제 💨  │           │
│    │                        │                        │           │
│    │                        │             ⑤ 기기 B 온라인 복귀    │
│    │                        │                        │           │
│    │                        │◀─ ⑥ sync_request ─────┤           │
│    │◀─── broadcast ────────│  버전벡터: {A→3}        │           │
│    │                        │                        │           │
│    │  ⑦ 저널 검색:          │                        │           │
│    │  B가 A의 seq 3까지      │                        │           │
│    │  받았으므로             │                        │           │
│    │  seq 4,5,6 누락         │                        │           │
│    │                        │                        │           │
│    ├── ⑧ sync_response ──▶│── broadcast 전달 ─────▶│           │
│    │  (seq 4, 5, 6 재전송)   │  (DB 저장 없음)        │           │
│    │                        │                        │           │
│    │                        │                   ⑨ 수신 + 적용    │
│    │                        │                   ⑩ 버전벡터 갱신   │
│    │                        │                      {A→6}        │
│    │                        │                        │           │
│    │                        │◀──⑪ delta_ack ────────┤           │
│    │◀─── broadcast ────────│                        │           │
│    │                        │                        │           │
└──────────────────────────────────────────────────────────────────┘
```

**단계 S410:** 기기 A에서 데이터 변경(seq 4, 5, 6)이 발생한다. 기기 A는 이를 서버에 업로드하고, 동시에 로컬 델타 저널(120)에 보관한다.

**단계 S420:** 기기 B가 오프라인 상태이므로 데이터를 수신하지 못한다. 서버의 TTL 경과 후 데이터가 자동 삭제된다.

**단계 S430:** 기기 B가 온라인으로 복귀한다. 기기 B의 재조정기(150)는 브로드캐스트 채널(220) 연결 완료 시, 자신의 버전 벡터 {A→3}를 포함한 sync_request 메시지를 자동으로 브로드캐스트한다.

**단계 S440:** 기기 A의 재조정기(150)가 상기 sync_request를 수신하고, 기기 B의 버전 벡터 {A→3}과 자신의 로컬 델타 저널(120)을 비교한다. 기기 B가 기기 A의 seq 3까지만 수신하였으므로, seq 4, 5, 6이 누락 델타로 식별된다.

**단계 S450:** 기기 A가 누락 델타(seq 4, 5, 6)를 sync_response 메시지로 브로드캐스트 채널을 통하여 전송한다. 대량의 누락 델타가 있는 경우 소정의 배치 크기(예: 50개)로 분할 전송한다.

**단계 S460:** 기기 B가 수신한 델타를 시퀀스 순으로 적용하고 버전 벡터를 {A→6}으로 갱신한다.

**단계 S470:** 기기 B가 수신 확인(delta_ack) 메시지를 발신한다.

### 실시예 5: 제3 계층 — 수동 전체 동기화

도 5는 소정의 보관 기간(30일)을 초과하여 오프라인이었던 기기의 수동 전체 동기화 흐름을 나타내는 시퀀스 다이어그램이다.

```

│              제3 계층: 수동 전체 동기화                              │
│                                                                  │
```

**단계 S510:** 사용자가 기기 A에서 수동 동기화를 요청한다 (예: UI의 "동기화" 버튼 탭).

**단계 S520:** 기기 A의 재조정기(150)가 자신이 보유한 모든 엔티티의 식별자 목록(매니페스트)을 생성한다. 매니페스트는 장기기억 청크 ID 목록, 대화 세션 ID 목록, 설정 키 목록을 포함한다.

**단계 S530:** 기기 A가 자신의 매니페스트를 포함한 full_sync_request 메시지를 브로드캐스트 채널을 통하여 전송한다.

**단계 S540:** 기기 B의 재조정기(150)가 상기 요청을 수신하고, 자신의 매니페스트를 생성하여 full_sync_manifest_response 메시지로 응답한다.

**단계 S550:** 양측 기기가 수신한 상대 매니페스트와 자신의 매니페스트를 비교한다. 비교는 엔티티 식별자 집합에 대한 차집합(set difference) 연산으로 수행된다. 이에 의하여 한쪽에만 존재하는 엔티티가 식별된다.

**단계 S560:** 각 기기가 자신이 보유하고 상대에게 누락된 엔티티를 full_sync_data 메시지로 하나씩 전송한다. 각 엔티티는 종단간 암호화된 상태로 전송된다.

**단계 S570:** 수신 기기가 각 엔티티를 복호화하고 로컬 저장소에 저장한다. 수신 진행 상황이 사용자에게 표시된다.

**단계 S580:** 모든 누락 엔티티의 전송이 완료되면, 전송 측이 full_sync_complete 메시지를 발신하여 수신 측에 완료를 통지한다.

### 실시예 7: 데이터 구조

이하, 본 발명의 실시에 사용되는 주요 데이터 구조를 설명한다.

#### 7-1. 동기화 델타 (SyncDelta)

```
SyncDelta {
  id: string                  // 전역 고유 식별자 (기기ID-시퀀스-타임스탬프)
  sourceDeviceId: string      // 생성 기기 식별자
  seq: number                 // 기기 내 단조 증가 시퀀스 넘버
  deltaType: "add" | "update" | "delete"  // 변경 유형
  entityType: "memory_chunk" | "conversation" | "setting"  // 대상 엔티티 유형 (기본)
              | "structured_object" | "structured_link" | "action_log"  // 구조화된 기억 확장
  encryptedPayload: string    // AES-256-GCM 암호화된 데이터
  iv: string                  // 초기화 벡터
  authTag: string             // 인증 태그
  createdAt: string           // 생성 시각 (ISO 8601)
}
```

#### 7-2. 버전 벡터 (Version Vector)

```
VersionVector = {
  [deviceId: string]: number  // 각 기기로부터 마지막 수신한 시퀀스 넘버
}

예시:
{
  "device-A": 42,   // 기기 A로부터 seq 42까지 수신 완료
  "device-B": 17,   // 기기 B로부터 seq 17까지 수신 완료
  "device-C": 8     // 기기 C로부터 seq 8까지 수신 완료
}
```

#### 7-3. 전체 동기화 매니페스트 (FullSyncManifest)

```
FullSyncManifest {
  memoryChunkIds: string[]     // 장기기억 청크 ID 목록
  conversationIds: string[]    // 대화 세션 ID 목록
  settingKeys: string[]        // 설정 키 목록
  structuredObjectIds: string[] // 구조화된 개체 ID 목록 (선택적 확장)
  structuredLinkIds: string[]   // 구조화된 관계 ID 목록 (선택적 확장)
  generatedAt: string          // 생성 시각 (ISO 8601)
}
```

#### 7-4. 브로드캐스트 메시지 유형

```
BroadcastMessage =
  // 자동 재동기화용
  | { type: "sync_request",  fromDeviceId, versionVector }
  | { type: "sync_response", fromDeviceId, deltas[] }
  | { type: "delta_ack",     fromDeviceId, sourceDeviceId, lastSeq }
  // 수동 전체 동기화용
  | { type: "full_sync_request",           fromDeviceId, manifest }
  | { type: "full_sync_manifest_response", fromDeviceId, manifest }
  | { type: "full_sync_data",    fromDeviceId, entityType, entityId, encryptedPayload, iv, authTag }
  | { type: "full_sync_complete", fromDeviceId, sentCount }
```

#### 7-5. 재조정기 상태 (ReconcilerState) — 영속화 대상

```
ReconcilerState {
  localSeq: number             // 현재 시퀀스 넘버
  versionVector: VersionVector // 버전 벡터
  journal: SyncDelta[]         // 로컬 델타 저널
  exportedAt: string           // 상태 내보내기 시각
}
```

### 실시예 8: 제5 구성 — HTTP 기반 시계 교정 및 타임스탬프 삼중 표현

도 8은 기기 시작 시 및 주기적으로 수행되는 시계 드리프트 검출과 타임스탬프 삼중 표현 생성의 흐름을 나타내는 도면이다.

```
┌──────────────────────────────────────────────────────────────────┐
│         제5 구성: HTTP 기반 시계 교정 및 타임스탬프 삼중 표현         │
│                                                                  │
│  기기 A                HTTPS 서버              타임스탬프 생성      │
│    │                  (google.com 등)               │             │
│    │                      │                        │             │
│    │  ① 기기 시작          │                        │             │
│    │  ② 기기 시간대 검출    │                        │             │
│    │  (TZ → /etc/localtime │                        │             │
│    │   → UTC 폴백)         │                        │             │
│    │                      │                        │             │
│    │  ③ T1 = 로컬시각()    │                        │             │
│    ├──④ HEAD 요청 ───────▶│                        │             │
│    │                      │                        │             │
│    │◀──⑤ HTTP 200 ───────┤                        │             │
│    │  Date: Tue, 18 Mar   │                        │             │
│    │  2026 05:30:12 GMT   │                        │             │
│    │  ⑥ T2 = 로컬시각()   │                        │             │
│    │                      │                        │             │
│    │  ⑦ 로컬중간값 =       │                        │             │
│    │     (T1 + T2) / 2    │                        │             │
│    │  ⑧ 드리프트 =         │                        │             │
│    │    |로컬중간값 - Date| │                        │             │
│    │                      │                        │             │
│    │  ⑨ 드리프트 > 5초?    │                        │             │
│    │  예 → 경고 발생 ⚠     │                        │             │
│    │  아니오 → OK ✅        │                        │             │
│    │                      │                        │             │
│    │         ⑩ 이벤트 발생 시:                      │             │
│    │         ┌─────────────────────────────────────┤             │
│    │         │ 타임스탬프 삼중 표현 생성               │             │
│    │         │                                     │             │
│    │         │  UTC = Utc::now()                   │             │
│    │         │    → "2026-03-18T05:30:00.000Z"     │             │
│    │         │  기기로컬 = UTC→기기시간대              │             │
│    │         │    → "2026-03-18T14:30:00.000+09:00" │             │
│    │         │  홈시간대 = UTC→홈시간대                │             │
│    │         │    → "2026-03-18T14:30:00.000+09:00" │             │
│    │         │  기기TZ = "Asia/Seoul"               │             │
│    │         │  홈TZ   = "Asia/Seoul"               │             │
│    │         └─────────────────────────────────────┘             │
└──────────────────────────────────────────────────────────────────┘
```

**단계 S810:** 기기 시작 시 시간대 자동 검출을 수행한다. TZ 환경 변수를 먼저 확인하고, 없으면 운영체제의 시간대 설정을 읽으며, 모두 실패 시 UTC를 폴백으로 사용한다.

**단계 S820:** HTTPS 엔드포인트에 HEAD 요청을 발신하고, 요청 전(T1)과 응답 후(T2)의 로컬 시각을 기록한다. 중간값 (T1+T2)/2와 응답 Date 헤더 시각의 차이를 드리프트로 산출한다.

**단계 S830:** 드리프트가 임계값(5초)을 초과하면 사용자에게 시계 보정을 권고한다. 기기 시간대, 홈 시간대, 드리프트 값을 전역 상태에 보관한다.

**단계 S840:** 이벤트 발생 시, 현재 UTC 시각을 기준으로 타임스탬프 삼중 표현(UTC + 기기 로컬 + 홈 시간대)을 자동 생성하여 행동 이력에 저장한다.

## [추가 발명] 에피소드 기억과 구조적 온톨로지 간 양방향 교차 참조 기억 시스템

### [발명의 명칭]

에피소드 기억과 구조적 온톨로지 간 양방향 교차 참조를 통한 AI 에이전트 기억 시스템 및 방법

(Bidirectional Cross-Referenced Dual-Store Memory System and Method
for AI Agents Using Episodic Memory and Structural Ontology)

### [기술분야]

본 발명은 인공지능 에이전트의 장기 기억 관리 시스템에 관한 것으로, 특히 에피소드
기억(대화, 문서, 코드 등 비정형 데이터)과 구조적 온톨로지(인물, 장소, 시간, 관계
등 정형 데이터) 간의 양방향 교차 검색을 통해 AI 에이전트의 문맥 이해력과 회상
정확도를 획기적으로 향상시키는 기술에 관한 것이다.

### [배경기술]

종래의 AI 비서 시스템은 기억 체계에 있어 다음과 같은 한계를 갖는다:

1. **단일 저장소 방식의 한계**: 대화 이력을 단순 텍스트 또는 벡터 임베딩으로만
   저장하여, "누구와 언제 어디서 무엇을 했는가"라는 맥락적 질문에 적절히
   답변할 수 없다.

2. **독립 검색 방식의 한계**: 벡터 유사도 검색과 키워드 검색을 결합하더라도,
   인물 간 관계, 프로젝트 소속, 조직 구조 등 구조적 관계를 파악하지 못한다.

3. **온톨로지 단독 방식의 한계**: 지식 그래프만으로는 실제 대화 내용과 작업
   결과물의 원문을 회상할 수 없어, 답변의 구체성이 부족하다.

4. **병렬 검색 후 단순 결합의 한계**: 두 저장소를 각각 검색한 후 결과를
   단순히 이어붙이면(concatenation), 두 결과 사이의 **숨겨진 연관 정보를
   체계적으로 발견하는 것이 불가능**하다. 예를 들어, "김대리"라는 인물에
   대해 메모리에서 대화 내용만 찾고, 온톨로지에서 인물 정보만 찾으면,
   김대리의 소속 부서에서 발생한 다른 관련 대화나, 김대리와 관련된
   프로젝트의 전체 맥락을 누락하게 된다.

### [발명의 내용]

#### [해결하고자 하는 과제]

본 발명은 상기 종래 기술의 문제점을 해결하기 위하여, 에피소드 기억 저장소와
구조적 온톨로지 저장소를 동일 데이터베이스 내에 공존시키되, **4단계 양방향 교차
검색 프로토콜**을 통해 두 저장소의 결과를 상호 보강하는 AI 에이전트 기억 시스템을
제공하는 것을 목적으로 한다.

#### [과제의 해결 수단]

본 발명의 시스템은 다음 구성요소로 이루어진다:

**1. 이중 저장소 구성 (Dual-Store Architecture)**

하나의 SQLite 데이터베이스(brain.db) 내에 두 가지 저장소를 공존시킨다:

- **에피소드 기억 저장소**: memories 테이블에 (key, content, category,
  embedding BLOB, created_at, updated_at) 구조로 저장. FTS5 가상 테이블
  (memories_fts)을 통한 전문검색과 벡터 임베딩을 통한 의미 유사도 검색을
  동시 지원. 하이브리드 검색 시 벡터 가중치(기본 0.7)와 키워드 가중치(기본 0.3)의
  가중 융합으로 최종 점수를 산출.

- **구조적 온톨로지 저장소**: ontology_objects(객체), ontology_links(관계),
  ontology_actions(행위) 테이블로 구성. 각 행위(action)는 5W1H 메타데이터를
  필수 포함: 행위자(actor_user_id), 행위유형(action_type_id),
  대상(primary_object_id), 시각(occurred_at_utc + occurred_at_local +
  occurred_at_home의 3중 시간 표현), 장소(location), 방법(params JSON).
  ontology_objects_fts FTS5 가상 테이블을 통한 전문검색 지원.

**2. 3단계 기억 파이프라인 (3-Stage Memory Pipeline)**

- **1단계(CAPTURE)**: 사용자 발화 수신 시 즉시 세션 관리자
  (SessionManager)에 단기 기억으로 저장. 동시에 메타데이터 추출:
  타임스탬프(ISO-8601, 기기 시간대 포함), 위치(채널명 또는 기기 컨텍스트),
  상대방(대화 상대 식별), 상호작용 카테고리(Chat, Document, Coding,
  Image, Music, Video, Translation의 7종 분류).

- **2단계(PROMOTE)**: LLM 응답 완료 후 자동으로 두 저장소에 동시 승격.
  에피소드 기억: 구조화된 메타데이터를 포함한 항목을 Core 카테고리에 저장
  (키 형식: promoted_{category}_{UUID}). 온톨로지: Context 객체 생성,
  Contact 객체 생성/갱신, 카테고리별 특화 객체(Document, Task 등) 생성,
  관계(Link) 설정, 행위(Action) 5W1H 기록.

- **3단계(RECALL)**: 하기 4단계 교차 검색 프로토콜에 의한 회상.

**3. 4단계 양방향 교차 검색 프로토콜 (4-Phase Cross-Search Protocol)**

본 발명의 핵심 구성으로, 사용자 질의 시 다음 4단계를 순차 수행한다:

**Phase 1 — 에피소드 기억 1차 검색**: 사용자 질의에 대해 벡터 임베딩
코사인 유사도 검색과 FTS5 BM25 키워드 검색을 동시 수행하여 가중 융합.
최소 관련도 점수(min_relevance_score) 이하의 결과를 필터링. 결과
항목의 내용에서 시간(시간:), 장소(장소:), 인물(상대방:), 행위(행위:)
키워드를 정규 표현식으로 추출하여 교차 검색 키워드 집합 A를 생성.

**Phase 2 — 온톨로지 1차 검색**: 사용자 질의에 대해 온톨로지 객체의
제목 및 속성에서 FTS5 전문검색 수행. 결과 객체의 제목과 JSON 속성에서
이름(name), 장소(location), 시간(time/date), 주제(topic/subject) 등
핵심 필드를 추출하여 교차 검색 키워드 집합 B를 생성.

**Phase 3 — 온톨로지→에피소드 교차 보강**: 키워드 집합 B(최대 5개)를
공백으로 결합하여 에피소드 기억을 재검색. Phase 1에서 이미 발견된
항목(중복)은 제외. 최대 20건의 추가 관련 대화/작업 결과물을 발견하여
[Cross-referenced memories (from ontology context)]로 출력.

**Phase 4 — 에피소드→온톨로지 교차 보강**: 키워드 집합 A(최대 5개)를
공백으로 결합하여 온톨로지를 재검색. Phase 2에서 이미 발견된
객체(중복)는 제외. 최대 20건의 추가 관련 인물 관계, 프로젝트 구조,
미팅 맥락을 발견하여 [Cross-referenced relationships (from memory
context)]로 출력.

4개 구간의 결과를 통합하여 AI 모델에 제공한다.

#### [발명의 효과]

1. **맥락 완전성**: 사용자가 인물명만으로 질문해도, 해당 인물과의
   전체 대화 내역(에피소드) + 소속/역할/관계(온톨로지) + 관련 프로젝트의
   다른 대화(교차 에피소드) + 관련 조직 구조(교차 온톨로지)를 모두
   발견하여 LLM에 제공.

2. **숨겨진 연관 정보 발견**: 원래 질의만으로는 매칭되지 않았을
   관련 정보를 교차 키워드를 통해 체계적으로 발견. 예: "김대리"를
   검색하면, 김대리의 소속 "영업팀"이라는 키워드로 재검색하여
   영업팀 관련 다른 대화와 프로젝트를 추가 발견.

3. **중복 제거**: 교차 검색 결과에서 이미 1차 검색에 포함된 항목을
   자동 제외하여 동일 정보의 중복 표시를 방지.

4. **성능 제어**: 교차 검색 키워드 수(최대 5개)와 추가 검색 결과 수
   (최대 20건)를 제한하여 검색 비용의 폭발적 증가를 방지.

5. **범용성**: 에피소드 기억의 벡터+키워드 하이브리드 검색과 온톨로지의
   FTS5 전문검색이 범용적으로 결합되어, 어떤 자연어 질의에도 적용 가능.

### [실시예]

#### 실시예 1: 인물 기반 질의

사용자 질의: "김대리가 지난번에 뭐라고 했지?"

1. Phase 1 (메모리 검색): "김대리" → 매칭 결과: 2026-03-15 대화 "프로젝트
   진행상황 논의". 추출 키워드: {시간:2026-03-15, 장소:사무실, 상대방:김대리}

2. Phase 2 (온톨로지 검색): "김대리" → Contact{name:김대리, dept:영업팀},
   Context{topic:Q1 리뷰 미팅}. 추출 키워드: {김대리, 영업팀, Q1 리뷰 미팅}

3. Phase 3 (온톨로지→메모리): "영업팀 Q1 리뷰 미팅" 재검색 → 추가 발견:
   "영업팀 주간회의 내용", "Q1 실적 보고서 작성"

4. Phase 4 (메모리→온톨로지): "2026-03-15 사무실 프로젝트" 재검색 → 추가 발견:
   Project{name:Q1 영업 프로젝트}, Meeting{date:2026-03-15, topic:Q1 리뷰}

최종 LLM 제공 문맥: 김대리의 발언 내용 + 소속 부서 + 관련 프로젝트 전체 맥락

#### 실시예 2: 시간 기반 질의

사용자 질의: "지난주에 한 일 정리해줘"

1. Phase 1: "지난주" → 최근 7일 대화 10건 발견. 추출: 다수의 인물/장소/행위

2. Phase 2: "지난주" → 최근 Action 10건 (5W1H 포함)

3. Phase 3: 온톨로지의 인물/프로젝트명으로 메모리 재검색 → 누락된 대화 5건 추가

4. Phase 4: 메모리의 시간/장소로 온톨로지 재검색 → 관련 Task/Meeting 3건 추가

최종: 시간순 통합 타임라인 + 관계 맥락 포함

저장소를 구성하고, 사용자 질의 시 양 저장소의 1차 검색 결과에서 각각

통합 문맥을 AI 모델에 제공하는 것을 특징으로 하는 AI 에이전트 기억 시스템.

청구항 18. 청구항 14에 있어서, 상기 에피소드 기억에서 추출하는 교차
검색 키워드는, 장기 기억으로 승격 시 부여된 구조화된 메타데이터
필드(시간, 장소, 상대방, 행위)를 정규 표현식으로 파싱하여 생성하는
것을 특징으로 하는 시스템.

청구항 19. 청구항 14에 있어서, 상기 에피소드 기억 저장소와 온톨로지
저장소의 변경 사항은 델타 기반 종단간 암호화 동기화 엔진을 통해 복수의
사용자 기기 간에 실시간 동기화되며, 릴레이 서버에 영구 저장되지 않는
것을 특징으로 하는 시스템.

청구항 20. 청구항 14에 있어서, AI 에이전트가 대화 개시 시 사용자의
질문 내용과 무관하게, 사용자의 성명, 호칭, 직업, 거주지, 가족관계,
생활 패턴, AI 호칭 선호를 포함하는 필수 프로필 키 집합을 장기 기억
저장소에서 자동으로 로드하여 LLM 컨텍스트에 주입하되, 상기 필수
프로필은 복수의 기기 간 델타 기반 동기화에 의해 실시간 동기화되는
것을 특징으로 하는 시스템.

청구항 21. 청구항 14에 있어서, 사용자의 비밀번호, 카드번호,

동기화 대상에서 제외되어 각 기기의 로컬 암호화 금고에만 저장되며,
장기 기억 저장소에는 해당 정보의 존재 여부만 기록되는 것을 특징으로
하는 시스템.

청구항 22. 청구항 14에 있어서, 대화 세션 로그가 시각 및 기기
식별자와 함께 자동 기록되고, 복수의 기기에서 동일 날짜에 발생한
세션 로그는 타임스탬프 순서로 병합되어 동기화되는 것을 특징으로

## [보강 실시예: 동기화 대상 기억 계층의 세부 구성]

> 2026-03-29 세션에서 추가된 기억 관리 혁신을 본 특허의 실시예로 보강.

### 실시예 8: 필수 사용자 프로필 자동 로드 및 동기화

본 발명의 일 실시예에서, AI 에이전트가 사용자와 대화를 시작할 때, 사용자의 질문 내용과 무관하게 **필수 프로필 키** 집합을 장기 기억 저장소에서 자동으로 로드하여 LLM 컨텍스트에 주입한다.

필수 프로필 키 집합은 다음을 포함한다:
- `user_profile_identity`: 성명, 호칭, 연령, 성별, 거주지, 국적, 학력
- `user_profile_family`: 배우자, 자녀, 부모, 형제, 친척 (성명 및 애칭 포함)
- `user_profile_work`: 직업, 직위, 회사, 사무실 주소, 동료, 전문 용어
- `user_profile_lifestyle`: 취미, 식성, 선호 식당, 생활 습관
- `user_profile_communication`: 말투, 표현 스타일, 격식 수준
- `user_profile_routine`: 일과, 통근 패턴, 정기 일정
- `user_moa_preferences`: AI 호칭, 사용자의 AI 호칭, 행동 선호

이들 키는 **쿼리 기반 메모리 검색(Phase 1)과 독립적으로** Phase 0에서 무조건 로드되며, 이로써 사용자가 "안녕"과 같은 단순 인사를 보내더라도 AI가 사용자의 이름, 직업, 선호 호칭을 즉시 알 수 있다.

상기 필수 프로필은 복수의 기기 간에 본 발명의 델타 기반 동기화를 통해 실시간 동기화되므로, 어떤 기기에서 프로필을 갱신하더라도 다른 모든 기기에서 즉시 반영된다.

### 실시예 9: 대화 세션 로그의 시간-기기 교차 기록

본 발명의 일 실시예에서, 의미 있는 대화가 시작되거나 주제가 전환될 때마다 다음 형식의 세션 로그를 자동 저장한다:

```
키: session_log_2026-03-29
값: [14:30 MoA-Desktop] 임대차 판례 검색 | [14:45 MoA-Desktop] 일정 확인 | [18:20 MoA-iPhone] 쿠팡 주문
```

세션 로그에는 **시각과 기기 식별자**가 함께 기록되며, 복수의 기기에서 동일 날짜에 대화한 내용이 하나의 키에 시간순으로 병합된다. 이 병합은 본 발명의 델타 기반 동기화 시 내용 추가(append) 전략으로 수행되며, 충돌 시 타임스탬프 순서로 정렬한다.

### 실시예 10: 민감 정보의 동기화 제외 및 로컬 전용 암호화 저장

본 발명의 일 실시예에서, 사용자의 민감 정보(비밀번호, 카드번호, 주민등록번호, 운전면허증번호, 여권번호 등)는 동기화 대상에서 **명시적으로 제외**되며, 각 기기의 로컬 암호화 금고(credential vault)에만 저장된다.

이는 동기화 엔진의 `memory_store`/`memory_observe`에 저장된 데이터는 릴레이 서버를 경유할 수 있으나, 암호화 금고의 데이터는 **절대 로컬을 벗어나지 않는** 설계 원칙에 기반한다.

장기 기억 저장소에는 민감 정보의 **존재 여부만** 기록한다:
```
키: user_private_id_number
값: "저장됨 (credential vault)" — 실제 번호는 포함하지 않음
```

이로써 다른 기기에서도 해당 정보가 특정 기기의 로컬에 저장되어 있음을 인지할 수 있으나, 실제 값에 접근하려면 해당 기기에서 직접 복호화해야 한다.

### 실시예 11: 메모리 recall 시 타임스탬프 포함

본 발명의 일 실시예에서, 장기 기억 항목을 LLM 컨텍스트로 전달할 때, 각 항목의 생성/갱신 타임스탬프를 함께 포함한다:

```
[User profile — always loaded]
- user_profile_identity: [2026-03-29T11:58] 김재철, 변호사, 서울 강남구
- user_profile_work: [2026-03-29T11:58] 법무법인 OO, 변호사

[Memory context]
- 임대차_판례_분석: [2026-03-29T14:30] 대법원 2026다12345 판례 분석 결과...
```

타임스탬프의 포함으로 LLM이 각 기억이 **언제** 형성되었는지를 인지하게 되어, 시간적 맥락을 고려한 응답 생성이 가능해진다. 이는 본 발명의 실시예 8(온톨로지 행동 이력 테이블)의 `occurred_at_utc`, `occurred_at_local`, `timezone` 필드와 연동되어, 기기 간 시간대가 다른 경우에도 정확한 시간 순서가 보장된다.
