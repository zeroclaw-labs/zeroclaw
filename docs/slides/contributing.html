How to contribute

A Rust-first agent runtime, dual-licensed MIT OR Apache‑2.0. Every kind of contribution counts — code, docs, bug reports, design feedback. All PRs target master — start small, validate locally, and one concern per PR.

The path — fork to merged PR start small
01
Grab a good first issue — typo, docs, tests, or a small bug is the fastest merge
02
Fork → branch feat/* / fix/* → change → test
03
Enable the gate: git config core.hooksPath .githooks (fmt · clippy · tests on push)
04
Validate full CI parity: ./dev/ci.sh all
05
Open a PR to master — complete every template section
Stuck? Open a draft PR early and ask in the description. Non-trivial architecture, config, security, or CI work? Read the architecture map first to find the right context.
Merge gates non-negotiable
  • One concern per PR — prefer small (XS / S / M); split big work into stacked PRs
  • Validation evidence is required — real command output, not "CI will check"
  • Privacy is a hard gate — never commit real names, emails, tokens, or PII
  • Squash-merge, conventional commits · AI-assisted is welcome, but no bot/AI attribution trailers
Know your risk tier sets the validation bar
  • LOWdocs · chore · tests-only
  • MEDIUMmost crates/*/src/** behavior changes
  • HIGHruntime/security · gateway · tools · workflows · access-control
When uncertain, classify higher.
Before you push — the local gate
cargo fmt --check› cargo clippy -D warnings› cargo test› ./dev/ci.sh all
