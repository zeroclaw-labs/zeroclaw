# Phase 1.4 — Summary

**Completed:** 2026-06-12
**Status:** ✅ Whole-crate drops + telemetry strip + MCP migration shipped. Quick-gate 23/23.

## What shipped

### Whole-crate drops (STRIP-01)
- 5 crate directories deleted from disk:
  - `crates/zeroclaw-hardware/` (9,545 LOC; USB/GPIO/I2C/SPI/RPi/datasheet)
  - `crates/robot-kit/` (3,468 LOC; ROS-adjacent robotics)
  - `crates/aardvark-sys/` (510 LOC; Total Phase Aardvark FFI)
  - `apps/tauri/` (758 LOC; desktop system tray)
  - `crates/zeroclaw-plugins/` (1,941 LOC; WASM/Extism plugin host)
- **Total**: ~16,222 LOC of consumer/IoT/hardware/desktop weight removed.

### Webhook channel removed (STRIP-06)
- `crates/zeroclaw-channels/src/webhook.rs` deleted (user-rejected on security grounds: "rather build a custom osAgent update for a weird stack than overengineer and give the capabilities to get hacked").
- `crates/zeroclaw-channels/src/lib.rs` `pub mod webhook` declaration removed.
- `channel-webhook` feature definitions stripped from root + zeroclaw-channels Cargo.toml.

### Telemetry strip (TELEMETRY-01)
- `crates/zeroclaw-runtime/Cargo.toml` `opentelemetry`, `opentelemetry_sdk`, `opentelemetry-otlp` optional deps deleted.
- `observability-otel` feature definition removed from zeroclaw-runtime + root Cargo.toml.
- `opentelemetry-otlp` ban in `deny.toml` is now meaningfully load-bearing (was unreachable when the optional dep existed).
- Audit findings: no `sentry`, `posthog`, `honeycomb`, or `datadog` deps anywhere in v0.7.5. The OTLP optional was the only phone-home pathway and is now gone.

### MCP file migration (Phase 1.3 deferred work)
- 5 MCP source files moved from `crates/zeroclaw-tools/src/mcp_*.rs` → `crates/osagent-tools-mcp/src/mcp_*.rs`.
- 5 duplicate MCP files deleted from `src/tools/mcp_*.rs` (these were the root-crate's redundant copy — dead weight).
- `crates/osagent-tools-mcp/src/lib.rs` upgraded from documentation stub to real `pub mod` declarations.
- `crates/zeroclaw-tools/src/lib.rs` MCP `pub mod` declarations replaced with a comment pointing consumers to `osagent_tools_mcp`.
- `crates/zeroclaw-runtime/src/tools/mod.rs` `pub use zeroclaw_tools::mcp_*` re-exports updated to `pub use osagent_tools_mcp::mcp_*`.
- `crates/zeroclaw-runtime/Cargo.toml` adds `osagent-tools-mcp = { path = "../osagent-tools-mcp" }` direct dep + comment documenting M3 runtime-split prerequisite.

### Workspace + root-crate cleanup
- Root `Cargo.toml` `[workspace].members`: 5 dead entries removed.
- Root `Cargo.toml` `[workspace.dependencies]`: 3 dead path-deps removed.
- Root `Cargo.toml` `[dependencies]` (the `zeroclawlabs` root crate): `zeroclaw-plugins`, `zeroclaw-hardware`, `aardvark-sys` removed.
- Root `Cargo.toml` `[features]`: `hardware`, `peripheral-rpi`, `probe`, `plugins-wasm`, `observability-otel` definitions removed; `ci-all` updated.
- `src/lib.rs` `pub mod peripherals` + `pub mod plugins` declarations removed.
- `src/main.rs` `mod hardware` + `mod peripherals` + `mod plugins` declarations removed.
- `src/hardware/`, `src/peripherals/`, `src/plugins/` directories deleted (root-crate's integration modules for the dropped crates).
- `crates/zeroclaw-gateway/Cargo.toml`: `zeroclaw-hardware`, `zeroclaw-plugins` deps removed; `plugins-wasm` feature definition removed.

## Pending Phase 1.5 cleanup

This phase intentionally left source-level cleanup of the remaining ~25 channels + ~50 providers + ~35 tools to Phase 1.5 (per the roadmap). The CI `workspace-build` job may fail until those source strips complete because:

- Source files (`src/agent/`, `crates/zeroclaw-runtime/src/`, etc.) likely still `use` the deleted modules (`use crate::hardware::*`, `use zeroclaw_plugins::*`).
- Tool/channel/provider registration in `crates/zeroclaw-runtime/` likely still references hardware tools, plugin tools, etc.
- `#[cfg(feature = "plugins-wasm")]` blocks throughout the source may now refer to a non-existent feature.

Phase 1.5 strips these cascading references. CI failure on Phase 1.4 is expected and accepted; the milestone-defining `wizard-no-mcp-gate` (Phase 1.3) remains green throughout because the wizard binary is hollow and has no source-level dependencies on the dropped crates.
