# Phase 1.2: Workspace Skeleton & Binary Split — Context

**Gathered:** 2026-06-12
**Status:** Ready for execution (infrastructure phase, inventory inlined)

<domain>
## Phase Boundary

Phase 1.2 establishes the two-binary workspace topology:
- New `bins/engineer/` and `bins/wizard/` Cargo packages added as workspace members.
- Each produces a binary that matches the agent name (`/usr/local/bin/engineer`, `/usr/local/bin/wizard`) per decision #31 drop-in-replace convention.
- Both are "hollow" at M1 (per FEATURES.md hollow-wizard pattern) — minimal `main.rs` that prints a placeholder and exits 0. The structural property that matters at M1 is the SHAPE of each binary's `Cargo.toml`, not the runtime behavior.
- Upstream's existing `[[bin]] zeroclaw` and `[[bin]] zeroclaw-acp-bridge` remain untouched. Phase 1.6 strips the latter; the former gets renamed/removed when osagent-engineer + osagent-wizard reach parity in M2+.

This phase does NOT yet:
- Extract `osagent-mcp` as a separate crate (that's Phase 1.3 — the load-bearing structural exclusion).
- Strip channels/providers/tools (Phase 1.4 / 1.5).

</domain>

<decisions>
## Implementation Decisions

### Inline inventory (RESEARCH.md merged into this CONTEXT.md for brevity)

Direct grep + find against the merged v0.7.5 tree resolved every open question Phase 1.2 needed answered:

1. **`inventory!` / `linkme::distributed_slice` / `ctor::ctor` in upstream?** — Zero occurrences across all `.rs` + `.toml` files. Upstream uses explicit registration today. Pitfall #6 (auto-discovery defeating structural feature exclusion) DOES NOT APPLY to v0.7.5. Phase 1.2 does not need a `Pattern 1 migration`; it only needs to RATIFY the existing pattern as a project rule (added to deny.toml ban list and to `osagent-policy.yml` CI as a grep gate).

2. **MCP module locations** — 5 files in `crates/zeroclaw-tools/src/mcp_*.rs` plus a mirror in `src/tools/mcp_*.rs`. The duplication between `src/` and `crates/zeroclaw-tools/src/` is an upstream quirk (the root crate has its own copy). Phase 1.3 will need to address both. For Phase 1.2: don't touch any MCP code yet.

3. **Existing binary topology** — root crate `zeroclawlabs` ships `[[bin]] zeroclaw` (default-run) + `[[bin]] zeroclaw-acp-bridge`. `bins/` directory does not exist. We add `bins/` cleanly.

4. **Workspace resolver** — already `resolver = "2"` (per Cargo.toml line 3). No change needed; WS-01 acceptance criterion satisfied by upstream.

### Decisions for this phase

- **Workspace layout** — new `bins/` directory at the workspace root with two member crates: `bins/engineer/` and `bins/wizard/`. Each contains `Cargo.toml` + `src/main.rs` (~30 LOC).
- **Package names** — `osagent-engineer` and `osagent-wizard` (long form for crate identity); binary names (the `[[bin]] name`) are `engineer` and `wizard` (short form for systemd drop-in).
- **Workspace.members** — append `"bins/engineer", "bins/wizard"` to the existing list in root `Cargo.toml`. Do NOT remove existing members; Phase 1.4 drops the dead ones.
- **Initial dependencies** — both binaries depend on nothing yet. The placeholder `main.rs` is pure `std`. As Phase 1.3+ extracts crates, engineer's `Cargo.toml` gets explicit deps; wizard's `Cargo.toml` gets a SUBSET (specifically: never depends on `osagent-mcp`).
- **`Cargo.toml` of each binary** — uses `version.workspace = true`, `edition.workspace = true`, `license.workspace = true`, `repository.workspace = true` to inherit. No per-binary version drift.
- **`[[bin]]` declaration vs implicit** — explicit `[[bin]] name = "engineer" path = "src/main.rs"` block (similar for wizard). This makes the binary name visible at the top of each `Cargo.toml`, which is decision #1's intent ("the Cargo.toml of each binary the human-readable manifest of compiled-in capabilities").
- **WS-04 explicit registration ratified** — add a comment to `bins/engineer/Cargo.toml` and `bins/wizard/Cargo.toml` that says: "Distributed-slice registration crates (`inventory`, `linkme`, `ctor`) MUST NOT be added as dependencies of this binary or its transitive deps. See deny.toml for the enforced ban list."
- **deny.toml extension (WS-04 enforcement)** — add `inventory`, `linkme`, `ctor` to `[bans].deny`. cargo-deny then fails any PR that introduces them.
- **WS-05 `default-features = false` audit** — `cargo tree --duplicates` script will be added in Phase 1.5 alongside MANIFEST.toml work. Phase 1.2 only confirms our two new Cargo.toml files use `default-features = false` for any deps they later add.

### Claude's Discretion (locked at execution time)

- Whether to wire up a basic `clap` `--version` flag in Phase 1.2 or defer to M2. Recommend defer — keep `main.rs` to pure-std at Phase 1.2; `clap` lands when there's actual functionality to flag-gate.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets

- Upstream `Cargo.toml` already pins `resolver = "2"` and uses `workspace.package` for shared metadata (version, edition, license, rust-version). Our two new binaries can inherit cleanly via `.workspace = true`.
- Existing `[[bin]] zeroclaw-acp-bridge` in `src/bin/zeroclaw-acp-bridge.rs` is a reference for how a small auxiliary binary is structured in this workspace. We use it as a template.

### Established Patterns

- Top-level `Cargo.toml` uses `[workspace.dependencies]` block for all shared crate path-deps. Our two binaries will eventually reference these by `dep:<name>` from their own `[dependencies]` blocks.
- No `inventory!` / `linkme!` / `ctor` usage. Explicit registration is the de-facto pattern. We ratify it in deny.toml.

### Integration Points

- `bins/engineer/` and `bins/wizard/` become workspace members. `cargo build --workspace` builds them alongside upstream's existing bins.
- Future: `bins/engineer/Cargo.toml` adds `osagent-tools-mcp` as a dep (Phase 1.3 onward); `bins/wizard/Cargo.toml` never adds it. This is the load-bearing structural exclusion property.

</code_context>

<specifics>
## Specific Ideas

- **Print build provenance in `main.rs` placeholder.** Use `env!("CARGO_PKG_VERSION")` + a `build.rs` (or `vergen`) that bakes in the upstream-zeroclaw SHA from `Cargo.toml` `[workspace.metadata.osagent]`. At Phase 1.2 we don't need a build.rs yet — just print `CARGO_PKG_VERSION` and the `[workspace.metadata.osagent]` SHA as a string literal in the source.
- **CI workflow extension for the new binaries.** Add a job `workspace-build` to `osagent-policy.yml` that runs `cargo build -p osagent-engineer -p osagent-wizard --release` (cheap; placeholder mains compile in <30s). Catches future regressions where someone breaks the binary's compile by adding a misconfigured dep.

</specifics>
