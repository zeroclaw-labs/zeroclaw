# Phase 1.1: Fork & Attribution & Sync Runbook - Context

**Gathered:** 2026-06-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 1.1 delivers the legal + operational foundation for a long-lived public fork of zeroclaw v0.7.5: a true GitHub fork relationship (`andreas2301/osAgent`), preserved upstream license + NOTICE, our own attribution, a working branch (`osagent-main`), the `cargo deny` license + advisory + bans configuration enforced in CI from PR #1 forward, and a written quarterly upstream-sync runbook documented in `sovereign-shield-backup/documentation/osAgent/UPSTREAM_SYNC.md`.

This phase produces ZERO Rust source code changes. The fork's `osagent-main` branch starts as an exact match of `upstream/v0.7.5`. All restructure, strip, and feature work happens in 1.2+.

The exit condition is: `cargo deny check` is green on the first PR opened against `osagent-main`, the UPSTREAM_SYNC.md runbook exists and is approved by the project owner, and the fork relationship is visible on GitHub's fork badge.

</domain>

## Implementation Decisions

### Pure infrastructure phase

All implementation choices are at Claude's discretion within the constraints below. The 42 ratified decisions in PROJECT.md plus the 3 cross-cutting refinements in SUMMARY.md cover every grey area for this phase. No grey-area discussion needed.

### Hard constraints (from PROJECT.md / research)

- **Fork target**: `zeroclaw-labs/zeroclaw` @ tag `v0.7.5` (SHA verified at install-guide pin: `8bc8276a8d8faefb3e4a824f33876929e7466f632ee7c5363936368a1af2e4f7` for the x86_64-linux release asset, but for fork we work from the git tag, not the release asset)
- **Fork destination**: `andreas2301/osAgent`, public, GitHub fork (true relationship, not standalone repo)
- **Working branch**: `osagent-main`. Upstream tracking branch: `upstream-master`. Tag mirror: keep upstream tags accessible via `upstream/v*`.
- **License**: dual MIT OR Apache-2.0 inherited from upstream. NOTICE preserved + osAgent NOTICE added with fork provenance (commit SHA of `upstream/v0.7.5`, fork date, project URL).
- **Versioning**: `0.1.0+zeroclaw-0.7.5` on the workspace package. Independent semver from this point.
- **`cargo deny` configuration** must enforce:
  - License allowlist: MIT, Apache-2.0, Apache-2.0 WITH LLVM-exception, BSD-2-Clause, BSD-3-Clause, ISC, Unicode-DFS-2016, Zlib
  - License banlist: AGPL-3.0 (target: presage, libsignal-service, libsignal), WTFPL (target: frankenstein), GPL-3.0 (target: any direct dep — signal-cli is acceptable because it's a separate process / mere aggregation, not a Rust dep)
  - Advisories: pull rustsec advisory DB current; deny on any unmaintained, yanked, or with-vulnerability entry
  - Sources: only crates.io + git-with-rev pins. No path deps outside the workspace.
  - Bans: deny direct deps on `sentry`, `posthog-rust`, `honeycomb`, `opentelemetry-otlp` (these are phone-home transports; opentelemetry as a tracing facade is fine if not exporting)
- **CI host**: GitHub Actions on the public fork; self-hosted runner labeled `osagent-release` for signed releases (decision #40). PR jobs use only public hosted runners.
- **UPSTREAM_SYNC.md** documentation target: `sovereign-shield-backup/documentation/osAgent/UPSTREAM_SYNC.md`. Cadence: 1st week of Q1/Q2/Q3/Q4. Out-of-cycle merges only for upstream security advisories. Per-quarter diff-stat budget: `<2000 LOC delta` automatic; `>2000 LOC` mandatory per-file review.

### Claude's Discretion

- Exact `cargo deny` strictness on duplicate dependencies (`--multiple-versions warn` vs `deny`): default to `warn` at M1 (lots of upstream deps); plan to escalate to `deny` once the strip phases (1.4 + 1.5) settle the dep tree
- CI workflow file structure: single `.github/workflows/ci.yml` with named jobs (`license`, `audit`, `nm-gate-placeholder`, `build`) vs multiple files. Recommend single file at this phase; split if it grows >300 lines.
- NOTICE format: short attribution paragraph + `---` + verbatim upstream NOTICE content. Standard pattern.
- UPSTREAM_SYNC.md location detail: file directly in `documentation/osAgent/` or under `documentation/osAgent/runbooks/UPSTREAM_SYNC.md`. Recommend direct child of `osAgent/` for discoverability; documentation README links to it.
- Whether to also add `cargo about` for license-summary auto-generation. Recommend yes — `cargo about generate about.hbs > NOTICE.md` keeps NOTICE accurate as deps churn.

<specifics>
## Specific Ideas

- **Mirror zeroclaw's upstream-tag taxonomy.** When the fork is created, ensure all upstream tags (v0.7.5 and older) are visible on `andreas2301/osAgent` so contributors can see the fork point. `git push --tags upstream-master` after the initial fork-and-clone.
- **`cargo deny` config in repo root**: `deny.toml` at the workspace root. Include version (`[graph] all-features = false; no-default-features = false`) and feature scope explicit so resolver=2 unification surprises don't silently mute bans.
- **UPSTREAM_SYNC.md must include a worked example.** First-time sync run is documented as a "tutorial" inside the runbook (using a hypothetical zeroclaw v0.8.0 as the example target). Reduces ambiguity when a future session has to actually do the sync.

</specifics>
