### 1.2 The mechanics question

Two options were considered:

| Option | Description | Verdict |
|---|---|---|
| **(a)** `gh repo fork zeroclaw-labs/zeroclaw --clone --remote` to a new dir, `mv` `.planning` in | **REJECTED.** Orphans the 5 existing planning commits (or forces a copy-without-history). Loses git provenance on the planning work. |
| **(b)** Configure remotes manually in the existing repo | **CHOSEN.** Preserves all 5 planning commits; lets us create `osagent-main` as a true merge of `upstream/v0.7.5` and the planning history. |

`gh repo fork` documentation confirms (verified at https://cli.github.com/manual/gh_repo_fork 2026-06-12): the `--clone` flag prompts the user; the behavior when the destination directory exists with content is **NOT specified** in the manual (implementation detail). Even if it worked, it would not solve the "preserve 5 commits of planning history" problem.

### 1.3 Recommended sequence (Path B)

**Step A — Create the GitHub-side fork relationship (no local clone):**

```bash
# Run from any directory; we're using gh ONLY for the GitHub-side fork registration.
gh repo fork zeroclaw-labs/zeroclaw \
  --org andreas2301 \
  --fork-name osAgent \
  --clone=false \
  --remote=false \
  --default-branch-only=false   # we want all upstream tags + branches
```

Notes:
- `--clone=false`: do not clone locally (we already have a local repo).
- `--remote=false`: do not auto-add a remote in the current directory's repo (we will configure remotes manually in the osAgent repo so the names match our convention).
- `--default-branch-only=false`: import all branches, not just upstream's default. This ensures `git fetch upstream --tags` later pulls v0.7.5.
- `--org andreas2301`: forks to the `andreas2301` user (treating it as the owner namespace — `gh` accepts both user and org names here).
- The `--fork-name osAgent` flag may not exist in older `gh` versions; if `gh --version` < 2.32, the fork inherits the parent name (`zeroclaw`) and must be renamed via `gh repo rename andreas2301/zeroclaw osAgent` immediately after.

Verification:
```bash
gh repo view andreas2301/osAgent --json parent --jq '.parent.nameWithOwner'
# Expected output: zeroclaw-labs/zeroclaw
```

**Step B — Configure remotes in the existing local repo:**

```bash
cd d:/Repositories/osAgent

# 1. Add origin pointing to our new fork
git remote add origin https://github.com/andreas2301/osAgent.git

# 2. Add upstream pointing to zeroclaw-labs/zeroclaw
git remote add upstream https://github.com/zeroclaw-labs/zeroclaw.git

# 3. Fetch upstream tags + history
git fetch upstream --tags

# 4. Verify v0.7.5 is reachable
git rev-parse upstream/v0.7.5
# Expected output: a2988f0dfffa0c14fa56e218c7bb9f28da494da4
```

**Step C — Create `osagent-main` as the working branch:**

The challenge: `master` currently has 5 `.planning/` commits with no shared history with `upstream/v0.7.5`. Three viable strategies:

| Strategy | Result | Recommendation |
|---|---|---|
| **C1: Merge with `--allow-unrelated-histories`** | `osagent-main` has both planning + upstream code on two parents. Clean history, both lineages visible. | **CHOSEN.** Matches GitHub fork-from-existing-state pattern; quarterly upstream merges (Phase 1.1's runbook) use the same `git merge` machinery. |
| **C2: Cherry-pick planning commits onto `upstream/v0.7.5`** | Loses original commit SHAs; planning commits get new SHAs but same content. | Acceptable, but breaks any external references to the original SHAs (none exist yet — `master` was never pushed). |
| **C3: Replace `master` with `upstream/v0.7.5`, add planning as a sub-directory in a fresh commit** | Loses planning commit granularity. | Rejected. We want the 5 planning commits visible as history. |

```bash
# C1 — chosen strategy
git checkout -b osagent-main upstream/v0.7.5
git merge --allow-unrelated-histories master -m "Merge .planning bootstrap onto upstream v0.7.5

Combines the 5 .planning/ commits from master with upstream zeroclaw v0.7.5
(SHA a2988f0dfffa0c14fa56e218c7bb9f28da494da4) as the canonical osagent-main
starting point.

Refs: FORK-01"

# Delete the bootstrap master branch (its history is now merged into osagent-main)
git branch -D master

# Push osagent-main as the default branch
git push -u origin osagent-main

# Push the upstream tags so contributors can see the fork point
git push origin 'refs/tags/v0.7.*'   # only the v0.7.x tags relevant to us
```

**Step D — Configure GitHub-side defaults:**

```bash
# Set osagent-main as the default branch on origin
gh repo edit andreas2301/osAgent --default-branch osagent-main

# (Optional but recommended) Delete the auto-created master branch on origin
gh api -X DELETE repos/andreas2301/osAgent/git/refs/heads/master 2>/dev/null || true
```

**Step E — Add upstream-tracking branch (for the quarterly sync workflow):**

```bash
# upstream-master is a mirror branch we update at each quarterly sync.
# At first sync it's identical to upstream/v0.7.5; future syncs reset it.
git checkout -b upstream-master upstream/v0.7.5
git push -u origin upstream-master
git checkout osagent-main
```

This dedicated branch makes diffing trivial during sync: `git log osagent-main..upstream-master` shows pending upstream changes.

**Key elements (each load-bearing):**

1. **"derivative work of ZeroClaw"** — Apache-2.0 §4(b) requires "prominent notices" of modification. This sentence is that notice.
2. **"NOT ZeroClaw" disambiguation paragraph** — defensive against upstream's TRADEMARK.md assertion that "the only official ZeroClaw repository is...". Eliminates any ambiguity about authority.
3. **Fork commit SHA + fork date** — provenance for the merge-base of upstream code in the fork. Anyone can `git log <SHA>` and see the bootstrap commit. Survives every future merge.
4. **"same dual-license model"** — explicit re-declaration of MIT OR Apache-2.0; required because Apache-2.0 §4(a) requires modifications to ship with a copy of the license.
5. **Verbatim upstream NOTICE included below `---`** — Apache-2.0 §4(d) requires unchanged retention of any NOTICE text from the original work. The `---` separator + "full upstream ZeroClaw NOTICE follows below" header makes the distinction unambiguous.

## 3. cargo deny + CI

### 3.1 `deny.toml` body (place at osAgent workspace root)

This config is verified against the **current cargo-deny schema** (https://embarkstudios.github.io/cargo-deny/checks/{cfg,licenses/cfg,bans/cfg,advisories/cfg,sources/cfg}.html, all fetched 2026-06-12 with cargo-deny 0.19.8 published 2026-05-28 as latest release).

**IMPORTANT SCHEMA UPDATES** (relative to older cargo-deny docs you may remember):

| Field | Status | Replacement |
|---|---|---|
| `[licenses] unlicensed = "deny"` | **REMOVED** | Open issue/PR if a crate is unlicensed; the `allow = [...]` allowlist is implicitly closed (everything not in `allow` is denied) |
| `[licenses] copyleft = "deny"` | **REMOVED** | Implicitly denied by the allow-only model |
| `[licenses] default = "deny"` | **REMOVED** | Implicitly denied by the allow-only model |
| `[licenses] allow-osi-fsf-free = "neither"` | **REMOVED** | Implicitly denied by the allow-only model |
| `[advisories] vulnerability = "deny"` | **REMOVED** | All vulnerabilities now emit errors by default; use `ignore = [...]` for exceptions |
| `[advisories] severity-threshold = ...` | **REMOVED** | All vulnerabilities are errors; severity is informational only |
| `[advisories] notice = "warn"` | **REMOVED** | All notices now emit errors by default |

The modern schema is **allow-only** for licenses and **deny-by-default** for advisories. The body below reflects that.

discussion in the project's planning notes about whether to skip this upstream release

## 5. `cargo about` deferral analysis

CONTEXT.md confirms: defer to Phase 1.5. Verify this is correct.

**Question:** is a hand-written NOTICE legally sufficient at Phase 1.1, or do we need `cargo about` now?

**Answer:** hand-written NOTICE is sufficient at Phase 1.1. Reasoning:

1. **Apache-2.0 §4 requirements** ("retain notices", "include modifications notice", "include NOTICE text") apply to the **distribution of the work**. At Phase 1.1, we are not distributing binaries — we are publishing source on a public repo. The source NOTICE file already satisfies §4(d) by retaining upstream's NOTICE verbatim.
2. **Per-crate transitive-dep license aggregation** is a binary-distribution concern (operators of binaries want to know what they're actually shipping). At Phase 1.1, no binary exists. Phase 1.5 ships the first releasable binary (alongside MANIFEST.toml) and IS the correct place to wire `cargo about generate about.hbs > NOTICE.md`.
3. **The hand-written NOTICE in Section 2.2 above** is what's actually required by Apache-2.0 §4(b) ("prominent notices of modification"). `cargo about` would generate a *third-party-licenses summary* — a different document, useful for SBOM hygiene but NOT a replacement for the modification notice.

**Conclusion:** CONTEXT.md's defer-to-1.5 decision is correct. Phase 1.1 ships hand-written NOTICE. Phase 1.5 adds `cargo about` for the runtime-binary-shipped third-party-license summary.

| License/ban enforcement | Custom shell script grepping `Cargo.lock` | `cargo deny check` (Section 3.1) | cargo-deny handles SPDX parsing, exception clarification, transitive-dep walking, advisory DB sync, and CI integration. A shell script handles none of these correctly. |

| Conflict resolution log format | Free-form prose at the bottom of `UPSTREAM_SYNC.md` | Markdown table with Date / Upstream Tag / File / Conflict Type / Resolution / Resolver columns (Section 4.3 §7) | Tables are diffable; prose isn't. A grep-able table also enables "show me every libsignal-adjacent merge conflict we've ever resolved." |
| Fork-relationship registration | `git clone` + push to a fresh GitHub repo with the same name | `gh repo fork` (Section 1.3 Step A) | A "true" fork shows the GitHub fork badge, lets us submit upstream PRs via the fork relationship, and inherits the network graph. **Key insight:** every component of Phase 1.1 has a mature, well-maintained tool. Hand-rolling supply-chain CI is the single most common project-rot vector for Rust forks — `cargo deny` exists precisely to prevent that.

## Common Pitfalls

### Pitfall 1: GitHub Action tag drift

**What goes wrong:** `EmbarkStudios/cargo-deny-action@v2` floats — upstream can rewrite the tag. A malicious or compromised tag rewrite would silently change what runs in CI.
**Why it happens:** convenience of `@v2` over `@<sha>`.
**How to avoid:** pin every action by 40-char SHA, as in Section 3.2.
**Warning signs:** CI behavior changes with no PR; `cargo deny` produces different output between runs of the same SHA.

### Pitfall 2: License-allowlist closed model misunderstood

**What goes wrong:** Engineer writes `default = "deny"` in `deny.toml` (an old-schema field), expecting it to do something. cargo-deny 0.19+ ignores it. They think the policy is closed-by-default when it actually... still is, but for a different reason.
**Why it happens:** stale cargo-deny tutorials online still show the old schema.
**How to avoid:** use the modern v2 schema (Section 3.1). The allow-only model is the modern default; do NOT use `unlicensed`/`copyleft`/`default`/`allow-osi-fsf-free` — they have been REMOVED. The presence of the `allow = [...]` list IS the closed model.
**Warning signs:** `cargo deny check licenses` warns about unknown fields in `deny.toml`.

### Pitfall 3: `gh repo fork` rename race

**What goes wrong:** Old `gh` versions (< 2.32) don't support `--fork-name`; the fork lands at `andreas2301/zeroclaw` not `andreas2301/osAgent`. Engineer panics, tries to delete and re-fork, gets cooldown-blocked by GitHub.
**Why it happens:** `gh repo fork` semantics changed across versions; current docs assume latest.
**How to avoid:** check `gh --version` BEFORE running fork. If < 2.32, do a fork-then-rename: `gh repo fork zeroclaw-labs/zeroclaw --clone=false --remote=false && gh repo rename andreas2301/zeroclaw osAgent`.
**Warning signs:** `gh repo fork` errors with "unknown flag --fork-name".

### Pitfall 4: Quarterly merge degenerates (Q1 clean, Q4 catastrophic)

**What goes wrong:** First quarter's merge is clean. Second quarter, a conflict is patched ad-hoc without a log entry. Third quarter, the patch reappears as a conflict because nobody remembers why we resolved it that way. Fourth quarter, the merge is so painful nobody runs it; fork rots.
**Why it happens:** no append-only memory of past conflict resolutions.
**How to avoid:** UPSTREAM_SYNC.md §7 conflict log is append-only by convention; every conflict gets a row BEFORE the resolution is staged. Reviewers reject sync PRs missing log entries.
**Warning signs:** quarterly sync PR description says "merged cleanly" but `git diff` shows non-trivial conflict resolutions.

### Pitfall 5: First-PR cargo-deny failure from upstream license heterogeneity

**What goes wrong:** Phase 1.1 PR opens against `osagent-main`. `cargo deny check licenses` fails because upstream zeroclaw v0.7.5 has a transitive dep with license `LGPL-2.1` or similar not in our allowlist. CONTEXT.md exit condition fails.
**Why it happens:** the upstream Cargo.toml dep graph at v0.7.5 has crates we'll strip in Phase 1.4–1.5, but they're present at Phase 1.1 because we inherit the whole tree.
**How to avoid:** Phase 1.1 PR description warns reviewers of this likelihood; if it happens, two surgical responses:
  - (a) add narrow `exceptions = [{ name = "<crate>", allow = ["LGPL-2.1"] }]` entries with a `TODO(phase-1.4)` comment
  - (b) temporarily set the failing check to `"warn"` (e.g., set `[bans] multiple-versions = "warn"`) with a documented commit message
The Phase 1.4 / 1.5 strips remove the offending deps; that's when we restore strict-deny.
**Warning signs:** first PR's `cargo deny check` fails on a crate that's already in the planned strip list.

### Pitfall 6: Hand-written NOTICE inadequate for Apache-2.0 §4(b)

**What goes wrong:** NOTICE only includes the osAgent paragraph; misses the "verbatim upstream NOTICE" requirement. Distribution violates Apache-2.0.
**Why it happens:** engineer reads `cp ../zeroclaw/NOTICE NOTICE` was unnecessary because "we have our own NOTICE now."
**How to avoid:** the template in Section 2.2 ends with `---\n[VERBATIM CONTENTS OF UPSTREAM NOTICE]` — literally copy the upstream NOTICE in below the separator. The acceptance check `grep -q "This product includes software developed at ZeroClaw Labs" NOTICE` confirms it.
**Warning signs:** NOTICE file is shorter than the upstream NOTICE file.

### Pitfall 7: `upstream-master` branch confused with `osagent-main`

**What goes wrong:** Engineer accidentally pushes work to `upstream-master`, forces upstream's tag history to drift, breaks the quarterly diff mechanism.
**Why it happens:** auto-completion in `git push origin <TAB>` lists both; no protection on `upstream-master`.
**How to avoid:** add branch protection on `upstream-master` (only the runbook script may force-push); document in UPSTREAM_SYNC.md §2 that `upstream-master` is mirror-only.
**Warning signs:** `git log upstream-master --not upstream/v0.7.5` shows commits — the mirror has drifted.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|---|---|---|---|
| `[licenses] unlicensed = "deny"`, `copyleft = "deny"`, `default = "deny"` | Allow-only model (just `allow = [...]`) | cargo-deny 0.15+ (mid-2024); fully enforced by 0.18 | Old tutorials will produce "unknown field" warnings; modern config is shorter |
| `[advisories] vulnerability = "deny"` etc. | All advisories error by default; `ignore = [...]` for exceptions | cargo-deny 0.15+ | More aggressive default; harder to silently downgrade severity |
| `[bans] deny = ["crate-name"]` (string form) | `[bans] deny = [{ crate = "crate-name", reason = "..." }]` (table form with reason) | cargo-deny 0.14+ | Better failure messages; old form still accepted but deprecated |
| `EmbarkStudios/cargo-deny-action@v1` | `@v2` (SHA-pinned) | v2 released 2024-10; v2.0.20 is current as of 2026-05-28 | v2 simpler config; SHA pin is independent good practice |

**Deprecated/outdated:**
- `name`/`version` pair fields in package specs → use `crate = "name@version"` table form (removed in cargo-deny 0.18+)
- `--features` action input → use `arguments: --features ...` instead
