## Critical clarification on M1 scope

This phase ships the **structural framework + CI gate**. The **physical migration** of MCP source files (`crates/zeroclaw-tools/src/mcp_*.rs` + `src/tools/mcp_*.rs` → `crates/osagent-tools-mcp/src/`) is deferred to Phase 1.4 (whole-crate drops) and Phase 1.5 (source strips) where the same files get refactored as part of the broader strip work.

**Why this is safe**: the property "wizard binary contains zero MCP code" is structurally true at M1 because `bins/wizard/Cargo.toml` has no `[dependencies]` block content. The CI gate makes that property load-bearing for future PRs. The MCP code's physical location is an implementation detail for Phase 1.4/1.5 to address.

**M3 prerequisite (documented here for forward planning)**: when the wizard binary grows real dependencies in M3, `crates/zeroclaw-runtime` must be split into `core` (no MCP) + `mcp-glue` (engineer-only). Otherwise wizard's `osagent-runtime` dep would pull MCP transitively. This split is the only known forward-blocker; documenting it now keeps M3 honest.
