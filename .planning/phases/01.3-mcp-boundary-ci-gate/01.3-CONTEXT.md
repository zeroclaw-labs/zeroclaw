# Phase 1.3: MCP Boundary & 4-Layer CI Gate — Context

**Gathered:** 2026-06-12
**Status:** Ready for execution (milestone-defining phase)

<domain>
## Phase Boundary

Phase 1.3 is the **milestone-defining gate** for M1. The wizard binary must contain ZERO MCP code, provable on every PR and release build by a 4-layer CI gate (source-grep + `nm --defined-only` + `cargo-bloat --crates` + `strings`).

Two complementary deliverables:
1. **Crate boundary**: `crates/osagent-tools-mcp/` exists as a workspace member. The wizard binary's `Cargo.toml` does NOT depend on it (and never will). The engineer binary's eventual transitive path goes through `crates/zeroclaw-runtime` (M3-prerequisite split notwithstanding).
2. **4-layer CI gate**: `scripts/wizard-no-mcp-gate.sh` runs all four layers locally; `.github/workflows/osagent-policy.yml` `wizard-no-mcp-gate` job runs them in CI after a release build of the wizard binary.

At M1 (this phase), all 4 layers pass trivially because the wizard binary is the hollow placeholder from Phase 1.2. The gate's real value materializes in M2+ when wizard's `[dependencies]` block grows. The gate then catches any PR that would inadvertently introduce an MCP dependency edge.

</domain>

<decisions>
## Implementation Decisions

### Minimum viable Phase 1.3 (the M1 boundary)

We ship the **structural framework** (crate exists, CI gate wired) but defer the **physical MCP file migration** to Phase 1.4/1.5 alongside the broader source-strip work. Rationale: physically moving `crates/zeroclaw-tools/src/mcp_*.rs` (5 files) + `src/tools/mcp_*.rs` (5 mirror files) into `crates/osagent-tools-mcp/` requires:
- Updating every `pub mod mcp_*` declaration in `zeroclaw-tools/src/lib.rs`
- Updating every `pub use zeroclaw_tools::mcp_*` re-export in `zeroclaw-runtime/src/tools/mod.rs`
- Adding the new crate as a dep in `zeroclaw-runtime/Cargo.toml`
- Verifying the existing `zeroclaw` binary still builds (no local cargo)

That's substantive code refactor work better done atomically with the Phase 1.4 whole-crate drops and Phase 1.5 source strips. Phase 1.3's milestone-defining property (4-layer CI gate green on every PR) does NOT require the physical migration to be complete — it requires that the wizard binary does not pull in MCP, which is already structurally true since wizard's `[dependencies]` is empty.

### Crate stub vs full crate
- `crates/osagent-tools-mcp/Cargo.toml` exists, listing package name + inheriting workspace metadata, empty `[dependencies]`.
- `crates/osagent-tools-mcp/src/lib.rs` documents the crate's purpose + the "Phase 1.4/1.5 migrates real code here" status note.
- Added to root `Cargo.toml` `[workspace].members`.

### 4-layer gate composition (per PITFALLS.md Pitfall 17)
- **Layer 1**: `grep -nE '^(osagent-tools-mcp|osagent-mcp)\s*(=|\.)' bins/wizard/Cargo.toml` + `grep -rnE '(use|extern crate)\s+osagent_tools_mcp' bins/wizard/src/`. Catches Cargo.toml dep declarations AND Rust source imports. Tolerates safety-prose comments.
- **Layer 2**: `nm --defined-only target/release/wizard | grep -iE '(\bmcp\b|McpClient|McpRegistry|McpTool|McpProtocol|McpTransport|McpDeferred)'` must be empty.
- **Layer 3**: `cargo bloat --crates --release -p osagent-wizard | grep -iE 'osagent-tools-mcp|osagent_tools_mcp'` must be empty.
- **Layer 4**: `strings target/release/wizard | grep -iE '(mcp_|model\.context\.protocol|McpClient|McpRegistry|McpToolWrapper)'` must be empty.

### Local vs CI execution
- The 4-layer script gracefully skips Layers 2/3/4 when the corresponding tool isn't available locally (Windows hosts lack `nm`/`strings`; `cargo-bloat` is install-on-demand). CI installs all tooling and runs all 4 layers on Linux.

### M3 prerequisite (documented, not yet implemented)
- For the wizard binary to grow real functionality (M3) while preserving the structural exclusion property, `crates/zeroclaw-runtime` must be split into `core` (no MCP) + `mcp-glue` (engineer-only). Engineer binary depends on both; wizard binary depends only on core. Otherwise wizard's eventual `osagent-runtime` dep would pull in MCP transitively.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- Phase 1.2 left `bins/wizard/Cargo.toml` with empty `[dependencies]` and a prominent safety-prose comment block naming `osagent-tools-mcp` as the forbidden dep edge. Phase 1.3's gate immediately latches onto that intent.

### Established Patterns
- CI workflow file (`osagent-policy.yml`) accumulates one job per phase's structural property. Phase 1.1 added `cargo-deny-*`. Phase 1.2 added `no-distributed-slice-registration` + `workspace-build`. Phase 1.3 adds `wizard-no-mcp-gate`.

### Integration Points
- `wizard-no-mcp-gate` job in CI uses `needs: [workspace-build]` so it runs only after the wizard binary release build succeeds. Sharing `Swatinem/rust-cache` key `workspace-build` between the two jobs avoids redundant compilation.

</code_context>

<specifics>
## Specific Ideas

- Stub crate's `lib.rs` is mostly documentation — the comment block reads as "this is where the MCP code WILL live; here's why; here's what the gate enforces." Future-Claude reading the crate gets the full picture without `.planning/` archaeology.
- Layer 1 of the gate is the most important at M1 because it's the only one that runs on Windows hosts. Layers 2/3/4 fire on CI only.

</specifics>

## Deferred Ideas

- **Physical MCP file migration** from `crates/zeroclaw-tools/src/mcp_*.rs` and `src/tools/mcp_*.rs` into `crates/osagent-tools-mcp/src/` → Phase 1.4 (alongside whole-crate drops) or Phase 1.5 (alongside source strips).
- **`crates/zeroclaw-runtime` split into core + mcp-glue** → M3 prerequisite, documented here for forward planning.
- **Active enforcement of `[bans].deny` for distributed-slice crates at the CI level** — already shipped in Phase 1.2.
