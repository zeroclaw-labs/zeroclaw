# [Feature]: ACP protocol extensions for diff/file-proposal message types

ACP protocol extensions for diff display and file-proposal message types. Enables TUI and web dashboard clients to show side-by-side diffs during `file_edit`/`file_write` approval cycles, and allows users to counter-propose edits ("no, use this text instead").

### What's already shipped

Diff display for `file_edit` and `file_write` tool calls in the ACP approval flow, shipped in PR #6649 (`5dd458004`):

- **`to_acp_content()`** (`acp_server.rs:1370`): emits `[{ "type": "diff", "path": ..., "oldText": ..., "newText": ... }]` content items for `file_edit`, and `[{ "type": "diff", "path": ..., "newText": ... }]` for `file_write` — ACP clients (Toad, Zed) render these as side-by-side diff editors
- **`to_acp_raw_input()`** (`acp_server.rs:1346`): translates ZeroClaw's internal `old_string`/`new_string` snake_case field names to ACP's `oldText`/`newText` camelCase schema

### What remains

The "counter-edit" flow. Currently `request_choice` (`acp_channel.rs:245`) only supports `allow_once` / `reject_once` option kinds — a binary approve/reject. There is no way for the user to reject a file edit and provide replacement content.

- [ ] New `reject_with_edit` option kind on `session/request_permission` — a rejection that carries replacement content in the response payload
- [ ] ACP server: handle `reject_with_edit` response, feed replacement content back to the agent as the tool result instead of the original edit
- [ ] ACP client (TUI): approval prompt → "reject with edit" → suspend TUI → open `$EDITOR` with the proposed content → return edited buffer as the counter-proposal
- [ ] Agent loop: receive the counter-proposal and apply the user's version instead of the original tool output
