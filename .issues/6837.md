# [Feature]: Runtime RPC dispatch layer and Unix socket transport

Runtime-level RPC dispatch and Unix socket transport for the daemon, enabling the TUI (and future clients) to connect without routing through the HTTP/WS gateway. Same operations, different pipe. The gateway remains for HTTP/webhook consumers — this is a peer transport, not a replacement.

All client surfaces (TUI, web dashboard, CLI) share common API primitives. The RPC dispatch layer is the shared core; the gateway and the socket transport are both thin adapters over it.

### RPC Dispatch Layer (`crates/zeroclaw-runtime/src/rpc/`)

- [x] Transport-agnostic request/response/error types — domain errors, not HTTP status codes
- [x] JSON-RPC 2.0 method dispatch routing to existing runtime functions:
  - Sessions: create, resume, list, prompt (streamed), abort, delete
  - Memory: list, search, store, delete
  - Cron: list, add, patch, delete, run history
  - Config: get, set, validate, reload
  - Agents: list, status
  - Cost: query, rollups
  - Skills: list, install, uninstall
  - Logs: subscribe (streaming)
  - Status: version, health
- [x] Bidirectional streaming for agent turns via JSON-RPC notifications (reuse `TurnEvent` enum)
- [ ] Tool approval request/response flow (same semantics as gateway `approval_request`/`approval_response`)
- [x] Session-scoped runtime overrides — per-session model, risk profile, runtime profile, reasoning effort changes without daemon reload (#6817); overrides are session-scoped, shadow config defaults, do not persist, do not affect other sessions, evaporate on session end
- [ ] Shared with existing ACP server protocol where possible — consolidate the ~800-1000 lines of duplicated session/turn logic between `ws.rs` and `acp_server.rs`

### Unix Socket Transport

- [x] Daemon listens on `$ZEROCLAW_SOCKET` or `$XDG_RUNTIME_DIR/zeroclaw/daemon.sock` (fallback: `~/.zeroclaw/daemon.sock`)
- [x] NDJSON framing (newline-delimited JSON-RPC 2.0) — no HTTP overhead
- [x] Spawned by `daemon::run()` as a supervised subsystem via `spawn_component_supervisor` (peer to gateway, channels, heartbeat)
- [x] New callback in `DaemonSubsystems` (e.g. `socket_start`)
- [x] ~~Authentication: pairing token validation on connect (same tokens as gateway)~~ Filesystem permissions (0o600 socket) handle local auth; pairing tokens removed from socket transport per 91c5211
- [x] Multiple concurrent client connections

### Connection Handshake

- [x] Version match enforcement on connect — client sends its version in the initial handshake, daemon rejects with a structured error if versions are incompatible (same check as `/api/status` provides today)
- [ ] Handshake response includes daemon version, capabilities, and active agent list

### Client Connection Lifecycle

- [ ] Structured close frame with reason code + human-readable message (intentional shutdown, reload, error)
- [ ] Close reason propagation — both sides know intentional vs unintentional disconnect
- [x] No heartbeat pings — rely on kernel fd close detection for liveness
- [x] Track connected client count per socket fd — expose to `--ephemeral` daemon logic (#6818)
- [ ] Reconnect token — on reconnect, client re-authenticates and resumes prior sessions without the daemon treating it as a new client; ephemeral client count tracks logical clients, not raw fd count
- [ ] Session resumption by ID on reconnect (client does not need to restart)

### WSS Transport (remote connections)

- [ ] `--connect wss://host:port` for remote daemon connections (TUI client-side)
- [ ] Same JSON-RPC protocol, WebSocket binary/text framing
- [ ] TLS via `rustls` + `tokio-rustls` (already in dependency tree)
- [ ] `tokio-tungstenite` for client-side WS (already in dependency tree)
- [ ] No new dependencies required

### vsock Transport (future, feature-gated)

- [ ] `vsock` Cargo feature gate (off by default): `tokio-vsock = { version = "...", optional = true }`
- [ ] Transport abstraction designed to accept vsock listener — but implementation deferred to #6293
- [ ] `--connect vsock://cid:port` URI scheme reserved

### Non-goals

- Replacing the HTTP/WS gateway — it remains for webhooks, web dashboard, and REST consumers
- Enclave security model, companion daemon split, network isolation — that is #6293
- TUI application code — that is #6824, #6823, #6825
- Binary frame support for file attachments — deferred to #6819, which will extend the wire format when it lands
