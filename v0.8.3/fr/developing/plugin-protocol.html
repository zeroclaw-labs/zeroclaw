Protocole de plugin

Ce document définit le protocole entre l’hôte de plugins de ZeroClaw et les composants de plugins WASM.

Qu’est-ce qu’un plugin

Un plugin est un composant WebAssembly autonome que ZeroClaw charge à l’exécution pour ajouter une fonctionnalité que le binaire principal n’inclut pas. Il réside dans son propre répertoire sous ~/.zeroclaw/plugins/, accompagné d’un manifeste qui lui donne un nom et déclare ce qu’il fournit. ZeroClaw le détecte au démarrage, le vérifie, et intègre ses fonctions exportées à l’agent en cours d’exécution afin qu’elles se comportent comme des fonctionnalités intégrées : un plugin d’outil apparaît pour le modèle comme un simple outil appelable (WasmTool implémente le même trait Tool qu’un outil natif), un plugin de canal se comporte comme un canal de messagerie, et un plugin de mémoire comme un backend de stockage.

Un plugin peut fournir une ou plusieurs des capacités définies dans PluginCapability (crates/zeroclaw-plugins/src/lib.rs) : un outil callable, un canal de messagerie, un backend mémoire, un backend d’observabilité ou un ensemble de compétences markdown. Le cas des compétences est particulier : il ne contient aucun WASM, uniquement un répertoire skills/ contenant du markdown, ce qui explique pourquoi il s’agit de la seule capacité à omettre le composant compilé.

Pourquoi en construire un

  • Étendez sans fork. Ajoutez un outil ou un canal sans modifier l’arborescence source de ZeroClaw ou attendre une version ; le plugin vous appartient et se charge depuis votre répertoire d’installation.
  • Comportement natif. Un plugin chargé n’est pas un complément de seconde classe. Le pont implémente les mêmes traits d’exécution que les fonctions intégrées, de sorte qu’un outil de plugin est proposé au modèle, attribué et invoqué exactement comme un outil officiel.
  • Choix de langage. Le contrat est le WIT et le WASI Component Model, et non une API Rust. Tout langage compilant vers un composant wasm32-wasip2 peut implémenter un world. Le guide pratique ci-dessous utilise Rust, car c’est l’approche qui offre le plus de support aujourd’hui, mais la frontière elle-même est agnostique du langage.
  • Sandboxé par défaut. L’hôte charge chaque plugin dans un contexte WASI sans pré-ouvertures de système de fichiers et sans réseau ambiant. Un plugin ne peut pas contacter silencieusement l’hôte ; il obtient exactement les fonctions de l’hôte connectées à son monde et rien de plus. Le HTTP sortant est la seule surface réseau qui peut être ouverte, et uniquement pour un plugin dont le manifeste autorise http_client.
  • Provenance vérifiable. Les manifests peuvent être signés avec Ed25519, et un opérateur peut exiger des signatures d’éditeurs de confiance avant le chargement de tout plugin.

Ce qu’un plugin ne peut pas faire (aujourd’hui)

Ce sont de réelles limites de l’hôte actuel, pas des préférences de style. Connaissez-les avant de concevoir autour d’une capacité qui n’est pas disponible.

  • **logging, l'injection de configuration, http_clientet les entrées entrantes fournies par l'hôte sont connectés.** Parmi les permissions qu'un manifeste peut déclarer,config_readinjecte la section de configuration du plugin, ethttp_clientattache une surfacewasi:http sortante afin que le plugin puisse effectuer des requêtes HTTP. Filesystem et les permissions d'accès à la mémoire sont toujours acceptées par le schéma du manifeste mais inertes : leurs fonctions hôtes ne sont pas encore enregistrées dans le linker. Voir Permissions et imports d'hôte ci-dessous.
  • Aucun réseau ni système de fichiers hôte ambiant. Le contexte WASI ne dispose pas de preopens ni de réseau ambiant, donc un plugin ne peut pas ouvrir de sockets bruts ni lire des fichiers hôtes via le WASI ambiant. Un plugin http_client obtient uniquement l’accès sortant wasi:http et rien d’autre ; il ne peut pas écouter. Les plugins de canal qui doivent recevoir du trafic entrant n’ouvrent pas eux-mêmes d’écouteur : l’hôte gère l’écouteur et transmet les messages via l’import inbound, que le plugin draine via son export poll-message.
  • Une frontière sur 32 bits. La cible est wasm32-wasip2. La mémoire de l’invité constitue un espace d’adressage sur 32 bits et l’ABI des composants représente les décalages sur 32 bits, indépendamment de la taille de mot de l’hôte. Les grandes valeurs (par exemple les octets bruts d’une liaison de canal) traversent la frontière par valeur. Consultez la section sur l’espace d’adressage sur 32 bits pour comprendre pourquoi il s’agit d’une contrainte imposée par la chaîne d’outils en amont, et non d’un drapeau que ce dépôt peut activer.
  • Un outil par plugin d’outil. Le monde tool-plugin exporte une seule interface tool avec un nom et un schéma. Un plugin qui doit exposer plusieurs outils livre plusieurs composants, ou un monde différent.
  • Contrat expérimental, non figé. wit/v0 ne comporte pas encore de marqueur .frozen, les interfaces peuvent donc encore évoluer avant la première version stable. Épinglez une version et prévoyez de recompiler lors d’une incrémentation de version WIT.

Architecture

Les plugins ZeroClaw sont des composants WebAssembly définis par des interfaces WIT sous wit/v0/ et hébergés directement via wasmtime (crates/zeroclaw-plugins). Un plugin est compilé en un composant WASI Preview 2 (wasm32-wasip2) qui exporte l’un des mondes de plugin (tool-plugin, channel-plugin, memory-plugin) et importe l’interface logging de l’hôte.

L’hôte se trouve dans crates/zeroclaw-plugins/src/component.rs. Il contient un wasmtime::Engine avec async activé, génère les liaisons du world avec wasmtime::component::bindgen! à partir de wit/v0, et branche une surface WASI p2 sandboxée dans le linker de chaque world. L’état hôte par appel (PluginState) transporte un WasiCtx construit sans preopens ni réseau, plus le ResourceTable requis par WASI. La seule importation hôte branchée dans le linker est logging ; l’autorité ambiante d’un plugin est donc le contexte WASI sandboxé et rien d’autre (voir Imports de l’hôte).

Les trois ponts de mondes mappent chaque monde WIT sur les traits natifs du runtime :

MondeModule de pontSurface d’exécution
tool-pluginruntime.rs, wasm_tool.rszeroclaw_api::tool::Tool
channel-pluginwasm_channel.rstrait de canal
memory-pluginwasm_memory.rstrait de backend mémoire

Les plugins d’outils utilisent un store frais par appel (sans état). Les plugins de canal et de mémoire maintiennent un store chaud protégé par un mutex asynchrone pour la durée de vie du plugin.

Les plugins d’outils sont découverts et enregistrés de bout en bout : le runtime parcourt la contrepartie outil de channel_plugin_details() et construit un WasmTool pour chacun. L’adaptateur hôte de canal (WasmChannel, son filtrage wasi:http, le confinement configure et la file inbound alimentée par l’hôte) est complet et couvert par des tests unitaires, et PluginHost::channel_plugin_details() expose les plugins de canal basés sur wasm à enregistrer. Le branchement de ceux-ci dans l’orchestrateur en direct (la boucle découverte-vers-canal dans le runtime, plus un écouteur hôte par fournisseur qui draine son transport dans la file inbound de chaque canal) est la couture restante et se concrétise avec le changement d’enregistrement de canal du runtime, pas cette tranche hôte. Le pont mémoire (WasmMemory) est dans la même position une étape plus tôt : l’adaptateur implémente le trait Memory complet contre le monde memory-plugin, mais l’hôte n’expose pas encore de contrepartie mémoire à channel_plugin_details() et le runtime ne construit pas encore un WasmMemory en tant que backend configurable.

Structure du plugin

Un plugin est un répertoire contenant :

my-plugin/
  manifest.toml    # Plugin metadata and permissions
  plugin.wasm      # Compiled WASM module (optional for skill-only plugins)

Les plugins sont découverts dans ~/.zeroclaw/plugins/ (configurable via plugins.plugins_dir dans la configuration).

Recherche et installation dans le registre

Le chemin d’installation local des plugins reste la source de vérité pour les plugins installés. Un registre n’est qu’un index JSON utilisé au moment de la commande pour découvrir et télécharger une archive de plugin :

zeroclaw plugin search calendar
zeroclaw plugin install team-calendar
zeroclaw plugin install team-calendar@0.2.0
zeroclaw plugin search calendar --registry https://example.invalid/registry.json
zeroclaw plugin install team-calendar --registry https://example.invalid/registry.json

zeroclaw plugin search récupère les métadonnées du registre et fait correspondre la requête aux noms et descriptions des plugins. Il n’installe pas, n’active pas et n’exécute pas le code des plugins.

zeroclaw plugin install <name> résout le nom depuis le registre, télécharge l’archive zip sélectionnée, vérifie le digest SHA-256 optionnel, extrait l’archive en toute sécurité, puis confie le répertoire du plugin extrait au chemin PluginHost::install existant. Les installations par chemin local restent inchangées :

Lorsqu’aucune version n’est épinglée, ZeroClaw choisit la dernière entrée correspondante dans l’index du registre, les éditeurs de registres doivent donc ordonner intentionnellement les noms répétés.

zeroclaw plugin install ./my-plugin
zeroclaw plugin install ./my-plugin/manifest.toml

L’URL du registre par défaut est :

https://raw.githubusercontent.com/zeroclaw-labs/zeroclaw-plugins/main/registry.jsonPour les registres privés ou de préproduction, utilisez --registry <url> par commande ou définissez ZEROCLAW_PLUGIN_REGISTRY_URL.

Les entrées du registre utilisent cette forme :

{
  extensions: [
    {
      « nom »: team-calendar,
      version: 0.8.3,
      "description": Planifier des réunions sur un calendrier d'équipe,
      auteur: Exemple d'équipe,
      capacités: [outil],
      "url": https://example.invalid/team-calendar-0.2.0.zip,
      sha256: "sha256:<hachage hexadécimal du zip>"
    }
  ]
}

L’archive doit contenir soit un manifest.toml au niveau racine, soit un répertoire de plugin imbriqué contenant manifest.toml. Les archives avec des chemins de traversée, des chemins absolus, des chemins préfixés par un lecteur Windows, ou plus d’un manifeste sont rejetées avant l’installation. Les téléchargements sont plafonnés pendant le streaming, de sorte qu’un serveur sans Content-Length ne peut pas forcer ZeroClaw à mettre en tampon une archive trop volumineuse. L’extraction est également plafonnée, de sorte qu’une archive compressée ne peut pas s’étendre sans limite dans la zone d’installation temporaire.

La recherche est une découverte non authentifiée. L’installation est la frontière de sécurité : les installations depuis le registre utilisent la politique de signature des plugins configurée et les clés d’éditeurs de confiance, de la même façon que les installations de plugins locaux via PluginHost::install.

Disposition de plugin avec compétences uniquement (bundle markdown)

Un plugin dont la seule capacité est skill fournit des skills dans un répertoire skills/ au format agentskills.io et omet wasm_path :

my-toolkit/
  manifest.toml              # declares the skill capability, no wasm_path
  README.md                  # optional bundle-level overview
  skills/
    design-review/
      SKILL.md
      scripts/
      references/
    code-review/
      SKILL.md
    data-analysis/
      SKILL.md
      references/

Chaque SKILL.md doit inclure un frontmatter YAML avec les champs name et description ; le runtime rejette les bundles dont les skills omettent l’un ou l’autre au moment de la découverte plutôt qu’à la première invocation. Les skills s’enregistrent sous des ID à espace de noms de plugin de la forme plugin:<plugin-name>/<skill-name> (par exemple plugin:my-toolkit/design-review) afin d’éviter les collisions avec les skills créés par l’utilisateur et entre les bundles.

Format du manifeste

Capacités

capabilities est une liste non vide de valeurs PluginCapability, définies dans crates/zeroclaw-plugins/src/lib.rs (sérialisées en snake_case). Chaque valeur sélectionne le monde WIT que le plugin exporte (tool, channel, memory), nomme un backend d’observabilité (observer), ou marque un bundle de compétences uniquement markdown (skill). Consultez l’enum pour l’ensemble canonique ; c’est la source de vérité et cette page ne le reformule pas.

Un manifeste doit déclarer au moins une capacité. wasm_path est requis pour chaque capacité, à l’exception d’un plugin dont la seule capacité est skill, qui ne transporte aucune charge utile WASM et qui est rejeté lors de la découverte s’il omet un bundle skills/ valide (validate_manifest_shape dans host.rs).

Autorisations

permissions est une liste de valeurs PluginPermission, également définies dans crates/zeroclaw-plugins/src/lib.rs. Consultez l’enum pour l’ensemble canonique.

Soyez conscient de l’écart entre les déclarations et les applications : aujourd’hui, au sein de l’hôte de composants, config_read et http_client ont un effet comportemental. runtime.rs transmet la section de configuration résolue d’un plugin outil à execute uniquement lorsque le manifeste accorde config_read, et supprime toute section __config fournie par l’appelant pour éviter qu’elle ne soit usurpée ; un plugin de canal reçoit la même section via son export configure selon la même règle. http_client attache un contexte sortant wasi:http au store du plugin et lie l’interface wasi:http, de sorte qu’un plugin autorisé puisse effectuer des requêtes HTTP et qu’un plugin sans cette permission n’ait aucune surface réseau. Les variantes restantes (file_read, file_write, memory_read, memory_write) sont acceptées par le schéma du manifeste mais ne sont pas encore connectées à un import hôte : les déclarer n’accorde rien par elles-mêmes. Elles réservent ces noms aux fonctions hôtes qui en contrôleront l’accès (voir Imports du système hôte ci-dessous).

interfaces WIT

Le contrat de plugin est l’ensemble des fichiers WIT dans wit/v0/, package zeroclaw:plugin@0.1.0. Chaque élément est soumis à @unstable(feature = plugins-wit-v0) jusqu’à ce que le package se stabilise ; voir wit/VERSIONING.md pour les règles de compatibilité. Les interfaces ci-dessous sont résumées à titre d’orientation ; les fichiers .wit font autorité pour les signatures exactes.

Mondes

wit/v0/ définit trois worlds, liés par bindgen! dans component.rs. Chacun importe logging (hôte) et exporte plugin-info ainsi que son interface principale : tool-plugin exporte tool, channel-plugin exporte channel, memory-plugin exporte memory. Les exports requis (no-default) pour chaque world sont listés dans le commentaire de documentation du world dans son fichier .wit.

interface tool

wit/v0/tool.wit définit la surface d’outil unique. L’hôte appelle name, description et parameters-schema une fois au moment du chargement, puis dispatche execute par invocation :

record tool-result {
    success: bool,
    output: string,
    error: option<string>,
}

name: func() -> string;
description: func() -> string;
parameters-schema: func() -> json-string;
execute: func(args: json-string) -> result<tool-result, string>;

parameters-schema renvoie une chaîne JSON Schema présentée au LLM pour l’appel d’outils. execute reçoit des arguments encodés en JSON correspondant à ce schéma et renvoie un tool-result ou une chaîne d’erreur. json-string est un alias de type string provenant de wit/v0/types.wit ; les appelants produisent du JSON valide, les récepteurs le parsent.

channel et memory interfaces

wit/v0/channel.wit et wit/v0/memory.wit définissent des surfaces contrôlées par des capacités. L’hôte appelle get-channel-capabilities / get-memory-capabilities une fois au chargement, et pour chaque flag non défini il utilise la valeur par défaut du trait Rust au lieu d’appeler le plugin. Un plugin doit toujours exporter chaque fonction (un stub renvoyant la valeur par défaut documentée est suffisant) ; l’hôte n’appelle simplement jamais celles dont le flag est absent. La valeur par défaut à laquelle se résout chaque flag non défini est documentée en ligne dans le WIT à côté des flags *-capabilities, qui est la source de vérité à la fois pour l’ensemble des flags et leurs valeurs par défaut.

Indicateurs de capacité

Les méthodes optionnelles sont annoncées via flags channel-capabilities et flags memory-capabilities. Parce que les flags sont un masque de bits, de nouvelles méthodes optionnelles peuvent être ajoutées à un package vN/ sans changement cassant, associées à une nouvelle fonction @since. Supprimer ou renommer un flag, une fonction, un champ ou un cas de variante est un changement cassant et nécessite un nouveau répertoire vN+1/.

Importations de l’hôte

Les fonctions hôtes sont importées par le plugin et fournies par le runtime. Le linker de chaque world lie logging (via l’implémentation hôte dans component_logging.rs, liée à add_wasi dans component.rs). Le world channel-plugin importe également inbound, la file d’attente de messages alimentée par l’hôte dont un channel puise via poll-message. Le module sortant wasi:http est lié en supplément pour tout plugin dont le manifest accorde http_client (add_wasi_http dans component.rs), conditionné pour que le contexte et l’interface liée correspondent toujours. Les autorisations d’accès au système de fichiers et à la mémoire restent inactives : les fonctions hôtes qui les régulent ne sont pas encore intégrées au linker. L’autorité ambiante d’un plugin correspond au contexte WASI (aucun préopens, aucun réseau ambiant), auxquels s’ajoutent strictement les imports hôte que son world et ses permissions lient.

inbound

wit/v0/inbound.wit est importé par le world channel-plugin. Un plugin de canal s’exécute sans écouteur propre, donc l’hôte exécute l’écouteur (un serveur webhook, un tunnel fournisseur, un client de polling) et met en file d’attente chaque message reçu. Le plugin vide la file d’attente depuis son export poll-message en appelant inbound-poll, avec inbound-pending disponible pour vider par lots :

inbound-poll: func() -> option<host-inbound-message>;
inbound-pending: func() -> u32;

Le côté hôte possède une InboundQueue par canal ; WasmChannel::inbound remet un clone à la tâche d’écoute afin que le trafic mis en file d’attente soit visible pour le drain du plugin.

logging

wit/v0/logging.wit est importé par les trois worlds. Les plugins appellent log-record pour émettre des événements structurés vers l’hôte :

log-record: func(level: log-level, event: plugin-event);

L’appel est fire-and-forget : il ne renvoie rien et l’hôte (component_logging.rs) absorbe toutes les erreurs, de sorte qu’une écriture de log échouée ne peut jamais faire planter l’exécution du plugin. plugin-action et plugin-outcome reflètent les taxonomies fermées Action / EventOutcome dans zeroclaw-log ; il n’existe volontairement aucune variante d’échappatoire. N’appelez pas wasi:logging directement, les événements de plugin seraient formatés de manière incohérente et n’atteindraient pas toutes les destinations vers lesquelles zeroclaw_log écrit.

Config par plugin (__config)

Autorisation : config_read

Un plugin ne lit pas les variables d’environnement du processus. Pour les plugins d’outils, l’hôte résout la section de configuration propre au plugin (le dictionnaire config par entrée sous le schéma plugins.entries) et l’injecte dans l’entrée execute sous la clé réservée __config, mais uniquement lorsque le manifeste accorde config_read :

{
  "invite": un coucher de soleil,
  __config: { api_key: "...", base_url: "..." }
}

runtime.rs supprime tout __config fourni par l’appelant avant d’injecter la section résolue, de sorte que la section ne peut pas être usurpée, et la retient entièrement lorsque la permission est absente. Les opérateurs renseignent cette section via les surfaces de configuration ci-dessus (zerocode, le CLI, la gateway) plutôt qu’en éditant un fichier à la main, avec une exception actuelle : un plugin fraîchement installé n’a pas encore d’entrée plugins.entries, et config set ne peut pas matérialiser une entrée à clé naturelle manquante, de sorte que la première entrée doit être ajoutée au fichier à la main (suivi dans l’issue #8636). Les clés de la section sont celles que le schéma du plugin déclare. Le champ est marqué secret, de sorte que les valeurs écrites par le CLI sont chiffrées au repos sous le .secret_key adjacent ; les valeurs en clair écrites à la main sont également acceptées au chargement. Un plugin ne voit jamais que sa propre section.

Hôte de composants WASI

L’hôte (crates/zeroclaw-plugins/src/component.rs) compile et instancie les composants avec un unique wasmtime::Engine asynchrone. La façon dont un fichier .wasm est chargé dépend du backend d’exécution du build :

  • plugins-wasm-cranelift: un backend JIT est présent, donc load_component compile un composant .wasm lors du chargement via Component::from_file.`
  • Aucun backend JIT (plugins-wasm-pulley ou runtime uniquement) : il n’y a pas de compilateur dans le binaire, donc load_component désérialise directement le fichier via Component::deserialize_file, en le traitant comme un .cwasm précompilé produit par un wasmtime correspondant. Un artefact incompatible est rejeté par le contrôle de version de la désérialisation.

Les deux fonctionnalités back-end incluent plugins-wasmtime ; le chemin de chargement dépend de la présence du compilateur cranelift dans le build, et non de pulley.

Limites d’exécution par appel

Chaque appel de plugin s’exécute sous les limites de ressources par appel que l’hôte applique au store. Le moteur active le comptage du carburant, et chaque appel reçoit un nouveau budget de carburant afin qu’un composant incontrôlé ou malveillant provoque un trap au lieu de bloquer l’hôte. Un plafond StoreLimits limite la mémoire linéaire, les éléments de table et le nombre d’instances. Le tool world obtient un store frais à chaque exécution ; les stores de la chaîne chaude et de la mémoire sont rechargés en carburant avant chaque appel, permettant ainsi à un plugin de longue durée de bénéficier d’un nouveau budget plutôt que de s’épuiser au cours de sa durée de vie.

Les quatre bornes sont réglables par l’opérateur et chaque valeur est validée comme non nulle : plugins.limits.call_fuel (par défaut 1,000,000,000 unités d’instruction), plugins.limits.max_memory_mb (par défaut 256), plugins.limits.max_table_elements (par défaut 100,000), et plugins.limits.max_instances (par défaut 64). Un store ne peut être construit qu’avec des limites explicites, de sorte qu’aucun chemin de chargement ne puisse construire un plugin non sandboxé. Les champs et valeurs par défaut canoniques se trouvent dans la référence de configuration.

Espace d’adressage 32 bits (wasip2 est wasm32)

La cible du plugin est wasm32-wasip2, et le moteur hôte est construit avec la mesure de fuel activée (Config::consume_fuel(true)) sans wasm_memory64. La frontière du plugin est un format 32 bits fixe, et cela a des conséquences qu’il convient d’énoncer clairement :

  • L’espace d’adressage de l’invité est de 32 bits. Un plugin s’exécute dans une mémoire linéaire wasm32. Les grandes valeurs franchissent la limite par valeur : le media-attachment d’un plugin de canal transporte l’intégralité de ses octets sous forme de list<u8>, et wit/v0/channel.wit indique déjà qu’il peut s’agir de plusieurs mégaoctets et reporte un modèle resource-handle à une version ultérieure. Dans cet espace 32 bits, l’hôte applique un plafond mémoire explicite par store défini via plugins.limits.max_memory_mb (valeur par défaut : 256), de sorte qu’un invité est limité par la plus petite valeur entre l’espace d’adressage wasm32 et ce plafond configuré pour ZeroClaw.
  • L’ABI du composant représente les décalages sur 32 bits, quelle que soit la taille du mot hôte. Même sur un hôte 64 bits, les décalages des listes et des chaînes dans l’ABI canonique sont de type i32. memory64 étend l’adressage de la mémoire linéaire de l’invité, et non l’ABI canonique du modèle de composant ; son activation ne ferait donc pas passer les champs de niveau WIT sur 64 bits.
  • Il n’existe pas de cible 64 bits pour wasip2 contre laquelle se lier. wasm32-wasip2 est la seule cible WASI Preview 2 dans rustc et LLVM à ce jour ; un plugin ne peut pas être compilé en un composant p2 64 bits, il n’y a donc rien à charger pour l’hôte même si le moteur active memory64.

Il s’agit d’une contrainte de la toolchain amont, et non d’une limitation de l’hôte qu’un flag de ce dépôt peut lever. Lorsqu’une cible p2 64 bits et une ABI de composant plus large arriveront en amont, la jointure bindgen! se régénèrera par rapport à elles et les largeurs de champs seront réexaminées dans le WIT sous la fenêtre wit/VERSIONING.md. D’ici là, traitez la frontière du plugin comme 32 bits par construction.

Signatures

Les manifestes de plugins peuvent porter une signature Ed25519 (crates/zeroclaw-plugins/src/signature.rs). La signature est encodée en base64url sur les octets du manifeste canonique (le TOML avec les lignes signature et publisher_key retirées) ; la clé publique de l’éditeur est encodée en hexadécimal. L’hôte applique l’un des trois modes de plugins.security.signature_mode :

ModePlugin non signéSignature non fiable ou invalide
strictrejetérejeté
permissivechargé avec un avertissementchargé avec un avertissement
disabledchargénon coché

La vérification s’exécute à la fois lors de la découverte et de l’installation. La découverte ignore un plugin qui échoue à sa politique plutôt que d’interrompre l’hôte entier ; l’installation renvoie l’erreur.

Écrire un plugin en Rust

Un plugin est une crate cdylib qui cible le modèle de composants. Générez les bindings guest à partir du même package wit/v0 utilisé par l’hôte, implémentez le monde exporté et compilez vers wasm32-wasip2. Pour les didacticiels complets allant d’un crate vide à un plugin installé, consultez les guides des plugins ; les notes ci-dessous couvrent les mécanismes de compilation et d’installation.

Construction

sh

# Installer la cible WASI Preview 2 (une seule fois)
rustup target add wasm32-wasip2

# Construire le composant
cargo build --target wasm32-wasip2 --release

Le composant de sortie se trouve à target/wasm32-wasip2/release/<crate_name>.wasm. Copiez-le à côté de votre manifest.toml. Pour un build d’hôte runtime uniquement sans backend JIT, précompilez le composant en un .cwasm avec un wasmtime correspondant et livrez celui-ci à la place, car un tel hôte désérialise plutôt que de compiler au chargement.

Le fixture de référence n’est pas commité dans l’arborescence (c’est un artefact de build, pas du code source). Lorsque crates/zeroclaw-plugins/tests/fixtures/reference-plugin.wasm est provisionné par un cargo build --target wasm32-wasip2 propre du plugin de référence publié, reference_plugin.rs et reference_plugin_e2e.rs le chargent via le même PluginHost et les mêmes chemins de résolution de configuration que le daemon utilise. Lorsque l’artefact est absent, ces tests sont ignorés.

Configuration

Vous éditez rarement le TOML à la main pour configurer un plugin. ZeroClaw expose le schéma de configuration du plugin via chaque surface, et chaque surface écrit le même état sous-jacent via le miroir de schéma (la seule exception actuelle : l’amorçage de l’entrée plugins.entries d’un nouveau plugin, selon la note sous Configuration par plugin). Choisissez celle qui convient au moment :

  • zerocode l’éditeur de configuration interactif. Accédez à la section plugins et définissez les champs avec validation et aide contextuelle.
  • L’interface de ligne de commande pour le cycle de vie des plugins. zeroclaw plugin propose list, search, install, remove, info et migrate. zeroclaw config set modifie les champs de configuration individuels de chaque plugin.
  • La passerelle web pour une vue tableau de bord. GET /api/plugins retourne les plugins chargés et indique si le système est activé.
  • Le schéma du plugin, si vous êtes l’auteur du plugin. Votre surface de configuration est définie par le schéma, et non en demandant aux opérateurs d’écrire du TOML. L’hôte injecte une section de configuration définie par l’auteur dans le plugin au moment de l’appel (voir Per-plugin config), de sorte que ce qu’un opérateur renseigne corresponde à ce que votre schéma déclare.

Le miroir de schéma est ce qui rend cela possible : les types de configuration de plugins dans crates/zeroclaw-config/src/schema.rs portent #[prefix = "plugins"], #[prefix = "plugins.entries"] et #[prefix = "plugins.security"], et le derive Configurable transforme chaque champ préfixé en un chemin que chaque surface lit et écrit. Les champs secrets (la map config d’une entrée de plugin est marquée #[secret]) sont chiffrés au repos sous le .secret_key adjacent. Les champs canoniques, les valeurs par défaut et les valeurs de signature_mode se trouvent dans la référence Config ; ce schéma est la source de vérité, pas cette page.

Fonctionnalités de build

L’hôte de plugins est une option d’inclusion à la compilation. Les fonctionnalités au niveau binaire dans le Cargo.toml de l’espace de travail sélectionnent si les plugins sont intégrés ou non et quel backend d’exécution est livré :

  • plugins-wasm est la fonctionnalité ombrelle qui intègre l’hôte de plugins et son intégration runtime dans le binaire. Elle est requise : les fonctionnalités backend ci-dessous sélectionnent un moteur d’exécution mais n’impliquent pas la fonctionnalité ombrelle, donc compiler avec seulement une fonctionnalité backend réussit et produit silencieusement un binaire sans la sous-commande plugin. Passez toujours plugins-wasm plus le backend choisi, p. ex. --features plugins-wasm,plugins-wasm-cranelift.
  • plugins-wasm-runtime-only est le plus petit et le plus rapide à démarrer : pas de JIT, donc les composants sont désérialisés à partir d’un .cwasm précompilé.
  • plugins-wasm-cranelift ajoute le JIT Cranelift, de sorte qu’un composant .wasm est compilé au chargement.
  • plugins-wasm-pulley est le plus portable, supportant la compilation sur des cibles que Cranelift ne couvre pas.

Celles-ci délèguent aux features du crate zeroclaw-plugins (plugins-wasmtime, plugins-wasm-cranelift, plugins-wasm-pulley) qui branchent wasmtime. Le chemin de chargement dépend de la présence du compilateur Cranelift dans le build, comme décrit sous WASI Component Host. Consultez les commentaires des features dans le Cargo.toml du workspace pour les descriptions faisant autorité.
